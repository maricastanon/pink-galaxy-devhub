ARCH-CMD04.1 — Viewing Processes: ps, top, htop, btop

A process is a running program. Every command you type, every application you open, every system service running in the background is a process with a unique ID number (PID). Think of your computer as a hotel: each process is a guest with a room number (PID), a check-in time, a resource budget (CPU and memory), and a status (active, sleeping, or checked out). The process viewing tools are your front desk monitors — they show you who is in the hotel, how much they are consuming, and whether anyone is causing trouble.

ps gives you a snapshot (a photograph of the current moment). top gives you a live dashboard (a security camera feed that updates continuously). htop and btop are modern replacements with better visuals — htop shows colorful CPU/memory bars and lets you scroll and search, while btop (installed on your Arch system) adds network and disk I/O monitoring with a beautiful terminal UI.

The ps command reads process information from the /proc virtual filesystem — each process has a directory /proc/PID/ containing files like status (state, memory), cmdline (command that started it), fd/ (open file descriptors), and maps (memory mappings). Common ps invocations: ps aux (BSD-style: all processes, user-oriented, including those without a terminal), ps -ef (POSIX-style: full format with PPID), ps --forest (show process tree). Process states in the STAT column: R (running/runnable), S (sleeping — waiting for an event), D (uninterruptible sleep — usually disk I/O), T (stopped by signal), Z (zombie — exited but parent has not called wait()). The top command refreshes every 3 seconds by default and sorts by CPU usage. Key bindings inside top: M (sort by memory), P (sort by CPU), k (kill a process), q (quit).

The /proc filesystem is a virtual filesystem (procfs) that does not exist on disk — it is generated on-the-fly by the kernel when you read from it. Each /proc/PID/status file is produced by proc_pid_status() in fs/proc/array.c, which reads fields directly from the process's task_struct in kernel memory. The task_struct (defined in include/linux/sched.h, approximately 700 bytes) is the kernel's complete representation of a process: it contains the process state, PID, parent PID, scheduling priority, CPU time counters, memory descriptor pointer (mm_struct), file descriptor table pointer, signal handlers, credentials (UID/GID), cgroup membership, namespace references, and much more. On your Dell 7420's 12th Gen Intel with P-cores and E-cores, the CFS (Completely Fair Scheduler) makes scheduling decisions using a red-black tree of runnable tasks, weighted by nice value and cgroup constraints, with awareness of the heterogeneous core topology — preferring P-cores for CPU-bound tasks and E-cores for background tasks (via the Intel Thread Director hardware, exposed to the kernel through the HFI interface).

bash
# ═══ ps — Process Snapshot ═══

# Show all processes (the most common invocation)
ps aux

# Show process tree (parent → child relationships)
ps aux --forest

# Find a specific process

pgrep -a docker                     # cleaner: list PIDs + command lines

# Show only your processes

# Top CPU consumers (snapshot — not live)
ps aux --sort=-%cpu | head -10

# Top memory consumers
ps aux --sort=-%mem | head -10

✅ The daily workflow:

Use btop for a quick visual health check of your system (CPU, memory, disk, network all in one view). Use ps aux | grep name or pgrep -a name to find a specific process. Use ps aux --sort=-%cpu | head to quickly find what is eating your CPU. Use pstree -p PID to see the parent/child relationships of a process.

❌ Common confusion: STAT column letters

S (sleeping) does NOT mean the process is broken — most processes spend 99.9% of their time sleeping, waiting for input. D (uninterruptible sleep) usually means disk I/O — if a process is stuck in D state for a long time, the disk or filesystem might have a problem. Z (zombie) means the process has exited but its parent has not read its exit code yet — a few zombies are normal, hundreds indicate a buggy parent process.

The /proc virtual filesystem — the kernel generates files like /proc/PID/status on-the-fly from its internal task_struct data structures when you read them
A database file at /var/lib/processes.db
It asks each process directly using inter-process communication

ARCH-CMD04.2 — kill & Signals: Talking to Processes

Despite its name, kill does not always kill processes — it sends them messages called "signals." Think of signals as different types of taps on the shoulder. SIGTERM (signal 15) is a polite tap: "Please shut down gracefully when you are ready." The process can clean up its resources, save its state, and exit orderly. SIGKILL (signal 9) is being carried out by security guards: the kernel terminates the process immediately with no chance to clean up. You should almost always try SIGTERM first and only use SIGKILL as a last resort, because SIGKILL can leave behind lock files, corrupt partially-written data, and orphan child processes.

When you press Ctrl+C in the terminal, you are actually sending SIGINT (signal 2, "interrupt") to the foreground process. When you press Ctrl+Z, you send SIGTSTP (signal 20, "terminal stop") which pauses the process. These are the signals you use most often without even thinking about it.

Signals are asynchronous notifications delivered by the kernel to processes. There are 31 standard signals (1-31) plus real-time signals (32-64). Key signals and their default actions: SIGHUP (1) — terminal disconnected; many daemons reload config on SIGHUP. SIGINT (2) — keyboard interrupt (Ctrl+C); terminates process. SIGQUIT (3) — keyboard quit (Ctrl+\); terminates with core dump. SIGKILL (9) — force kill; cannot be caught, blocked, or ignored. SIGTERM (15) — graceful termination; default signal sent by kill. SIGSTOP (19) — pause process; cannot be caught. SIGCONT (18) — resume paused process. SIGTSTP (20) — terminal stop (Ctrl+Z); can be caught. Processes can register signal handlers with sigaction() to perform custom cleanup when receiving catchable signals. Only SIGKILL and SIGSTOP cannot be caught or ignored.

Signal delivery works through the kernel's signal infrastructure in kernel/signal.c. When you call kill(pid, sig), the kernel sets a bit in the target process's pending signal bitmask (inside the task_struct). The signal is not delivered immediately — it is delivered the next time the target process returns from kernel space to user space (at the end of a syscall, interrupt, or exception). At that point, do_signal() checks the pending bitmask, finds the highest-priority pending signal, and either executes the registered handler (by modifying the process's user-space stack to call the handler function when it returns) or performs the default action (terminate, terminate+core dump, stop, or ignore). For SIGKILL, the kernel calls do_group_exit() which sets the process state to TASK_DEAD and releases all resources — the process has no opportunity to execute any user-space code, which is why SIGKILL cannot be caught. On your Alder Lake CPU, signal delivery latency is typically under 10 microseconds because the kernel checks pending signals on every return-to-userspace path.

bash
# ═══ Sending Signals ═══

# Graceful termination (SIGTERM — the default, always try this first)
kill 1234                     # sends SIGTERM to PID 1234
kill -15 1234                  # same thing, explicitly
kill -SIGTERM 1234             # same thing, by name

# Force kill (SIGKILL — only if SIGTERM didn't work after ~5 seconds)
kill -9 1234                   # ☠️ immediate, no cleanup
kill -SIGKILL 1234             # same thing

# Kill by name (no need to find the PID first)
pkill firefox                   # SIGTERM all processes named "firefox"
pkill -9 firefox                # SIGKILL all firefox processes
killall waybar                  # SIGTERM all processes named exactly "waybar"

# Reload config without restarting (SIGHUP — many daemons support this)
sudo kill -HUP $(pgrep nginx)  # nginx reloads its config

# Pause and resume a process
kill -STOP 1234                # freeze the process
kill -CONT 1234                # unfreeze it

# ═══ The Correct Kill Sequence ═══
# 1. Try SIGTERM first (graceful):
kill 1234
# 2. Wait 5 seconds
sleep 5
# 3. Check if it's still running:
ps -p 1234 > /dev/null 2>&1 && echo "still alive" || echo "dead"
# 4. Only if still alive, SIGKILL:
kill -9 1234
✅ Always SIGTERM before SIGKILL:

SIGTERM gives the process a chance to close files cleanly, flush buffers to disk, release locks, notify connected clients, and remove temporary files. SIGKILL does none of this — it is instant death. A database killed with SIGKILL may corrupt its data files. A web server killed with SIGKILL drops all active connections without sending error responses. Always be polite first.

❌ When SIGKILL does not work:

A process in D state (uninterruptible sleep — usually waiting for disk I/O) cannot be killed even with SIGKILL because the kernel does not deliver signals to processes in this state. The only fix is to resolve the I/O issue (unmount the stuck filesystem, reboot the machine, or fix the failing disk). Zombie (Z) processes also cannot be killed because they are already dead — their parent needs to call wait() to reap them.

🧪 Why should you try kill 1234 (SIGTERM) before kill -9 1234 (SIGKILL)?
SIGTERM lets the process clean up gracefully — close files, flush data, release locks. SIGKILL terminates instantly with no cleanup, which can corrupt data, leave lock files behind, and orphan child processes.
SIGTERM is faster than SIGKILL
SIGKILL does not actually stop the process

ARCH-CMD04.3 — Job Control: bg, fg, jobs, &, nohup, disown

Job control lets you manage multiple tasks from a single terminal. When you press Ctrl+Z, the current command pauses (stops, but stays in memory). You can then start a new command, and later bring the paused one back. The & at the end of a command runs it in the background immediately, giving you your prompt back while it works. bg resumes a paused job in the background. fg brings a background job to the foreground. jobs lists all jobs in the current shell session.

The really useful trick is nohup and disown: normally, when you close your terminal, all jobs started in that terminal are killed (they receive SIGHUP). If you want a job to survive the terminal closing (e.g., a long download or a script that should keep running after you log out of SSH), wrap it with nohup or use disown after starting it.

Bash's job control is built on process groups and the terminal's foreground process group. Each pipeline is assigned a process group ID (PGID). The terminal (TTY) has one foreground process group — only processes in this group receive keyboard signals (SIGINT from Ctrl+C, SIGTSTP from Ctrl+Z). When you run cmd &, bash puts the process in a background process group. When you type fg %1, bash calls tcsetpgrp() to make job 1's process group the terminal's foreground group. The nohup command sets the SIGHUP handler to SIG_IGN (ignore) and redirects stdout/stderr to nohup.out. The disown builtin removes a job from bash's job table so bash does not send SIGHUP to it when the shell exits. For long-running tasks, tmux (which you set up in Part 2) is the professional solution — it detaches the entire terminal session from the physical terminal, so everything survives logout.

bash
# ═══ Background & Foreground ═══

# Run a command in the background (append &)
sleep 300 &                   # starts in background, prompt returns immediately
# [1] 5678                    ← job number [1], PID 5678

# List background jobs
jobs                           # shows all jobs in this shell
# [1]+  Running    sleep 300 &

# Bring background job to foreground
fg %1                          # %1 = job number 1

# Pause a running foreground command
# Press Ctrl+Z while it's running
# [1]+  Stopped    sleep 300

# Resume it in the background
bg %1

# ═══ Surviving Terminal Close ═══

# nohup: immune to hangup signal, output goes to nohup.out
nohup ./long_running_script.sh &

# disown: remove from job table (no SIGHUP on terminal close)
./background_task.sh &
disown %1

# Best solution: use tmux (from Part 2 of your Arch Guide)
tmux new -s work              # create a named session
# ... run commands ...
# Ctrl+B, D → detach (session keeps running)
tmux attach -t work           # reconnect later (even from different SSH session!)
🧪 You are connected via SSH and start a long backup script. You need to close your laptop and leave. What is the safest way to keep the script running?
Just close the terminal — Linux will keep it running
Press Ctrl+Z to pause it

ARCH-CMD04.4 — systemctl: Managing System Services

systemd is the system manager that runs as PID 1 on your Arch Linux machine — it is the very first process the kernel starts, and it is responsible for starting and managing all other services: your network (NetworkManager), your audio (PipeWire), your firewall (ufw), your Docker daemon, your SSH server, and hundreds of other background services. The systemctl command is how you talk to systemd — you use it to start, stop, restart, enable (auto-start on boot), and check the status of services. This is arguably the most important system administration command on modern Linux.

Systemd organizes its work into "units" of different types. The most common types: .service (daemon processes: docker.service, sshd.service), .timer (scheduled tasks — systemd's replacement for cron), .socket (network sockets that activate services on demand), .target (groups of units — like runlevels: multi-user.target, graphical.target), .mount (filesystem mount points), .path (file/directory watches). Unit files live in /usr/lib/systemd/system/ (package-installed, do not edit), /etc/systemd/system/ (admin customizations — overrides the former), and ~/.config/systemd/user/ (user services). The enable command creates a symlink in the appropriate .wants/ directory so the unit starts automatically at boot. The --now flag combines enable+start or disable+stop into one command.

Systemd (PID 1) communicates with systemctl via D-Bus (the system message bus — a local IPC mechanism using Unix domain sockets). When you run systemctl start docker, the systemctl client sends a D-Bus method call to the org.freedesktop.systemd1 service, which is PID 1 itself. Systemd then reads the docker.service unit file, resolves its dependencies (After=, Requires=, Wants=), creates a control group (cgroup) for the service via the cgroup2 filesystem at /sys/fs/cgroup/, forks a child process within that cgroup, applies resource limits (MemoryMax=, CPUWeight=, etc.), executes the ExecStart= command, and monitors the process. If the process crashes and Restart=on-failure is set, systemd automatically restarts it. The cgroup membership ensures that all child processes spawned by the service are tracked and can be cleanly stopped together (no orphan processes). On your system, every service you configured in Parts 1-4 (NetworkManager, PipeWire, Docker, fail2ban, fstrim.timer) is managed through this mechanism.

bash
# ═══ systemctl — The Essential Commands ═══

# Status of a service (the first thing to check when troubleshooting)
systemctl status docker
# Shows: active/inactive, PID, memory usage, recent log entries

# Start / Stop / Restart
sudo systemctl start docker          # start now
sudo systemctl stop docker           # stop now
sudo systemctl restart docker        # stop + start
sudo systemctl reload nginx          # reload config without restarting

# Enable / Disable auto-start on boot
sudo systemctl enable docker          # start on every boot
sudo systemctl disable docker         # don't start on boot
sudo systemctl enable --now docker    # enable AND start (one command!)

# Check if a service is active (for scripts)
systemctl is-active --quiet docker && echo "running" || echo "stopped"
systemctl is-enabled docker            # "enabled" or "disabled"

# ═══ Listing and Inspecting ═══

# All running services
systemctl list-units --type=service --state=running

# All failed services (🚨 check this regularly!)
systemctl --failed

# All enabled timers (scheduled tasks)
systemctl list-timers

# View the unit file for a service
systemctl cat docker.service

# Show all dependencies of a service
systemctl list-dependencies docker.service

# ═══ Your Arch System's Key Services (from Parts 1-4) ═══

✅ The troubleshooting workflow:

Service not working? Step 1: systemctl status servicename — is it active, failed, or inactive? Step 2: journalctl -u servicename -n 50 — read the last 50 log lines (next section). Step 3: Fix the issue. Step 4: sudo systemctl restart servicename. This four-step workflow solves 90% of service problems.

❌ Common mistake: enable vs start

enable only configures auto-start on boot — it does NOT start the service right now. start starts it right now but does NOT configure it for boot. You usually want both: sudo systemctl enable --now servicename. Forgetting enable means the service works now but disappears after reboot. Forgetting start means it will work after reboot but not right now.

🧪 You run sudo systemctl start docker and Docker works. But after rebooting, Docker is not running. What did you forget?
You forgot to enable it — "start" only runs the service now, "enable" configures it to start automatically at boot. Use "enable --now" to do both.

ARCH-CMD04.5 — journalctl: Reading System Logs

journalctl is your window into everything that has happened on your system. Every service, every kernel message, every login attempt, every error — all recorded in the systemd journal. When something goes wrong, journalctl is where you find out what happened and when. Think of it as the flight recorder (black box) of your system — it records everything, and you use journalctl to play back the recording, filtered by time, service, severity, or any combination.

The systemd journal (journald) stores logs in a binary format at /var/log/journal/ (persistent) or /run/log/journal/ (volatile — RAM only, lost on reboot). The binary format enables indexed searching by any field (unit, priority, PID, etc.) without grep. Logs are structured: each entry has key-value metadata fields (_SYSTEMD_UNIT, PRIORITY, _PID, MESSAGE, etc.) that enable precise filtering. Priority levels follow syslog convention: 0=emerg, 1=alert, 2=crit, 3=err, 4=warning, 5=notice, 6=info, 7=debug. The journal automatically rotates based on disk usage (configured in /etc/systemd/journald.conf: SystemMaxUse=, SystemKeepFree=). Boot sessions are identified by a boot ID, making it easy to see logs from previous boots (useful for diagnosing boot failures).

bash
# ═══ journalctl — Essential Queries ═══

# All logs (newest last — scroll with arrow keys, q to quit)
journalctl

# Follow live (like tail -f) — great for real-time debugging
journalctl -f

# ═══ Filter by Service ═══
journalctl -u docker                  # all Docker logs
journalctl -u docker -n 50            # last 50 lines
journalctl -u docker -f               # follow Docker logs live
journalctl -u sshd -u fail2ban        # logs from TWO services

# ═══ Filter by Time ═══
journalctl --since "1 hour ago"
journalctl --since "today"
journalctl --since "2026-03-15 14:00" --until "2026-03-15 16:00"

# ═══ Filter by Priority (severity) ═══
journalctl -p err                     # errors and above only (err, crit, alert, emerg)
journalctl -p warning                 # warnings and above
journalctl -p err --since "today"     # today's errors

# ═══ Kernel Messages ═══
journalctl -k                          # kernel ring buffer (same as dmesg)
journalctl -k -p err                  # kernel errors only

# ═══ Previous Boot (diagnosing boot problems) ═══
journalctl -b -1                       # logs from the previous boot
journalctl -b -1 -p err               # errors from previous boot
journalctl --list-boots                # list all recorded boots

# ═══ Disk Usage ═══
journalctl --disk-usage                # how much space logs are using
sudo journalctl --vacuum-size=500M    # reduce logs to 500MB
sudo journalctl --vacuum-time=7d      # keep only last 7 days

ARCH-CMD04.6 — Scheduled Tasks: cron & systemd Timers

Scheduled tasks let you automate things that need to happen at specific times — daily backups at 3 AM, weekly system maintenance every Sunday, disk usage reports every hour, or certificate renewal checks twice a day. Linux offers two systems for this: the classic cron and the modern systemd timers. On Arch Linux with systemd, timers are the preferred method because they integrate with journalctl for logging, can be managed with systemctl, support calendar expressions, and can trigger on events (not just time schedules). You already set up fstrim.timer in Part 4 of the Arch guide — that is a systemd timer that runs weekly SSD TRIM operations.

Cron uses a crontab (cron table) file with lines in the format minute hour day month weekday command. Edit with crontab -e, list with crontab -l. Special values: * (every), */5 (every 5th), 1-5 (range), 1,15 (specific values). Systemd timers consist of two unit files: a .timer file (defines the schedule) and a matching .service file (defines what to run). Timer types: OnCalendar= (absolute: "daily", "weekly", "Mon *-*-* 03:00:00") and OnBootSec= / OnUnitActiveSec= (relative: "15min after boot", "1h after last run"). Timers support Persistent=true which means if the system was off during a scheduled run, the timer fires immediately on next boot. View active timers with systemctl list-timers.

bash
# ═══ cron — The Classic Scheduler ═══

# Edit your crontab
crontab -e

# Crontab format:  minute  hour  day  month  weekday  command
# ┌───── minute (0-59)
# │ ┌─── hour (0-23)
# │ │ ┌─ day of month (1-31)
# │ │ │ ┌ month (1-12)
# │ │ │ │ ┌ weekday (0-7, 0&7=Sunday)
# │ │ │ │ │
# * * * * *  command to run

# Examples:
# 0 3 * * *    ~/.local/bin/backup.sh           # daily at 3:00 AM
# */15 * * * * ~/.local/bin/health-check.sh     # every 15 minutes
# 0 9 * * 1-5  echo "Start of work day"         # weekdays at 9 AM
# 0 0 * * 0    ~/.local/bin/maintain.sh          # Sundays at midnight

# List your crontab
crontab -l

# ═══ systemd Timers — The Modern Way ═══

# Example: Create a daily backup timer

# Step 1: Create the service file (what to run)
sudo tee /etc/systemd/system/backup.service << 'EOF'
[Unit]
Description=Daily Backup

[Service]
Type=oneshot
ExecStart=/home/mari/.local/bin/backup.sh
User=mari
EOF

# Step 2: Create the timer file (when to run)
sudo tee /etc/systemd/system/backup.timer << 'EOF'
[Unit]
Description=Run backup daily at 3 AM

[Timer]
OnCalendar=*-*-* 03:00:00
Persistent=true

[Install]
WantedBy=timers.target
EOF

# Step 3: Enable and start the timer
sudo systemctl enable --now backup.timer

# Check it
systemctl list-timers | grep backup
# Shows: next run time, last run time, time until next run

# Test it manually
sudo systemctl start backup.service   # run the service NOW (timer activates on schedule)

# Check logs
journalctl -u backup.service -n 20
Expected Output (systemctl list-timers)
NEXT                        LEFT       LAST                        PASSED    UNIT
Sun 2026-03-16 03:00:00 CET 12h left   Sat 2026-03-15 03:00:00 CET 12h ago  backup.timer
Mon 2026-03-16 00:00:00 CET 8h left    Mon 2026-03-09 00:00:00 CET 6 days   fstrim.timer
✅ Why systemd timers over cron:

Timers integrate with journalctl (logs are structured and searchable), support Persistent=true (catches up on missed runs), can depend on other services, have precise calendar expressions, and can be managed with the same systemctl commands you already know. Use systemd timers for all new scheduled tasks on Arch.

❌ Cron gotcha — environment:

Cron runs with a minimal environment (almost no PATH, no display). Scripts that work in your terminal often fail in cron because commands like docker or paru are not in cron's PATH. Always use full paths in cron scripts (/usr/bin/docker instead of docker), or set PATH at the top of the crontab.

Exercise 1 (Easy) — Process Explorer

Find the PID of your Hyprland compositor. Check how much memory it is using. Find all its child processes. Then check the status of three services from your Arch installation (Docker, NetworkManager, sshd).

solution
#!/bin/bash
# Find Hyprland

# Check memory (RSS column in kilobytes)

Exercise 2 (Medium) — Service Monitor Script

Write a script that checks a list of critical services (docker, sshd, NetworkManager, fail2ban, postgresql), reports their status with colored output, and attempts to restart any that are not running. Log all actions with timestamps. Use the logging functions from BASH-F08.

solution
#!/bin/bash
set -euo pipefail

SERVICES=("docker" "sshd" "NetworkManager" "fail2ban" "postgresql")
G='\033[32m' R='\033[31m' Y='\033[33m' N='\033[0m'

log() { echo -e "[$(date +%H:%M:%S)] $*"; }

log "═══ Service Health Check ═══"
issues=0

for svc in "${SERVICES[@]}"; do
    if systemctl is-active --quiet "$svc"; then
        log "${G}  ✓ $svc${N}"
    else
        log "${R}  ✗ $svc is DOWN${N}"
        ((issues++))
        log "${Y}    → Attempting restart...${N}"
        if sudo systemctl start "$svc" 2>/dev/null; then
            log "${G}    → Restarted successfully${N}"
        else
            log "${R}    → FAILED to restart! Check: journalctl -u $svc${N}"
        fi
    fi
done

log "Result: $issues issue(s) found"

Exercise 3 (Hard) — Create a System Health Timer

Create a complete systemd timer that runs a health check script every 30 minutes. The script should check CPU temperature (from /sys/class/thermal/), memory usage, disk usage, and failed services, then write a one-line summary to a log file. If any metric exceeds a threshold (temp >80°C, memory >85%, disk >90%), send a desktop notification using notify-send. Include the .service file, .timer file, and the health check script itself.

solution — three files
# ═══ File 1: ~/.local/bin/health-check.sh ═══
#!/bin/bash
set -euo pipefail

LOG="/tmp/health.log"
ALERT=false

# CPU temperature
temp=$(cat /sys/class/thermal/thermal_zone0/temp 2>/dev/null || echo 0)
temp_c=$((temp / 1000))

# Memory usage
mem_pct=$(free | awk '/Mem/{printf "%.0f", $3/$2*100}')

# Disk usage
disk_pct=$(df / | awk 'NR==2{gsub(/%/,""); print $5}')

# Failed services
failed=$(systemctl --failed --no-legend | wc -l)

# Log summary
echo "$(date +%Y-%m-%dT%H:%M:%S) CPU:${temp_c}°C MEM:${mem_pct}% DISK:${disk_pct}% FAILED:${failed}" >> "$LOG"

# Alert on thresholds
[[ $temp_c -gt 80 ]] && ALERT=true
[[ $mem_pct -gt 85 ]] && ALERT=true
[[ $disk_pct -gt 90 ]] && ALERT=true
[[ $failed -gt 0 ]] && ALERT=true

if $ALERT; then
    DISPLAY=:0 notify-send "🌸 System Alert" \
        "CPU:${temp_c}°C MEM:${mem_pct}% DISK:${disk_pct}% Failed:${failed}" \
        --urgency=critical
fi

# ═══ File 2: /etc/systemd/system/health-check.service ═══
[Unit]
Description=Pink Galaxy Health Check

[Service]
Type=oneshot
ExecStart=/home/mari/.local/bin/health-check.sh
User=mari
Environment=DISPLAY=:0 DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/1000/bus

# ═══ File 3: /etc/systemd/system/health-check.timer ═══
[Unit]
Description=Run health check every 30 minutes

[Timer]
OnBootSec=5min
OnUnitActiveSec=30min
Persistent=true

[Install]
WantedBy=timers.target
