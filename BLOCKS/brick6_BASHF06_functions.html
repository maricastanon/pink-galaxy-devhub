BASH-F06.1 — Defining Functions

A function is a named recipe. Instead of writing the same 10 lines of code every time you want to check if a service is running, you wrap those 10 lines in a function called check_service, and then whenever you need that logic, you just write check_service nginx. The function runs all 10 lines for you. If you later discover a bug in those 10 lines, you fix it in one place (the function definition) and every call site is automatically fixed.

Bash offers two ways to define functions: the function name { } style (bash-specific) and the name() { } style (POSIX-compatible). Both do the same thing. The POSIX style is more common and portable, so use that unless you have a reason not to. The important thing is: the function must be defined BEFORE it is called, because bash reads scripts top to bottom.

A function definition creates a named entry in the shell's function hash table. When bash encounters a function call (a word that matches a defined function name), it pushes a new variable scope context, binds the positional parameters ($1, $2, ...) to the function's arguments (saving and restoring the script-level positional parameters), and executes the function body. Functions are NOT separate processes — they execute within the current shell (unlike scripts called by name, which fork a subshell). This means functions can modify global variables, change the working directory, and set shell options — all of which persist after the function returns. The return statement exits the function (not the script) with a numeric exit status (0-255).

Internally, bash stores functions as SHELL_VAR entries in the function hash table (separate from the variable hash table), where the "value" is a pointer to a COMMAND structure — the parsed AST of the function body. When a function is called, execute_function() in execute_cmd.c saves the current dollar_vars[] array (positional parameters), the FUNCNAME, BASH_SOURCE, and BASH_LINENO arrays, pushes a new variable context (push_context() which creates a new scope for local variables), binds the new positional parameters, and calls execute_command(function_body). On return, the variable context is popped (pop_context()), positional parameters are restored, and the return value is set in last_command_exit_value. Functions are not forked — there is no fork() or new PID — they run in the same process, which is why they are much faster than calling external scripts.

bash
#!/bin/bash
# ═══ Two Ways to Define Functions ═══

# Style 1: POSIX (recommended — portable)
greet() {
    echo "🌸 Hello from the greet function!"
}

# Style 2: bash keyword (also fine, just less portable)
function farewell {
    echo "👋 Goodbye from the farewell function!"
}

# Call them (just use the name — no parentheses!)
greet
farewell

# ═══ A Useful Function: Colored Logging ═══
log_info() {
    echo -e "\033[1;32m[INFO]\033[0m $*"
}
log_warn() {
    echo -e "\033[1;33m[WARN]\033[0m $*" >&2
}
log_error() {
    echo -e "\033[1;31m[ERROR]\033[0m $*" >&2
}

log_info "Starting deployment..."
log_warn "Disk usage is at 78%"
log_error "Cannot connect to database"

✅ Functions execute in the current shell:

This means they are fast (no fork/exec overhead) and they share the same variable space, working directory, and environment as the rest of the script. A function that does cd /tmp changes the directory for the entire script, not just inside the function — which is both powerful and dangerous.

❌ Common mistake — calling with parentheses:

In bash, you call functions by name only: greet, not greet(). Writing greet() at the call site is actually a function DEFINITION (with empty body), not a call. This trips up people coming from Python/JavaScript.

BASH-F06.2 — Arguments ($1, $2, $@, $#)

Functions receive arguments the same way scripts do — through positional parameters. When you call greet "Mari" "Galaxy", inside the function, $1 is "Mari" and $2 is "Galaxy". The variable $# tells you how many arguments were passed, and $@ gives you all of them as a list. These are the same special variables you use for script arguments, but inside a function they refer to the function's arguments, not the script's arguments.

Think of it like this: when you call a function, the "incoming mail" ($1, $2, ...) temporarily switches to the function's mail. When the function returns, the script's original mail is restored. This happens automatically — you do not need to save/restore anything.

Positional parameters inside functions shadow the script-level positional parameters for the duration of the function call. $0 is NOT affected — it remains the script name, not the function name (use ${FUNCNAME[0]} for the current function name). The special variables: $# = argument count, $@ = all arguments as separate words (when double-quoted: "$@" preserves each argument as a distinct word), $* = all arguments as a single string (when double-quoted: joined by the first character of IFS). Always use "$@" (not $*) when forwarding arguments to other commands, because it preserves arguments that contain spaces.

bash
#!/bin/bash
# ═══ Function Arguments ═══

# Basic: named arguments via position
create_user() {
    local username="$1"
    local role="${2:-viewer}"    # default to "viewer" if not given
    echo "Creating user: $username (role: $role)"
}

create_user "mari" "admin"     # mari, admin
create_user "luna"              # luna, viewer (default)

# Counting arguments
install_packages() {
    echo "Installing $# packages..."
    for pkg in "$@"; do
        echo "  📦 $pkg"
    done
}

install_packages "nmap" "wireshark" "burp suite"

# Argument validation
backup_dir() {
    if [[ $# -lt 1 ]]; then
        echo "Usage: ${FUNCNAME[0]} " >&2
        return 1
    fi
    local dir="$1"
    if [[ ! -d "$dir" ]]; then
        echo "Error: '$dir' is not a directory" >&2
        return 1
    fi
    local archive="$(basename "$dir")_$(date +%Y%m%d).tar.gz"
    echo "Backing up '$dir' → $archive"
}

backup_dir ~/.config
backup_dir               # no argument → shows usage

  📦 burp suite

BASH-F06.3 — Return Values (exit status vs output capture)

This is one of bash's most confusing aspects for people coming from other languages. In Python, a function returns a value with return x and you capture it with result = my_func(). In bash, return sets an exit status (a number 0-255, where 0 means success), NOT a data value. To "return data" from a bash function, you have two options.

Option 1 (best): The function echos its result, and the caller captures it with command substitution: result=$(my_func). This is the standard pattern for functions that compute and return data.

Option 2: The function sets a global variable that the caller reads. This is simpler but messier — it creates hidden dependencies between the function and the caller.

Use return for success/failure signaling (so the function works with if and &&), and echo for returning data.

The return [n] builtin exits the function with exit status n (default: the exit status of the last command executed). This status is available in $? immediately after the function call. It is an integer in the range 0-255 — not a string, not arbitrary data. To return strings, arrays, or complex data from a function, the function writes to stdout (via echo, printf, or cat), and the caller captures stdout using command substitution: result=$(func). This works because command substitution runs the function in the current shell (not a subshell, unlike pipeline command substitution — this is a special optimization for simple function calls), captures its stdout, and strips trailing newlines.

When return n executes, bash sets last_command_exit_value = n and sets the return_catch_flag, which causes execute_command() to stop executing the function body and return to the caller. The value is strictly limited to 0-255 because it is stored in the low 8 bits of a process exit status (matching the POSIX waitpid() convention). When capturing output via result=$(func), bash creates a pipe, redirects the function's stdout to the write end, executes the function, reads all data from the read end into a buffer, strips trailing newlines (0x0A bytes), and assigns the buffer to the variable. If the function has side effects on stderr, those go directly to the terminal (stderr is NOT captured by command substitution). This is why logging functions should write to stderr while data-returning functions write to stdout.

bash
#!/bin/bash
# ═══ Return: Exit Status (success/failure) ═══

is_installed() {
    command -v "$1" &>/dev/null    # returns 0 if found, 1 if not
}

if is_installed "docker"; then
    echo "✓ Docker found"
else
    echo "✗ Docker not found"
fi

# ═══ Return: Data via stdout (echo + capture) ═══

get_disk_usage() {
    df / | awk 'NR==2 {gsub(/%/,""); print $5}'
}

usage=$(get_disk_usage)
echo "Disk: ${usage}%"

# ═══ Combining both: status + data ═══

fetch_ip() {
    local ip
    ip=$(ip -4 addr show wlan0 2>/dev/null | awk '/inet/{print $2}')
    if [[ -z "$ip" ]]; then
        echo "No IP found" >&2    # error to stderr (not captured)
        return 1                  # signal failure
    fi
    echo "$ip"                    # data to stdout (captured)
    return 0                      # signal success
}

if result=$(fetch_ip); then
    echo "IP address: $result"
else
    echo "Failed to get IP"
fi
Expected Output
✓ Docker found
Disk: 32%
IP address: 192.168.178.42/24
✅ The pattern to memorize:

Functions that check conditions use return 0 (success) or return 1 (failure) — they work with if. Functions that compute data use echo — the caller captures with $(func). Functions that need both send errors to stderr (>&2) and data to stdout, so the caller can capture data while still seeing errors.

❌ The #1 bash function mistake:

Trying to return "hello" — this causes an error because return only accepts integers 0-255. People coming from Python/JS expect return to send back data. In bash, return is for exit codes only. Use echo for data.

🧪 How do you capture the string output of a function get_name into a variable?
result = get_name()
result=$(get_name) — command substitution captures the function's stdout

BASH-F06.4 — Local Variables (Preventing Scope Leaks)

This is a revisit of the scope lesson from BASH-F02, now applied to functions. The rule is simple and absolute: every variable inside a function must be declared local unless you intentionally want it to be global. Without local, any variable you set inside a function silently becomes a global variable that persists after the function returns and can be accidentally overwritten by other functions. This is the single most common source of hard-to-find bugs in bash scripts.

Think of local as putting a privacy fence around your function's variables. Without the fence, your function's variables are sitting in the front yard where anyone can see and trample them.

The local builtin creates a variable in the function's private scope (the variable context pushed by push_context() during function entry). When the function returns, pop_context() destroys all local variables, restoring any previously-shadowed global variables. Without local, assignments create or modify global variables via bind_variable() in the global scope. The local keyword can also set attributes: local -i num=5 (integer), local -r const="fixed" (readonly), local -a arr=() (indexed array), local -A map=() (associative array). Best practice: declare all function variables on the first lines of the function body, just like variable declarations at the top of a C function.

bash
#!/bin/bash
# ═══ The Danger of Missing 'local' ═══

counter=100    # global counter

bad_function() {
    counter=0   # ⚠️ OVERWRITES the global counter!
    temp="leaked"  # ⚠️ Creates a global variable!
}

good_function() {
    local counter=0    # ✓ Shadow — does NOT touch the global
    local temp="safe"   # ✓ Private — dies when function returns
    echo "Inside good: counter=$counter, temp=$temp"
}

echo "Before: counter=$counter"   # 100

bad_function
echo "After bad: counter=$counter, temp=$temp"  # 0, leaked ←BUG!

counter=100  # reset
good_function
echo "After good: counter=$counter"   # 100 ← preserved!

# ═══ Best Practice: Declare All Locals at the Top ═══

process_file() {
    local file="$1"
    local line_count
    local word_count
    local -a errors=()
    
    line_count=$(wc -l < "$file")
    word_count=$(wc -w < "$file")
    
    echo "$file: $line_count lines, $word_count words"
}

BASH-F06.5 — Practical Function Patterns

Real-world scripts use a handful of recurring function patterns. Learning these patterns means you can build any script by assembling the right pieces. Here are the four most common patterns: a guard function (checks prerequisites before the script runs), a cleanup function (runs automatically when the script exits), a retry function (wraps unreliable operations with automatic retries), and a main function (organizes the script's top-level logic into a clean entry point).

The main() pattern is particularly important. Instead of scattering top-level code throughout the file, you define all functions first, then call main "$@" at the very bottom. This ensures all functions are defined before any are called, makes the entry point explicit, and allows the script to be sourced by other scripts without executing side effects (by checking if [[ "${BASH_SOURCE[0]}" == "${0}" ]]). The cleanup pattern uses trap to register a function that runs on EXIT, ensuring temporary files and resources are cleaned up even if the script crashes.

bash
#!/bin/bash
# ═══ Pattern 1: Guard / Prerequisite Check ═══

require() {
    local cmd="$1"
    if ! command -v "$cmd" &>/dev/null; then
        echo "❌ Required command not found: $cmd" >&2
        echo "   Install with: sudo pacman -S $cmd" >&2
        exit 1
    fi
}

require "git"
require "docker"
require "jq"

# ═══ Pattern 2: Cleanup on Exit ═══

cleanup() {
    echo "🧹 Cleaning up..."
    rm -f "$TMPFILE"
    # restore anything you changed
}
trap cleanup EXIT   # runs cleanup on ANY exit (normal, error, Ctrl+C)

TMPFILE=$(mktemp)
echo "Working with temp: $TMPFILE"

# ═══ Pattern 3: Retry Wrapper ═══

retry() {
    local max="$1"; shift
    local delay="$1"; shift
    local attempt=1
    
    while [[ $attempt -le $max ]]; do
        if "$@"; then    # run the command (all remaining arguments)
            return 0
        fi
        echo "  ⏳ Attempt $attempt/$max failed, retrying in ${delay}s..." >&2
        sleep "$delay"
        ((attempt++))
    done
    echo "  ✗ All $max attempts failed" >&2
    return 1
}

# Usage: retry 3 2 curl -sS https://api.example.com/health
# = try 3 times, 2 seconds between retries, running curl...

# ═══ Pattern 4: main() Entry Point ═══

setup() {
    echo "⚙️ Setting up..."
}

deploy() {
    echo "🚀 Deploying..."
}

main() {
    require "git"
    setup
    deploy
    echo "✅ Done!"
}

# Only run main if script is executed directly (not sourced)
if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
    main "$@"
fi
Expected Output
Working with temp: /tmp/tmp.xK3j2mN
⚙️ Setting up...
🚀 Deploying...
✅ Done!
🧹 Cleaning up...
✅ The main() pattern gives you:

Clean script structure, explicit entry point, all functions defined before use, the ability to source the script as a library without executing it, and argument forwarding via "$@". This is how every professional bash script is structured.

❌ Without main():

Top-level code scattered throughout the file, functions called before they are defined (crash), scripts that cannot be sourced by other scripts without unwanted side effects, and no clear starting point for someone reading the code.

Exercise 2 (Medium) — System Report Generator

Write a script with separate functions for each piece of system info: get_hostname, get_kernel, get_uptime, get_disk_usage, get_memory_usage, get_ip. Each function echoes its result (for data capture). A generate_report function calls them all and formats the output. Use the main() pattern.

solution
#!/bin/bash
# ═══ System Report Generator ═══

get_hostname()    { hostname; }
get_kernel()      { uname -r; }
get_uptime()      { uptime -p; }
get_disk()        { df -h / | awk 'NR==2{print $3 "/" $2 " (" $5 ")"}'; }
get_memory()      { free -h | awk '/^Mem/{print $3 "/" $2}'; }

generate_report() {
    local width=40
    printf "╔%${width}s╗\n" | tr ' ' '═'
    printf "║ 🌸 %-$((width-4))s ║\n" "System Report"
    printf "╠%${width}s╣\n" | tr ' ' '═'
    printf "║ Host:   %-$((width-12))s ║\n" "$(get_hostname)"
    printf "║ Kernel: %-$((width-12))s ║\n" "$(get_kernel)"

Exercise 3 (Hard) — Plugin System

Write a script framework where each "plugin" is a function with a standard naming convention (plugin_name). The script auto-discovers all defined plugin_* functions and runs them. Use declare -F to list functions and grep to filter by prefix. Each plugin should echo a status line. Add a retry wrapper for unreliable plugins.

solution
#!/bin/bash
# ═══ Auto-Discovery Plugin System ═══

retry() {
    local max="$1"; shift
    local n=1
    while [[ $n -le $max ]]; do
        "$@" && return 0
        echo "    ↻ retry $n/$max" >&2
        ((n++))
    done
    return 1
}

# ─── Plugins (just define functions with plugin_ prefix) ───
plugin_disk_check() {
    local usage=$(df / | awk 'NR==2{gsub(/%/,"");print $5}')
    [[ $usage -lt 90 ]] && echo "✓ Disk OK (${usage}%)" || { echo "✗ Disk critical"; return 1; }
}

plugin_docker_check() {
    systemctl is-active --quiet docker && echo "✓ Docker running" || { echo "✗ Docker down"; return 1; }
}

plugin_network_check() {
    ping -c1 -W1 8.8.8.8 &>/dev/null && echo "✓ Network OK" || { echo "✗ Network down"; return 1; }
}

# ─── Auto-discover and run all plugins ───
main() {
    echo "═══ Running Health Checks ═══"
    local passed=0 failed=0
    
    for func in $(declare -F | awk '{print $3}' | grep '^plugin_'); do
        local name="${func#plugin_}"   # strip prefix for display
        printf "  %-20s" "$name:"
        if retry 2 "$func"; then
            ((passed++))
        else
            ((failed++))
        fi
    done
    
    echo "═══ Results: $passed passed, $failed failed ═══"
    [[ $failed -eq 0 ]]  # exit 0 if all passed, 1 if any failed
}

[[ "${BASH_SOURCE[0]}" == "${0}" ]] && main "$@"
Output

  disk_check:         ✓ Disk OK (32%)
  docker_check:       ✓ Docker running
  network_check:      ✓ Network OK
═══ Results: 3 passed, 0 failed ═══
