ARCH-CMD05.1 — ip: Your Network Identity

The ip command is the modern Swiss Army knife for network configuration on Linux. It replaces the older ifconfig, route, and arp commands (which you might see in old tutorials but should not use anymore). With ip, you can see your IP addresses, check which network interfaces are active, view the routing table (how your machine decides where to send packets), and manage network connections. On your Arch system, NetworkManager handles most configuration automatically, but ip is how you inspect and troubleshoot the results.

Think of your computer as a house. Each network interface (Wi-Fi card, Ethernet port, VPN tunnel) is a different door. Each door has an address (IP address) written on it. The routing table is your GPS — it tells your computer which door to use to reach any destination on the internet.

The ip command (from the iproute2 package) communicates with the kernel's networking stack via Netlink sockets — a special socket family (AF_NETLINK) for kernel-userspace communication. Subcommands: ip addr (show/modify IP addresses on interfaces), ip link (show/modify interface state: up/down, MTU, MAC address), ip route (show/modify the routing table), ip neigh (show the ARP/NDP neighbor table — maps IP addresses to MAC addresses on the local network). Interface naming on Arch follows the Predictable Network Interface Names scheme: wlan0 or wlp0s20f3 for Wi-Fi (your Intel Wi-Fi card), enp2s0 for Ethernet, lo for loopback (127.0.0.1), docker0 for the Docker bridge network. The -c flag enables colorized output; -br gives a brief, human-readable format.

The ip command opens a Netlink socket (socket(AF_NETLINK, SOCK_DGRAM, NETLINK_ROUTE)), constructs a request message containing a Netlink header (struct nlmsghdr) and a payload specifying the operation (e.g., RTM_GETADDR to list addresses), sends it to the kernel via sendmsg(), and receives the response via recvmsg(). The kernel's networking stack processes the request in net/core/rtnetlink.c, iterates through the appropriate data structures (the network device list for ip link, the FIB table for ip route, the neighbor table for ip neigh), and sends back response messages containing the requested information as Netlink attributes (type-length-value encoded). This is much more efficient than the old ioctl()-based interface used by ifconfig, because Netlink supports multipart messages, notification subscriptions (RTNLGRP_LINK for link state changes), and atomic batch operations.

bash
# ═══ ip addr — See Your IP Addresses ═══

# Show all interfaces and their IPs (the most common command)
ip addr                                # full output
ip -br -c addr                        # brief + colorized (much cleaner!)

# Show only IPv4 addresses
ip -4 addr

# Show a specific interface
ip addr show wlan0                    # your Wi-Fi interface

# Just get your IP (for scripts)
ip -4 -br addr | awk '/wlan/ {print $3}'

# ═══ ip link — Interface Status ═══

ip -br -c link                        # shows UP/DOWN state of each interface

# ═══ ip route — Where Do Packets Go? ═══

ip route                               # show routing table
# Key line: "default via 192.168.178.1 dev wlan0" ← your FRITZ!Box gateway

# How would the system reach a specific IP?
ip route get 8.8.8.8                  # shows which interface and gateway it would use

# ═══ ip neigh — Who Else is on the Local Network? ═══

ip neigh                               # ARP table: IP → MAC address mappings
# Shows your FRITZ!Box, other devices on the same Wi-Fi network

# ═══ Quick Network Summary (one-liner) ═══
echo "IP: $(ip -4 -br addr | awk '/wlan/{print $3}') GW: $(ip route | awk '/default/{print $3}') DNS: $(resolvectl status | awk '/DNS Server/{print $NF}' | head -1)"

ARCH-CMD05.2 — ss: Open Ports & Active Connections

ss (socket statistics) shows you every network connection on your system — which programs are listening for incoming connections, which are actively communicating with remote servers, and on which ports. This is essential for security (checking what services are exposed to the network), debugging (is my web server actually listening on port 8080?), and troubleshooting (why is "port already in use" when I try to start my application?). It replaces the older netstat command which is slower and less capable.

The ss command reads socket information directly from kernel Netlink interfaces (NETLINK_SOCK_DIAG), which is significantly faster than netstat (which parses /proc/net/tcp text files). Key flags: -t (TCP only), -u (UDP only), -l (listening sockets — services waiting for connections), -n (numeric — do not resolve hostnames, much faster), -p (show process name/PID — requires root or matching UID), -a (all sockets including non-listening). The most useful combination is ss -tlnp (TCP listening sockets with numeric addresses and process info), which answers "what services are listening on my machine and on which ports?" Socket states: LISTEN (waiting for connections), ESTAB (active connection), TIME-WAIT (connection recently closed, waiting for straggler packets), CLOSE-WAIT (remote closed, local has not yet closed — can indicate a bug).

bash
# ═══ ss — What's Listening? (THE command for port troubleshooting) ═══

# Show all TCP listening sockets with process info
sudo ss -tlnp
# t=TCP, l=listening, n=numeric, p=process
# This answers: "What services are exposed on my machine?"

# Show all active TCP connections
ss -tn

# Check if a specific port is in use
ss -tlnp | grep ":22"              # is SSH listening?
ss -tlnp | grep ":8080"            # is something on port 8080?

# What process is using port 5432? (PostgreSQL)
sudo ss -tlnp | grep ":5432"
# Output: LISTEN  0  128  *:5432  *:*  users:(("postgres",pid=890,fd=7))

# ═══ Useful Combinations ═══

# All connections to a specific remote host
ss -tn dst 140.82.121.4            # connections to GitHub

# Count connections by state
ss -t | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn

# All UDP listeners (DNS, DHCP, etc.)
sudo ss -ulnp
Expected Output (sudo ss -tlnp)
State    Recv-Q  Send-Q  Local Address:Port   Peer Address:Port  Process
LISTEN   0       128     0.0.0.0:22           0.0.0.0:*          users:(("sshd",pid=432,fd=3))
LISTEN   0       128     127.0.0.1:5432       0.0.0.0:*          users:(("postgres",pid=890,fd=7))
LISTEN   0       4096    0.0.0.0:6379         0.0.0.0:*          users:(("redis-server",pid=901,fd=6))
LISTEN   0       4096    127.0.0.1:631        0.0.0.0:*          users:(("cupsd",pid=445,fd=8))
✅ Reading ss output for security:

0.0.0.0:22 means SSH is listening on ALL interfaces (accessible from the network). 127.0.0.1:5432 means PostgreSQL is listening only on localhost (not accessible from the network — correct for a dev setup). If you see an unexpected service on 0.0.0.0, it is exposed to your network and potentially the internet.

❌ "Address already in use" error:

When starting a service and getting this error, run sudo ss -tlnp | grep :PORT to find what is already using that port. Either stop the conflicting service, or configure your application to use a different port. A common cause is a crashed process that left a socket in TIME-WAIT state — wait 60 seconds or set SO_REUSEADDR in the application.

ARCH-CMD05.3 — ping, traceroute, mtr: Connectivity Testing

ping is the most basic network test — it sends a tiny packet to a destination and measures how long the reply takes. If ping works, the destination is reachable and the round-trip time tells you how fast the connection is. If ping fails, something between you and the destination is broken. traceroute (or its modern replacement mtr) shows you every hop your packets take on the way to the destination, so you can see exactly WHERE a problem is occurring — is it your Wi-Fi, your FRITZ!Box, your ISP, or the destination server?

The ping command sends ICMP Echo Request packets and measures the ICMP Echo Reply. Key flags: -c N (send exactly N packets then stop — without this, ping runs forever), -W N (timeout in seconds), -i N (interval between packets). The traceroute command sends packets with incrementing TTL (Time-To-Live) values starting from 1. Each router along the path decrements TTL by 1 and sends back a "TTL exceeded" ICMP message when it hits 0 — this reveals that router's IP address. The mtr command (My TraceRoute, install with pacman -S mtr) combines ping and traceroute into a single live display, showing each hop with continuous ping statistics (packet loss, latency, jitter). This is the professional tool for diagnosing routing problems.

bash
# ═══ ping — Is It Reachable? ═══

# Test connectivity (Ctrl+C to stop)
ping google.com                      # runs until you stop it
ping -c 4 google.com                 # send exactly 4 pings then stop
ping -c 1 -W 2 192.168.178.1        # ping FRITZ!Box, 2s timeout

# The diagnostic ladder (test each layer):
ping -c 1 127.0.0.1                  # 1. Loopback (your own machine)
ping -c 1 192.168.178.1              # 2. FRITZ!Box gateway
ping -c 1 8.8.8.8                    # 3. External IP (Google DNS)
ping -c 1 google.com                  # 4. DNS name (tests DNS resolution too)
# If step 3 works but step 4 fails → DNS problem
# If step 2 works but step 3 fails → ISP/routing problem
# If step 1 works but step 2 fails → local network/Wi-Fi problem

# ═══ traceroute / mtr — WHERE is the Problem? ═══

traceroute google.com                 # show every hop to Google
mtr google.com                        # live traceroute + ping combined (best tool)
mtr -r -c 10 github.com              # report mode: 10 pings per hop, then exit

ARCH-CMD05.4 — curl & wget: HTTP from the Command Line

curl lets you make HTTP requests from the terminal — the same requests your web browser makes, but visible and scriptable. You can download files, call APIs, submit forms, test web servers, and debug HTTP communication. It supports dozens of protocols (HTTP, HTTPS, FTP, SFTP, SCP, etc.) and is the universal tool for interacting with web services from scripts. wget is simpler and focused on downloading files (it can resume interrupted downloads and mirror entire websites). Use curl for API interaction and debugging, wget for bulk downloads.

The curl command uses libcurl to perform URL transfers. Key flags: -o file (save output to file), -O (save with server's filename), -L (follow HTTP redirects — essential for most URLs), -s (silent — no progress bar), -S (show errors even in silent mode), -I (HEAD request — show only headers), -v (verbose — show the entire HTTP conversation including headers and TLS handshake), -X METHOD (set HTTP method: GET, POST, PUT, DELETE), -H "Header: value" (add custom headers), -d "data" (send POST data), -u user:pass (HTTP basic auth), -k (skip TLS certificate verification — use only for testing). For API work, -H "Content-Type: application/json" -d '{"key":"value"}' is the standard pattern for sending JSON.

bash
# ═══ curl — HTTP Swiss Army Knife ═══

# Fetch a web page (output to terminal)
curl https://archlinux.org

# Download a file
curl -LO https://github.com/user/repo/archive/main.tar.gz
# -L follows redirects, -O saves with the original filename

# See HTTP headers only (check if a server is responding)
curl -sI https://github.com | head -10

# Get your public IP address
curl -s https://ifconfig.me

# Test API endpoint (GET request with JSON response)
curl -sS https://api.github.com/users/torvalds | jq '.name, .bio'

# POST request with JSON body
curl -sS -X POST \
  -H "Content-Type: application/json" \
  -d '{"title": "Test", "body": "Hello from curl"}' \
  https://httpbin.org/post | jq '.json'

# Verbose mode — debug the entire HTTP conversation
curl -v https://google.com 2>&1 | head -30
# Shows: DNS resolution, TCP connect, TLS handshake, HTTP request/response

# ═══ wget — Download Specialist ═══

# Download a file (resume-capable by default)
wget https://archlinux.org/iso/latest/archlinux-x86_64.iso

# Resume an interrupted download
wget -c https://example.com/largefile.iso

# Download quietly (for scripts)
wget -q -O /tmp/data.json https://api.example.com/data
Expected Output (curl -sI https://github.com | head -5)
HTTP/2 200
server: GitHub.com
date: Sat, 15 Mar 2026 15:42:00 GMT
content-type: text/html; charset=utf-8
x-frame-options: DENY
✅ curl for daily use:

curl -sS (silent + show errors) is the standard for scripts. curl -v is your debugging tool when an API is not behaving as expected. Always pipe JSON responses through jq for readable output. Always use -L for web URLs (most sites redirect HTTP→HTTPS).

❌ curl vs wget:

Do not use wget for API interaction — it is designed for downloading files, not HTTP debugging. Do not use curl for mirroring websites — wget has --mirror mode designed for that. Use the right tool for the right job.

ARCH-CMD05.5 — ssh: Secure Remote Access

ssh (Secure Shell) lets you log into another computer's terminal over the network, as if you were sitting at its keyboard. All communication is encrypted, so nobody between you and the server can see what you are typing (including passwords, code, and commands). Beyond remote shells, SSH also does port forwarding (tunneling other protocols through the encrypted connection), file transfers (via scp and sftp), and serves as the transport layer for Git operations with GitHub. In Part 1 of your Arch guide, you configured the SSH server and set up key-based authentication — this section shows you how to use the client side effectively.

SSH key authentication uses asymmetric cryptography: you generated a key pair (ed25519) in Part 3 — the private key (~/.ssh/id_ed25519, permissions 600) stays on your machine, the public key (~/.ssh/id_ed25519.pub) is placed on remote servers. During authentication, the server sends a challenge, your SSH client signs it with the private key, and the server verifies the signature against the stored public key. No password ever crosses the network. The ~/.ssh/config file lets you define named connections with preconfigured hostnames, usernames, ports, and key files — so you can type ssh myserver instead of ssh -i ~/.ssh/special_key -p 2222 mari@192.168.178.100. Port forwarding: -L localport:remotehost:remoteport (local forward — access a remote service through a local port) and -R remoteport:localhost:localport (remote forward — expose a local service to the remote network).

bash
# ═══ ssh — Connect to a Remote Machine ═══

# Basic connection
ssh mari@192.168.178.100
ssh user@example.com

# Specify a port (if not default 22)
ssh -p 2222 mari@server.com

# Run a single command remotely (no interactive shell)
ssh mari@server "df -h && free -h"

# ═══ ~/.ssh/config — Named Connections (THE pro move) ═══
# Create/edit ~/.ssh/config:
# Host myserver
#     HostName 192.168.178.100
#     User mari
#     Port 22
#     IdentityFile ~/.ssh/id_ed25519
#
# Host github
#     HostName github.com
#     User git
#     IdentityFile ~/.ssh/github_ed25519

# Now you can just type:
ssh myserver                          # connects with all the right settings

# ═══ scp / sftp — File Transfer ═══

# Copy file TO a remote machine
scp report.pdf mari@server:/home/mari/Documents/

# Copy file FROM a remote machine
scp mari@server:/var/log/app.log ./

# Copy entire directory recursively
scp -r project/ mari@server:~/projects/

# Interactive file browser (sftp — like FTP but encrypted)
sftp mari@server

# ═══ Port Forwarding — SSH Tunnels ═══

# Access remote PostgreSQL (port 5432) through local port 5433
ssh -L 5433:localhost:5432 mari@server
# Now: psql -h localhost -p 5433 → connects to the REMOTE database

# Access remote web app (port 8080) locally
ssh -L 8080:localhost:8080 mari@server
# Now: open http://localhost:8080 in your browser → shows remote app
🧪 You need to access a PostgreSQL database running on a remote server (only listening on localhost:5432) from your local machine. How?
SSH port forwarding: ssh -L 5433:localhost:5432 user@server — this creates an encrypted tunnel, and connecting to localhost:5433 on your machine reaches the remote PostgreSQL on port 5432
Configure PostgreSQL to listen on 0.0.0.0 (dangerous — exposes it to the internet)
It is not possible to access services only listening on localhost

ARCH-CMD05.6 — dig & DNS: Name Resolution

DNS (Domain Name System) is the phone book of the internet — it translates human-readable names like github.com into machine-readable IP addresses like 140.82.121.4. Every time you visit a website, run pacman -Syu, or push to Git, your system performs a DNS lookup first. The dig command (Domain Information Groper) lets you perform DNS lookups manually and see exactly what is happening — which server answered, what records were returned, and how long it took. When websites are not loading but you can ping IP addresses directly, DNS is almost always the problem, and dig is how you diagnose it.

DNS record types you will encounter: A (IPv4 address for a hostname), AAAA (IPv6 address), CNAME (alias — one name pointing to another name), MX (mail server for a domain), NS (nameservers for a domain), TXT (arbitrary text — used for SPF, DKIM, domain verification). The dig command queries DNS servers directly using UDP port 53 (or TCP for large responses). On your Arch system, systemd-resolved handles DNS with the configuration visible via resolvectl status. Your FRITZ!Box likely provides DNS via DHCP (usually forwarding to your ISP's DNS or to its own recursive resolver). For faster and more private DNS, you can configure systemd-resolved to use DNS-over-TLS with providers like Cloudflare (1.1.1.1) or Quad9 (9.9.9.9).

bash
# ═══ dig — DNS Debugging ═══

# Basic lookup (A record — the IP address)
dig github.com

# Short answer only (for scripts)
dig +short github.com                # just the IP address

# Specific record types
dig github.com MX                   # mail servers
dig github.com NS                   # nameservers
dig github.com TXT                  # TXT records (SPF, verification)
dig github.com AAAA                 # IPv6 address

# Query a specific DNS server (bypass your local resolver)
dig @1.1.1.1 github.com             # ask Cloudflare's DNS
dig @8.8.8.8 github.com             # ask Google's DNS

# Reverse lookup (IP → hostname)
dig -x 140.82.121.4

# ═══ Simpler Alternative: host ═══
host github.com                       # quick and human-readable
host 140.82.121.4                     # reverse lookup

# ═══ Check Your DNS Configuration ═══
resolvectl status                     # systemd-resolved config
cat /etc/resolv.conf                  # the classic DNS config file
Expected Output (dig +short github.com)
140.82.121.4

Exercise 1 (Easy) — Network Status Report

Write a script that prints a clean network status report: your hostname, your Wi-Fi IP, your public IP (via curl), your default gateway, your DNS server, and the latency to google.com.

solution
#!/bin/bash
set -euo pipefail

printf "═══ 🌸 Network Status ═══\n\n"
printf "  %-16s %s\n" "Hostname:" "$(hostname)"
printf "  %-16s %s\n" "Local IP:" "$(ip -4 -br addr | awk '/wlan/{print $3}')"
printf "  %-16s %s\n" "Public IP:" "$(curl -sS --max-time 3 https://ifconfig.me || echo 'unreachable')"
printf "  %-16s %s\n" "Gateway:" "$(ip route | awk '/default/{print $3}')"
printf "  %-16s %s\n" "DNS:" "$(resolvectl status 2>/dev/null | awk '/DNS Servers/{print $NF; exit}' || awk '/nameserver/{print $2; exit}' /etc/resolv.conf)"

latency=$(ping -c 1 -W 2 google.com 2>/dev/null | awk -F'[=/]' '/avg/{printf "%.1f ms", $5}') || latency="timeout"
printf "  %-16s %s\n" "Latency:" "$latency"

Exercise 2 (Medium) — Port Scanner

Write a script that scans a given host for common service ports (22, 80, 443, 3000, 5432, 6379, 8080, 8443, 9090) using bash's built-in /dev/tcp feature (from BASH-F05). Report which are open and which are closed, with a summary count. Add a timeout so closed ports do not hang the script.

solution
#!/bin/bash
set -euo pipefail

host="${1:-localhost}"
ports=(22 80 443 3000 5432 6379 8080 8443 9090)
open=0 closed=0

printf "Scanning %s...\n\n" "$host"
printf "%-8s  %-8s  %s\n" "PORT" "STATE" "SERVICE"
printf "%-8s  %-8s  %s\n" "────" "─────" "───────"

# Simple service name lookup
svc_name() {
    case "$1" in
        22) echo "ssh";;       80) echo "http";;
        443) echo "https";;    3000) echo "dev-server";;
        5432) echo "postgres";; 6379) echo "redis";;
        8080) echo "http-alt";; 8443) echo "https-alt";;
        9090) echo "prometheus";; *) echo "unknown";;
    esac
}

for port in "${ports[@]}"; do
    if (echo >/dev/tcp/$host/$port) 2>/dev/null; then
        printf "%-8s  \033[32m%-8s\033[0m  %s\n" "$port" "OPEN" "$(svc_name "$port")"
        ((open++))
    else
        printf "%-8s  %-8s  %s\n" "$port" "closed" "$(svc_name "$port")"
        ((closed++))
    fi
done

printf "\n%d open, %d closed (of %d scanned)\n" "$open" "$closed" "${#ports[@]}"

Exercise 3 (Hard) — Network Diagnostic Ladder

Write a comprehensive network diagnostic script that runs the diagnostic ladder from Section 3 (loopback → gateway → external IP → DNS) automatically, tests specific services (GitHub, Arch mirrors, Docker Hub), checks for DNS resolution speed, tests download speed with a small file, and produces a color-coded report with pass/fail for each test. Use the production script template from BASH-F08.

solution
#!/bin/bash
set -euo pipefail
G='\033[32m' R='\033[31m' Y='\033[33m' N='\033[0m'
pass=0 fail=0

check() {
    local label="$1"; shift
    printf "  %-30s" "$label"
    if "$@" &>/dev/null; then
        printf "${G}✓ PASS${N}\n"; ((pass++))
    else
        printf "${R}✗ FAIL${N}\n"; ((fail++))
    fi
}

echo "═══ 🌸 Network Diagnostic ═══"
echo ""
echo "--- Layer 1: Connectivity ---"
check "Loopback"            ping -c1 -W1 127.0.0.1
check "Gateway (FRITZ!Box)"  ping -c1 -W2 $(ip route | awk '/default/{print $3}')
check "External IP (8.8.8.8)" ping -c1 -W3 8.8.8.8
check "DNS Resolution"      dig +short google.com

echo ""
echo "--- Layer 2: Services ---"
check "GitHub"              curl -sS --max-time 5 https://github.com
check "Arch Mirrors"        curl -sS --max-time 5 https://archlinux.org
check "Docker Hub"          curl -sS --max-time 5 https://hub.docker.com
check "PyPI"               curl -sS --max-time 5 https://pypi.org

echo ""
echo "--- Layer 3: Performance ---"
dns_ms=$(dig github.com | awk '/Query time/{print $4}')
printf "  %-30s %s ms\n" "DNS lookup time:" "${dns_ms:-?}"

dl_speed=$(curl -sS -o /dev/null -w '%{speed_download}' --max-time 10 https://speed.cloudflare.com/__down?bytes=1000000 2>/dev/null) || dl_speed=0
dl_mbps=$(echo "$dl_speed" | awk '{printf "%.1f", $1/125000}')
printf "  %-30s %s Mbps\n" "Download speed:" "$dl_mbps"

echo ""
printf "═══ Result: ${G}%d passed${N}, ${R}%d failed${N} ═══\n" "$pass" "$fail"
