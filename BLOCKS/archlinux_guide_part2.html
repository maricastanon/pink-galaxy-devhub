8.1 Wayland vs Xorg — Why Wayland

Layman version: Think of the display server as a translator between your programs (which want to draw windows) and your screen (which needs to know what pixels to show). Xorg (X11) is the old translator — born in 1984, it is like a medieval messenger system with complex handshakes and no security guards. Wayland is the modern replacement — leaner, faster, more secure, and built for today's hardware. Hyprland is a Wayland compositor, meaning it handles both the window management (tiling, floating, moving windows) and the display compositing (layering windows, transparency, animations) in a single program.

Technical detail: In the X11 architecture, the X server owns the display hardware and clients (applications) send drawing requests via the X protocol over a Unix socket. The window manager is a separate client that arranges windows but cannot control compositing directly — that requires yet another program (like picom/compton). This three-layer architecture (X server → WM → compositor) creates latency and tearing because frame presentation is not synchronized. Wayland eliminates this by merging the server, WM, and compositor into a single "compositor" process. Clients render their own content into shared-memory buffers (using EGL/Vulkan), then the compositor takes those buffers and composites them onto the output using DRM/KMS (Direct Rendering Manager / Kernel Mode Setting). This yields lower latency, eliminates tearing by design (atomic page-flipping), provides per-client security isolation (no key-logging between clients), and supports fractional scaling natively for HiDPI displays — all of which matter on your Dell 7420 with its touchscreen.

The qt5-wayland and qt6-wayland packages are essential because VS Code (Electron-based) and other Qt applications need them to run natively on Wayland instead of falling back to XWayland (the X11 compatibility layer). 8.3 First Launch & Basic Configuration

Layman version: Right now you have a black screen with a blinking cursor (TTY). We are going to type one command that launches Hyprland, which will transform that text console into a tiling window manager with wallpapers, transparency, and animations. The first time you launch it, Hyprland creates a default config file that we will then customize.

bash
# Launch Hyprland from the TTY (you must be logged in as your user, NOT root)
Hyprland

# If it works, you'll see a tiled desktop. Press Super+Q to open Kitty terminal.
# If the screen is blank or crashes, check logs:
cat ~/.local/share/hyprland/hyprland.log
💡 AUTO-START HYPRLAND ON LOGIN

Add this to the end of your ~/.bash_profile (or ~/.zprofile after switching to zsh):

bash — ~/.bash_profile
# Auto-start Hyprland on TTY1 login
if [ -z "$DISPLAY" ] && [ "$XDG_VTNR" -eq 1 ]; then
  exec Hyprland
fi

This checks if there is no display server running ($DISPLAY is empty) and if you are on virtual terminal 1 (the default login TTY). If both conditions are true, it replaces the shell process with Hyprland via exec. The exec is important — it means when Hyprland exits, you go back to the login prompt rather than having an orphaned shell.

The exec-once directives run only on the first launch, not on config reload. The binde keyword (vs bind) means the key can be held and repeated, which is why we use it for volume and brightness controls.

9.3 Shell Plugins & Aliases

The aliases defined in .zshrc above replace legacy tools with modern Rust-based alternatives. Here is what each one does: eza replaces ls with colors, icons, and Git status in every listing. bat replaces cat with syntax highlighting for over 200 languages. ripgrep replaces grep and is 10-100x faster because it uses SIMD-optimized string search algorithms and skips binary files and .gitignored paths by default. fd replaces find with a simpler syntax and faster performance (also Rust-based). zoxide replaces cd by learning your most-visited directories and letting you jump to them with fuzzy matching — type z proj and it takes you to ~/projects.

10.1 PipeWire Audio Setup

Layman version: PipeWire is the modern audio and video routing system for Linux. It replaces both PulseAudio (for desktop audio) and JACK (for professional audio) with a single, unified daemon. Think of it as the sound mixing board for your entire system — every application sends its audio through PipeWire, which mixes it and sends it to your speakers, headphones, or Bluetooth headset.

bash
# Install PipeWire and its compatibility layers
sudo pacman -S pipewire pipewire-pulse pipewire-alsa pipewire-jack wireplumber

# PipeWire runs as a user service (NOT system). Enable it:
systemctl --user enable --now pipewire pipewire-pulse wireplumber

# Install GUI volume control
sudo pacman -S pavucontrol

# Test audio
speaker-test -c 2                  # plays white noise from left/right speakers

# Check PipeWire status
wpctl status                       # shows all audio sinks, sources, and their connections
Expected Output (wpctl status)
PipeWire 'pipewire-0' [1.x, user@pinkgalaxy, cookie:xxxx]
 └─ Clients:
        31. WirePlumber                 [1.x]
        32. wpctl                       [1.x]
        ...
Audio
 ├─ Sinks:
 │      46. Built-in Audio [vol: 0.74]    ← your laptop speakers
 │     *47. USB Headset [vol: 0.80]       ← headset (if connected)
 └─ Sources:
        48. Built-in Audio                ← internal mic

Technical detail: PipeWire uses a graph-based architecture where audio nodes are connected via links. The WirePlumber session manager automatically creates and manages these links based on routing policies. When you plug in a USB headset, WirePlumber detects the new ALSA device, creates a PipeWire node for it, and reroutes all audio streams from the built-in speakers to the headset. The pipewire-pulse package creates a PulseAudio-compatible socket so existing applications that use the PulseAudio API (Firefox, Discord, etc.) work without any changes. For your Dell 7420, the sof-firmware package (installed in Part 1) loads the audio DSP firmware into the Intel SOF chip during boot — without it, PipeWire sees no audio devices.

10.2 Bluetooth — Keyboard, Mouse, Headset

bash
# Install Bluetooth stack + GUI manager
sudo pacman -S bluez bluez-utils blueman

# Enable and start the Bluetooth service
sudo systemctl enable --now bluetooth

# ═══ METHOD 1: GUI (blueman) ═══
# blueman-applet is already in Hyprland autostart
# Click the Bluetooth icon in Waybar system tray
# Or run: blueman-manager

# ═══ METHOD 2: CLI (bluetoothctl) ═══
bluetoothctl
# Inside the interactive shell:
power on                           # turn on Bluetooth radio
agent on                           # enable pairing agent
default-agent
scan on                            # start scanning for devices
# Wait for your device to appear, note its MAC address (XX:XX:XX:XX:XX:XX)
pair XX:XX:XX:XX:XX:XX             # pair with device
connect XX:XX:XX:XX:XX:XX          # connect
trust XX:XX:XX:XX:XX:XX            # auto-connect in future
exit
Expected Scan Output
[NEW] Device AA:BB:CC:DD:EE:FF Logitech MX Keys
[NEW] Device 11:22:33:44:55:66 WH-1000XM4
[NEW] Device 77:88:99:AA:BB:CC Logitech MX Master 3S

When you pair a Bluetooth keyboard, the kernel's btusb module communicates with the Intel Bluetooth firmware (loaded from linux-firmware) via HCI (Host Controller Interface) commands. The bluez daemon manages the pairing handshake (including PIN exchange for legacy devices and Secure Simple Pairing for BLE 4.0+ devices), creates a /dev/input/eventN device via the uhid kernel module, and Hyprland's libinput backend picks it up automatically as a keyboard/mouse input. The trust command stores the device's link key in /var/lib/bluetooth/, enabling automatic reconnection on future boots.

	1920x1080@60.00 at 0x0

The monitor configuration in hyprland.conf (section 8.3) already handles the basic layout. The first entry monitor = eDP-1, preferred, 0x0, 1 tells Hyprland to place the internal laptop display at coordinates (0,0) which is the top-left origin. The second entry places the HDMI monitor at (1920,0), which means immediately to the right of the laptop screen. If you want the external monitor above the laptop, change it to monitor = HDMI-A-1, preferred, 0x-1440, 1 (negative Y places it above). You can also use nwg-displays for a visual drag-and-drop layout editor.

For automatic switching when you plug/unplug HDMI, create a kanshi config:

bash — ~/.config/kanshi/config
# Laptop only
profile undocked {
    output eDP-1 enable mode 1920x1080 position 0,0
}

# Laptop + external HDMI
profile docked {
    output eDP-1 enable mode 1920x1080 position 0,0
    output HDMI-A-1 enable mode 2560x1440 position 1920,0
}

10.4 Elgato Stream Deck Setup

Layman version: On Windows/macOS, Elgato provides the Stream Deck software. On Linux, the community built streamdeck-ui, which provides the same functionality — assign icons, actions, and multi-actions to each button. Your Stream Deck+ with its LCD keys and rotary encoders is fully supported.

sudo tee /etc/udev/rules.d/70-streamdeck.rules << 'EOF'
# Elgato Stream Deck Original
SUBSYSTEM=="usb", ATTRS{idVendor}=="0fd9", ATTRS{idProduct}=="0060", TAG+="uaccess"
# Elgato Stream Deck Mini
SUBSYSTEM=="usb", ATTRS{idVendor}=="0fd9", ATTRS{idProduct}=="0063", TAG+="uaccess"

Technical detail: The Stream Deck communicates via USB HID (Human Interface Device) protocol. Each button press sends an HID input report to the host. The host sends HID output reports to update the button LCD displays (each button has a tiny 72×72 or 96×96 pixel screen). The hidapi library provides a cross-platform API to read/write these HID reports. The streamdeck-ui app runs a background daemon that listens for button presses and executes the configured actions (shell commands, hotkeys, or D-Bus calls). The udev rule with TAG+="uaccess" grants the currently logged-in user (via systemd-logind) access to the USB device without needing root or adding yourself to special groups.

# ═══ SOLAAR — Logitech Unifying/Bolt Receiver Manager ═══
sudo pacman -S solaar

# Launch Solaar (already in Hyprland autostart as 'solaar --window=hide')

# Solaar shows:
#   - All paired Logitech devices
#   - Battery levels
#   - DPI settings (for mice)
#   - Button/key remapping
#   - Smart Shift / Smooth Scrolling toggle

# ═══ PIPER — Advanced Mouse Configuration GUI ═══
sudo pacman -S libratbag piper

# Start the ratbagd daemon (required by Piper)
sudo systemctl enable --now ratbagd

# Launch Piper for detailed mouse config
piper

# Piper provides:
#   - DPI profiles (multiple DPI levels, switch with button)
#   - Button mapping (assign any key/macro to any button)
#   - LED color customization
#   - Scroll speed and direction
#   - Angle snapping and acceleration curves

# ═══ KEYD — System-Level Key Remapping ═══
paru -S keyd
sudo systemctl enable --now keyd

Solaar communicates with Logitech receivers via the HID++ protocol, a proprietary extension that Logitech devices use for advanced features like per-device DPI, battery reporting, and Smart Shift (where the scroll wheel switches between ratchet and free-spin based on scroll speed). Piper and libratbag use the ratbagd daemon which directly writes to the mouse's onboard memory to store DPI profiles and button mappings — these persist even if you plug the mouse into a different computer.

For key remapping, keyd operates at the evdev level (between the kernel's input subsystem and userspace), which means it works for every application including Wayland compositors, TTY consoles, and even the login screen. Example config at /etc/keyd/default.conf:

conf — /etc/keyd/default.conf
[ids]
*                                  # apply to all keyboards

[main]
capslock = overload(control, esc)  # Caps = Ctrl when held, Esc when tapped
