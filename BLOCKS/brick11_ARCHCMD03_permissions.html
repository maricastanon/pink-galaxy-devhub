ARCH-CMD03.1 — Reading Permission Strings

When you run ls -l, the first column shows a 10-character string like -rwxr-xr--. This string is a complete access control specification. It answers three questions for three groups of people. The three questions are: Can you read it? Can you write to it? Can you execute it? The three groups are: the file's owner (the person who created it), the file's group (a team of users), and others (everyone else on the system).

Think of it like a building with three security zones. The owner is in the executive suite — they might have full access (rwx). The group is employees on the same floor — they can probably read and execute but not modify (r-x). Others are visitors in the lobby — they might only be able to read (r--) or have no access at all (---).

The 10-character permission string decomposes as: position 0 is the file type (- regular file, d directory, l symlink, c character device, b block device, p pipe, s socket), positions 1-3 are owner permissions (rwx), 4-6 are group permissions, 7-9 are other permissions. Each position is either the letter (permission granted) or a dash (permission denied). For directories, the meaning changes subtly: r means you can list the directory contents (readdir), w means you can create/delete files in the directory (modify directory entries), and x means you can traverse the directory (cd into it, access files inside by name). This means a directory with --x lets you access files if you know their names, but you cannot list what is inside — a useful security technique.

Permissions are stored in the inode's i_mode field — a 16-bit integer. The low 12 bits encode permissions: bits 0-2 are other (rwx), bits 3-5 are group, bits 6-8 are owner, and bits 9-11 are the special bits (sticky, setgid, setuid). Bits 12-15 encode the file type (regular, directory, symlink, etc.). When a process attempts a file operation (open for read, open for write, execute), the kernel's inode_permission() function in fs/namei.c checks: if the process's effective UID matches the inode's i_uid, the owner bits are checked. Otherwise if any of the process's GIDs (primary + supplementary) match the inode's i_gid, the group bits are checked. Otherwise the other bits are checked. If the effective UID is 0 (root), all permission checks are bypassed (CAP_DAC_OVERRIDE capability). This three-tier lookup happens on every single file access operation — thousands of times per second on a busy system.

visual breakdown
    ┌─ file type (- = regular file, d = directory, l = symlink)
    │  ┌── owner permissions (user who owns the file)
    │  │     ┌── group permissions (users in the file's group)
    │  │     │     ┌── other permissions (everyone else)
    │  │     │     │
    -  rwx  r-x  r--

    r = read    (files: view contents | dirs: list files)
    w = write   (files: modify contents | dirs: create/delete files)
    x = execute (files: run as program | dirs: cd into / traverse)
    - = denied  (permission not granted)

Example: -rwxr-xr--
  Owner (mari):  rwx = can read, write, and execute
  Group (dev):   r-x = can read and execute, but NOT modify
  Others:        r-- = can only read, nothing else

Every permission triplet (rwx) maps to a 3-bit binary number, which maps to an octal digit. This is where the "magic numbers" like 755 and 644 come from — they are just the three octal digits for owner, group, and other.

PermissionBinaryOctalMeaning
---0000No access
--x0011Execute only
-w-0102Write only
-wx0113Write + execute
r--1004Read only
r-x1015Read + execute
rw-1106Read + write
rwx1117Full access

So chmod 754 means: owner gets rwx (7), group gets r-x (5), others get r-- (4). The mental math is simple: r=4, w=2, x=1. Add them up for each group. Read+write = 4+2 = 6. Read+execute = 4+1 = 5. Full access = 4+2+1 = 7.

✅ The four permission numbers you use 90% of the time:

755 — directories and executable scripts (owner: full, group+others: read+traverse). 644 — regular files (owner: read+write, group+others: read only). 700 — private directories (owner only). 600 — private files like SSH keys (owner read+write only).

❌ Permission mistakes that break things:

777 on anything — everyone can read, write, and execute. This is a security disaster. SSH keys at 644 — SSH refuses to use keys that are group/other readable (requires 600). Scripts without x — "permission denied" when trying to run them.

ARCH-CMD03.2 — chmod: Changing Permissions

chmod (change mode) modifies a file's permission bits. You can use it in two ways: octal mode where you specify the complete permission set as a three-digit number (chmod 755 file), or symbolic mode where you add or remove specific permissions (chmod +x file adds execute permission). Octal mode is precise and unambiguous — you set all nine permission bits at once. Symbolic mode is more readable for simple changes like "make this script executable" without changing anything else.

Symbolic mode syntax: chmod [ugoa][+-=][rwxXst] file. The first letter selects who: u (owner), g (group), o (others), a (all three). The operator: + (add permission), - (remove), = (set exactly). The permission letters: r, w, x, plus X (execute only if it is a directory or already has execute for someone — useful for recursive chmod that should make directories traversable without making all files executable), s (setuid/setgid), t (sticky bit). The -R flag applies recursively. The chmod command calls the chmod() syscall which modifies the inode's i_mode field directly. Only the file owner and root can change permissions.

bash
# ═══ Octal Mode (set ALL permissions at once) ═══

chmod 755 script.sh         # rwxr-xr-x (standard for scripts/programs)
chmod 644 readme.md         # rw-r--r-- (standard for regular files)
chmod 700 ~/.ssh             # rwx------ (private — only owner)
chmod 600 ~/.ssh/id_ed25519  # rw------- (SSH keys MUST be 600)
chmod 400 secret.key         # r-------- (read-only, even for owner)

# ═══ Symbolic Mode (add/remove specific permissions) ═══

chmod +x script.sh          # add execute for everyone
chmod u+x script.sh         # add execute for owner only
chmod go-w config.yaml      # remove write from group and others
chmod a+r public.html       # add read for all (a = ugo)
chmod o= private.txt        # remove ALL permissions for others
chmod u=rwx,g=rx,o= app    # set specific permissions for each

# ═══ Recursive — Apply to Directory Trees ═══

# Make all files in a project readable
chmod -R 644 project/       # ⚠️ BAD — this removes execute from directories!

# CORRECT way: different permissions for files and directories
find project/ -type f -exec chmod 644 {} +    # files: rw-r--r--
find project/ -type d -exec chmod 755 {} +    # dirs:  rwxr-xr-x

# OR use the clever X permission (capital X):
chmod -R u=rwX,g=rX,o=rX project/
# X = execute ONLY if it's a directory or already has execute
# This correctly sets dirs to 755 and files to 644 in one command!
Key Takeaway
Files:       chmod 644 (rw-r--r--)  — readable by all, writable by owner
Scripts:     chmod 755 (rwxr-xr-x)  — executable by all, writable by owner
Private:     chmod 600 (rw-------)  — owner only (SSH keys, secrets)
Directories: chmod 755 (rwxr-xr-x)  — traversable by all, writable by owner
🧪 Why is chmod -R 644 project/ a mistake?
Because it removes execute permission from directories — without x on a directory, nobody can cd into it or access its contents, effectively locking you out of the entire tree
Because 644 is not a valid permission number
Because -R doesn't work with octal notation

ARCH-CMD03.3 — chown: Changing Ownership

chown (change owner) changes who owns a file and which group it belongs to. This matters because permission checks are always relative to the owner and group. If you download a file and it is owned by root, you cannot edit it even if you have permission on the directory — you need to change its ownership to yourself first. The command requires root privileges (sudo) because allowing any user to change ownership would break the entire permission model.

Syntax: chown [owner][:group] file. Variations: chown mari file (change owner only), chown mari:developers file (change both), chown :developers file (change group only — same as chgrp), chown mari: file (change owner and set group to owner's primary group). The -R flag applies recursively. The chown() syscall modifies the inode's i_uid and i_gid fields. On Linux, only root (CAP_CHOWN capability) can change the owner of a file. This is a security requirement — if users could give away ownership, they could escape disk quotas and attribution.

bash
# ═══ chown — Change Ownership ═══

# Take ownership of a file (requires sudo)
sudo chown mari file.txt

# Change owner AND group
sudo chown mari:developers project/

# Change group only (colon with no owner name)
sudo chown :docker /var/run/docker.sock

# Recursive — take ownership of entire directory tree
sudo chown -R mari:mari ~/projects/

# ═══ Common Real-World Scenarios ═══

# After extracting a tarball created by another user/system:
sudo chown -R mari:mari extracted_folder/

# Fix web server document root ownership:
sudo chown -R http:http /srv/http/

# Docker socket access (Part 3 of the Arch Guide):
sudo usermod -aG docker mari   # add user to docker group instead of chown

# Verify ownership
ls -la file.txt
# -rw-r--r-- 1 mari developers 1234 Mar 15 16:00 file.txt
#              ↑    ↑
#              owner group

ARCH-CMD03.4 — umask: Default Permissions for New Files

Every time you create a new file or directory, Linux applies default permissions. But where do these defaults come from? The answer is the umask — a "permission mask" that REMOVES certain permissions from new files. Think of it as a filter. The system starts with maximum permissions (666 for files, 777 for directories), then the umask blocks certain bits from getting through. A umask of 022 blocks write permission for group and others, resulting in 644 for files and 755 for directories. This is the standard umask on most Linux systems, including your Arch installation.

The math is: new file permissions = base permissions AND NOT umask. For files: 666 minus 022 = 644. For directories: 777 minus 022 = 755. (It is technically a bitwise AND with the complement, not subtraction, but the subtraction shortcut works for common umask values.)

The umask is a per-process attribute inherited from the parent. It is set with the umask builtin and applies to all subsequent file creation operations (open() with O_CREAT, mkdir(), mkfifo()). The kernel computes the actual permission as mode & ~umask. Base modes: open() typically requests 0666 (rw-rw-rw-), mkdir() requests 0777 (rwxrwxrwx). With umask 022: files get 0666 & ~0022 = 0644, directories get 0777 & ~0022 = 0755. Common umask values: 022 (standard — group/others read only), 027 (restrictive — others get nothing), 077 (private — only owner). Set permanently in ~/.bashrc, ~/.zshrc, or /etc/profile.

The umask is stored in the process's fs_struct (field umask), shared with threads via CLONE_FS. When do_sys_open() is called with O_CREAT, the VFS layer applies mode &= ~current->fs->umask before passing the mode to the filesystem's create inode operation. The umask() syscall atomically sets the new umask and returns the old one (which is why umask with no arguments displays the current value). Note that the umask only affects the initial creation mode — it does not affect subsequent chmod() calls, which set permissions directly. Programs that need specific permissions (like SSH writing key files at 0600) call open() with mode 0600 and then immediately fchmod() to ensure the correct mode regardless of umask.

bash
# ═══ Check Your Current umask ═══
umask                        # 0022 (typical default)
umask -S                     # u=rwx,g=rx,o=rx (symbolic format)

# ═══ See the Effect ═══
touch testfile.txt
mkdir testdir
ls -la testfile.txt testdir
# -rw-r--r-- testfile.txt    ← 666 minus 022 = 644
# drwxr-xr-x testdir         ← 777 minus 022 = 755

# ═══ Make It More Restrictive ═══
umask 077                    # only owner gets permissions
touch private.txt
ls -l private.txt
# -rw------- private.txt     ← 666 minus 077 = 600

# Reset to default
umask 022

# ═══ Common umask Values ═══
# 022 = standard (files: 644, dirs: 755)
# 027 = restrictive (files: 640, dirs: 750) — others get nothing
# 077 = private (files: 600, dirs: 700) — only owner
# 002 = collaborative (files: 664, dirs: 775) — group can write
🧪 With umask 027, what permissions will a new directory get?

ARCH-CMD03.5 — sudo, su, and User Management

sudo (superuser do) lets you run a single command with root privileges. Instead of logging in as root (dangerous — every typo has nuclear consequences), you stay as your normal user and temporarily elevate permissions when needed. This is the principle of least privilege — you have minimal permissions by default and only escalate when necessary. On your Arch system (from Part 1 of the guide), you configured sudo access for your user via the wheel group.

User management commands let you create accounts, manage groups, and control who has access to what. Groups are the primary mechanism for shared access — instead of making files world-readable, you put collaborators in a shared group and give group access.

The sudo binary is setuid root (-rwsr-xr-x) — when any user executes it, it runs with root's effective UID. Sudo reads /etc/sudoers (edited only via visudo which validates syntax before saving) to check if the calling user is authorized. The wheel group convention: the line %wheel ALL=(ALL:ALL) ALL grants all members of the wheel group full sudo access. User management: useradd -m -G wheel,docker username (create user with home dir and group memberships), usermod -aG group user (add user to group — -a is critical, without it you REPLACE all groups), passwd user (set password), userdel -r user (delete user and home dir), groups user (list groups), id user (show UID/GID/groups).

bash
# ═══ sudo — Temporary Root Access ═══

# Run a single command as root
sudo pacman -Syu              # system update (needs root)
sudo systemctl restart nginx  # restart a service
sudo vim /etc/pacman.conf     # edit a system config file

# Open a root shell (use sparingly!)
sudo -i                        # root login shell
sudo -s                        # root shell (keeps your env)

# Run as a different user (not root)
sudo -u postgres psql         # run psql as the postgres user

# Repeat last command with sudo (⚠️ know what you're running!)
sudo !!

# ═══ User & Group Management ═══

# See your identity
whoami                         # mari
id                             # uid=1000(mari) gid=1000(mari) groups=1000(mari),998(wheel),997(docker)
groups                         # mari wheel docker

# Add user to a group (ALWAYS use -a to APPEND!)
sudo usermod -aG wireshark mari    # add to wireshark group
# ⚠️ Without -a: usermod -G wireshark mari REMOVES you from ALL other groups!

# Create a new group
sudo groupadd developers

# Create a new user (if you needed another account)
sudo useradd -m -G wheel,docker,developers -s /bin/zsh luna
sudo passwd luna               # set password

# List all users on the system
cut -d: -f1 /etc/passwd | sort

# List all groups
cut -d: -f1 /etc/group | sort
⚠️ CRITICAL: The -a flag in usermod

When adding a user to a group, ALWAYS use usermod -aG group user (with -a for append). Without -a, the command REPLACES all existing group memberships with only the specified group. Running sudo usermod -G docker mari without -a would remove you from the wheel group — locking you out of sudo entirely. This is one of the most dangerous mistakes in Linux user management.

🧪 What happens if you run sudo usermod -G docker mari (without -a)?
It adds mari to the docker group
It REPLACES all of mari's supplementary groups with ONLY docker — removing her from wheel, wireshark, and every other group. She loses sudo access and can't fix it without root password or recovery boot.
It fails because -a is required

Exercise 2 (Medium) — Permission Auditor Script

Write a script that scans a directory and reports files with potentially insecure permissions: files writable by others (world-writable), scripts/executables owned by root that are writable by non-root users, and SSH keys that are not 600. Use find with permission-based tests.

solution
#!/bin/bash
set -euo pipefail

dir="${1:-$HOME}"
issues=0

echo "═══ Permission Audit: $dir ═══"

echo ""
echo "--- World-Writable Files ---"
while IFS= read -r f; do
    echo "  ⚠️ $f"
    ((issues++))
done < <(find "$dir" -type f -perm -002 2>/dev/null)

echo ""
echo "--- SSH Keys with Wrong Permissions ---"
if [[ -d "$HOME/.ssh" ]]; then
    while IFS= read -r f; do
        local perm=$(stat -c %a "$f")
        if [[ "$perm" != "600" && "$perm" != "400" ]]; then
            echo "  🔑 $f has permissions $perm (should be 600)"
            ((issues++))
        fi
    done < <(find "$HOME/.ssh" -type f -name "id_*" -not -name "*.pub")
fi

echo ""
echo "--- Setuid/Setgid Binaries (info only) ---"
find "$dir" -type f \( -perm -4000 -o -perm -2000 \) 2>/dev/null | head -10 | \
    while IFS= read -r f; do
        echo "  🔒 $(ls -la "$f" | awk '{print $1, $9}')"
    done

echo ""
if [[ $issues -eq 0 ]]; then
    echo "✅ No permission issues found!"
else
    echo "⚠️ Found $issues permission issue(s)"
fi

Exercise 3 (Hard) — Shared Project Setup Script

Write a script that creates a shared project directory for a team. It should create a group, add specified users to it, create the project directory with proper ownership and permissions (group-writable, setgid bit so new files inherit the group), create subdirectories (src, tests, docs, config), and set up a README with the project info. Validate all inputs and handle errors properly (BASH-F08 patterns).

solution
#!/bin/bash
set -euo pipefail

# ═══ Shared Project Setup ═══
PROJECT="${1:?Usage: $0   [user2] ...}"
shift
USERS=("$@")
[[ ${#USERS[@]} -eq 0 ]] && { echo "Need at least one user" >&2; exit 1; }

GROUP="proj-${PROJECT}"
DIR="/srv/projects/${PROJECT}"

echo "🌸 Setting up shared project: $PROJECT"

# Create group
sudo groupadd "$GROUP" 2>/dev/null || echo "  Group $GROUP already exists"

# Add users to group
for user in "${USERS[@]}"; do
    sudo usermod -aG "$GROUP" "$user"
    echo "  ✓ Added $user to $GROUP"
done

# Create directory structure
sudo mkdir -p "$DIR"/{src,tests,docs,config}

# Set ownership and permissions
sudo chown -R "root:${GROUP}" "$DIR"
sudo chmod -R 2775 "$DIR"    # 2 = setgid bit! New files inherit group

# Create README
sudo tee "$DIR/README.md" > /dev/null << EOF
# $PROJECT
Created: $(date +%Y-%m-%d)
Team: ${USERS[*]}
Group: $GROUP
EOF

echo "✅ Project ready at $DIR"
echo "   Group: $GROUP"
echo "   Permissions: 2775 (setgid — new files inherit group)"
echo "   Users: ${USERS[*]}"
echo "   ⚠️ Users must log out and back in for group change to take effect"
Output (./shared-project.sh webapp mari luna)

  ✓ Added mari to proj-webapp
  ✓ Added luna to proj-webapp
✅ Project ready at /srv/projects/webapp
   Group: proj-webapp
