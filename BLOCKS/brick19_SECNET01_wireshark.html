SEC-NET01.1 — Packet Capture Fundamentals

Every piece of data that travels over a network — every web page, every API call, every DNS lookup, every nmap probe — is broken into small pieces called packets. Each packet has a header (the envelope — who sent it, who it is for, what protocol it uses) and a payload (the contents — the actual data). A packet capture records every packet that passes through a network interface, allowing you to inspect what happened after the fact or watch traffic in real time.

Two tools dominate this space. tcpdump is the command-line packet capture tool — fast, scriptable, available on every Linux system, and perfect for quick captures and server-side analysis. Wireshark is the graphical packet analyzer — it provides a visual dissection of every protocol layer with color coding, search, filtering, and deep protocol analysis. The professional workflow is: capture with tcpdump (lightweight, can run on servers without a GUI), then analyze with Wireshark (rich visual analysis on your workstation).

Packet capture works through the kernel's packet socket mechanism. Both tcpdump and Wireshark use libpcap (the Packet Capture Library) which opens a raw socket on the specified interface and attaches a BPF (Berkeley Packet Filter) program to filter packets in kernel space (much faster than filtering in userspace). On your Arch system, capture requires either root privileges or membership in the wireshark group (which grants the CAP_NET_RAW and CAP_NET_ADMIN capabilities). Captured packets are stored in the pcap format (legacy) or pcapng format (modern — supports multiple interfaces, annotations, and name resolution records). The pcapng format is Wireshark's default and can be read by all modern tools. For capturing traffic to/from your Docker containers (which communicate over the docker0 bridge or custom Docker networks), capture on the lo (loopback) interface since the containers are accessed via localhost port mappings.

When you capture on an interface, libpcap calls socket(AF_PACKET, SOCK_RAW, htons(ETH_P_ALL)) which creates a socket that receives copies of ALL packets traversing the interface — both incoming and outgoing. The kernel delivers packet copies to this socket via a memory-mapped ring buffer (PACKET_MMAP) to avoid the overhead of per-packet syscalls. The BPF filter (compiled from tcpdump's filter expression via pcap_compile()) is attached with setsockopt(SO_ATTACH_FILTER) and runs as a virtual machine inside the kernel — packets that do not match the filter are dropped before ever being copied to userspace, which is critical for performance on high-traffic interfaces. On your Dell 7420's Intel Wi-Fi adapter, enabling monitor mode (iwconfig wlan0 mode monitor) would let you capture all wireless frames on the channel (including other devices' traffic), but this is typically only used for wireless security analysis and breaks your normal Wi-Fi connection.

SEC-NET01.2 — tcpdump: Command-Line Packet Capture

tcpdump is the Swiss Army knife of packet capture — it runs in the terminal, works on every Linux system (including remote servers via SSH where Wireshark cannot run), and can capture millions of packets to a file for later analysis. The basic pattern is tcpdump -i interface filter-expression. The filter expression tells tcpdump which packets to capture — you can filter by host, port, protocol, or complex combinations. Without a filter, tcpdump captures everything, which can be overwhelming on a busy network.

Essential tcpdump flags: -i IFACE (capture on this interface; any for all), -n (do not resolve hostnames — much faster), -nn (do not resolve hostnames OR port names), -v/-vv/-vvv (increasing verbosity), -c N (capture exactly N packets then stop), -w FILE (write raw packets to pcap file — for later Wireshark analysis), -r FILE (read from a pcap file instead of live capture), -A (print packet contents in ASCII — great for seeing HTTP traffic), -X (print contents in hex AND ASCII), -s 0 or -s0 (capture full packets, not just first 262144 bytes). BPF filter primitives: host IP, src IP, dst IP, port N, src port N, dst port N, tcp, udp, icmp, net CIDR. Combine with and, or, not, and parentheses: tcp port 80 and host 192.168.178.42.

bash — tcpdump essentials
# ═══ Basic Captures ═══

# Capture ALL traffic on loopback (for Docker lab)
sudo tcpdump -i lo -nn
# -nn = no name resolution (faster, shows raw IPs and ports)
# Ctrl+C to stop

# Capture only traffic to DVWA (port 8081)
sudo tcpdump -i lo -nn port 8081

# Capture HTTP traffic and show the content in ASCII
sudo tcpdump -i lo -nn -A port 8081
# -A shows the HTTP request/response bodies — you can SEE the SQL injection!

# Capture exactly 100 packets
sudo tcpdump -i lo -nn -c 100 port 8081

# ═══ Save to File (for Wireshark analysis) ═══

# Capture DVWA traffic to a pcap file
sudo tcpdump -i lo -nn -w ~/captures/dvwa_session.pcap port 8081
# Run this in one terminal, then perform your attacks in the browser
# Ctrl+C when done → open the pcap in Wireshark

# ═══ Filter Expressions (BPF) ═══

# Only TCP SYN packets (connection initiations — see nmap scans!)
sudo tcpdump -i lo -nn 'tcp[tcpflags] & (tcp-syn) != 0 and tcp[tcpflags] & (tcp-ack) == 0'

# DNS queries only
sudo tcpdump -i wlan0 -nn port 53

# Traffic to/from your FRITZ!Box gateway
sudo tcpdump -i wlan0 -nn host 192.168.178.1

# Combine: HTTP POST requests to DVWA (see form submissions)
sudo tcpdump -i lo -nn -A port 8081 | grep -A 10 "POST"

# ═══ Read a Saved Capture ═══

# Read and display a pcap file
tcpdump -nn -r ~/captures/dvwa_session.pcap

# Read with filter (only HTTP GET requests)
tcpdump -nn -A -r ~/captures/dvwa_session.pcap | grep -B 2 "GET /"

✅ The capture-then-analyze workflow:

Run tcpdump -i lo -w capture.pcap port 8081 in one terminal. Perform your web attacks in the browser. Stop tcpdump with Ctrl+C. Open capture.pcap in Wireshark for deep visual analysis. This gives you the best of both worlds — lightweight capture + powerful analysis.

❌ Common mistake — forgetting -w:

Without -w, tcpdump only shows a summary to the terminal and does NOT save the raw packet data. If you want to analyze packets in Wireshark later, you MUST use -w file.pcap. The terminal output is just a summary — the pcap file contains the complete packet data.

🛡️ You need to capture traffic between your browser and DVWA running on port 8081. Which tcpdump command captures this to a file for later Wireshark analysis?
sudo tcpdump -i lo -nn -w capture.pcap port 8081 — captures on loopback (since DVWA is on localhost), with no name resolution, saves raw packets to a pcap file filtered to port 8081
sudo tcpdump -i wlan0 port 8081 — this captures on the Wi-Fi interface, not loopback
tcpdump port 8081 — this does not save to a file and may fail without sudo

SEC-NET01.3 — Wireshark: Visual Packet Analysis

Wireshark is the world's most popular network protocol analyzer. Where tcpdump shows you a text summary of each packet, Wireshark displays the complete contents in a three-pane view: the packet list (top — all captured packets in sequence), the protocol detail tree (middle — every field of the selected packet broken down by protocol layer), and the raw hex/ASCII view (bottom — the actual bytes on the wire). Every protocol is color-coded — HTTP in green, TCP in grey, DNS in blue, errors in red — so you can spot patterns at a glance. The display filter bar at the top lets you narrow the view to exactly what you need: http.request.method == "POST" shows only HTTP POST requests, tcp.port == 8081 shows only DVWA traffic.

Wireshark's display filters (different from tcpdump's capture filters!) use a protocol-aware expression language. Key filter expressions: http (all HTTP traffic), http.request (HTTP requests only), http.response (responses only), http.request.method == "POST" (POST requests), http.request.uri contains "sqli" (URLs containing "sqli"), http.cookie (packets with cookies), tcp.port == 8081 (DVWA traffic), dns (DNS queries/responses), tcp.flags.syn == 1 and tcp.flags.ack == 0 (SYN-only — nmap scan probes), ip.addr == 192.168.178.42 (traffic to/from your machine). Right-click any field in the detail pane → "Apply as Filter" to instantly filter on that value. The "Follow TCP Stream" feature (right-click a packet → Follow → TCP Stream) reassembles the entire TCP conversation and shows it as a continuous dialog — perfect for reading complete HTTP request/response exchanges, including SQL injection payloads and responses.

wireshark — essential display filters
# ═══ Wireshark Display Filters (type these in the filter bar) ═══

# All HTTP traffic
http

# Only HTTP POST requests (form submissions, login attempts)
http.request.method == "POST"

# HTTP requests containing "sqli" in the URL (SQL injection pages)
http.request.uri contains "sqli"

# HTTP requests containing a single quote (SQL injection probes!)
http.request.uri contains "'"

# Only packets with cookies (session tracking)
http.cookie

# TCP SYN packets without ACK (nmap SYN scan probes)
tcp.flags.syn == 1 and tcp.flags.ack == 0

# DNS queries (name resolution)
dns.qr == 0

# Only error responses (HTTP 4xx/5xx)
http.response.code >= 400

# Traffic to DVWA only
tcp.port == 8081

# Combine filters with and/or/not
http.request.method == "POST" and tcp.port == 8081
http and not http.request.method == "GET"

# ═══ Essential Wireshark Actions (keyboard shortcuts) ═══

# Ctrl+E      → Start/stop capture
# Ctrl+F      → Find packet by content
# Ctrl+G      → Go to packet number
# Ctrl+Shift+F → Follow TCP/HTTP stream (selected packet)
# Right-click → Follow → TCP Stream (reassemble entire conversation)
# Right-click → Apply as Filter → Selected (filter on clicked value)
# Statistics → Conversations (see who talked to whom)
# Statistics → Protocol Hierarchy (what protocols are in the capture)

The most powerful Wireshark feature for security analysis is Follow TCP Stream. Right-click any packet in an HTTP conversation and select Follow → TCP Stream. Wireshark reassembles the entire exchange into a readable dialogue — the client's requests in red, the server's responses in blue. For HTTP traffic to DVWA, you will see the complete SQL injection payload in the GET parameter, the server's HTML response containing the extracted database data, the session cookies being exchanged, and every other detail of the attack — all in plain text because HTTP is unencrypted.

🛡️ You want to find all packets in a Wireshark capture where a SQL injection was attempted against DVWA. Which display filter would most effectively identify these?
http.request.uri contains "'" or http.request.uri contains "UNION" or http.request.uri contains "OR" — SQL injection payloads almost always contain single quotes, UNION keywords, or OR conditions in the URL parameters
tcp.port == 3306 — this is the MySQL port, but the injection goes through the web server on port 8081
http — this shows all HTTP traffic, not specifically injection attempts

SEC-NET01.4 — Capturing Your Own Attacks: See SQLi & XSS on the Wire

This is where everything connects. You will capture the traffic while performing the attacks from Brick 18, then analyze the captures to see exactly what the attack looks like at the packet level. This is the perspective of a defender with a network monitoring system — they see your attack traffic and must identify it. It is also the perspective of a forensic analyst investigating after a breach — they have a packet capture and must reconstruct what happened.

The exercise flow: Terminal 1 runs tcpdump to capture traffic. Your browser performs the attack against DVWA. Terminal 1 captures every packet of the attack. You analyze the capture either with tcpdump filters or by opening the pcap in Wireshark. For SQL injection, you will see the payload in the HTTP GET parameters (URL-encoded). For XSS, you will see the injected script tags in the request. For command injection, you will see the chained commands in the POST data. The session cookie (PHPSESSID) is visible in every request — demonstrating why cookie theft via XSS is so dangerous.

bash — capture while attacking
# ═══ EXERCISE: Capture SQL Injection Traffic ═══

# Terminal 1: Start capturing DVWA traffic
mkdir -p ~/captures
sudo tcpdump -i lo -nn -w ~/captures/sqli_attack.pcap port 8081

# Terminal 2 / Browser: Perform the SQL injection from Brick 18

# Terminal 1: Stop capture with Ctrl+C after the attack

# ═══ Analyze with tcpdump ═══

tcpdump -nn -A -r ~/captures/sqli_attack.pcap | grep -A 5 "GET /vulnerabilities/sqli"

# Find the exact SQL injection payload in the URL
tcpdump -nn -A -r ~/captures/sqli_attack.pcap | grep "UNION"

# Find session cookies being transmitted
tcpdump -nn -A -r ~/captures/sqli_attack.pcap | grep "PHPSESSID"

# ═══ Open in Wireshark for Visual Analysis ═══
wireshark ~/captures/sqli_attack.pcap &

✅ What this teaches you as a defender:

If you were monitoring a production web server's traffic, you could detect SQL injection attempts by looking for SQL keywords (UNION, SELECT, OR, AND, single quotes) in HTTP request parameters. A WAF (Web Application Firewall) does exactly this — it inspects every HTTP request for attack signatures and blocks them before they reach the application.

❌ Why HTTPS matters:

Everything you just saw — the SQL injection payload, the extracted credentials, the session cookie — was visible in plain text because DVWA uses HTTP (not HTTPS). If the traffic were encrypted with TLS, tcpdump and Wireshark would only see encrypted gibberish. This is why HTTPS is critical — it protects data in transit from network-level interception. An attacker on the same Wi-Fi cannot read HTTPS traffic.

SEC-NET01.5 — Protocol Analysis: DNS, HTTP, and TLS

Beyond capturing attacks, network analysis lets you understand how protocols work at the deepest level. You can watch a DNS query resolve a domain name to an IP address, see the TCP three-way handshake establish a connection, watch the TLS handshake negotiate encryption, and follow the HTTP request-response exchange that delivers a web page. This understanding is invaluable for both security (detecting anomalies) and debugging (figuring out why a connection fails).

Key protocols to analyze: DNS — filter dns. Look at the query name (what domain was looked up), query type (A, AAAA, MX), response code (NOERROR, NXDOMAIN = domain does not exist, SERVFAIL = server error), and response data (the resolved IP). DNS exfiltration is a real attack technique where stolen data is encoded in DNS query names (e.g., stolen-data-hex.attacker.com). HTTP — filter http. Analyze request methods (GET, POST), headers (Host, User-Agent, Cookie, Authorization), request bodies (POST data — form submissions, JSON), response codes (200, 301, 403, 404, 500), and response bodies. TLS — filter tls. You can see the Client Hello (supported cipher suites, TLS version, SNI hostname), Server Hello (chosen cipher suite and certificate), and the encrypted application data (unreadable without the session key). The SNI (Server Name Indication) in the Client Hello reveals which domain is being accessed even though the content is encrypted — this is how firewalls can block HTTPS sites without decrypting traffic.

SEC-NET01.6 — Detection & Forensics: Identifying Attacks in Captures

Now flip the perspective — you are the defender. Someone hands you a packet capture and says "we think we were hacked last night." How do you find the evidence? This is network forensics, and the techniques you have learned in this module give you the tools to answer: What happened? When? From where? What data was compromised? This is one of the most valuable skills in cybersecurity — the ability to read a packet capture like a detective reads a crime scene.

Attack signatures in packet captures: Port scanning — many SYN packets from one source to many ports, most receiving RST (closed) or no response (filtered). In Wireshark: tcp.flags.syn == 1 and tcp.flags.ack == 0 then check Statistics → Conversations for the source IP. SQL injection — HTTP requests containing SQL keywords in parameters: http.request.uri contains "UNION" or http.request.uri contains "'" or http.request.uri contains "SELECT". XSS attempts — HTTP requests containing <script, onerror=, javascript: in parameters. Brute force — many HTTP POST requests to a login page from one IP in rapid succession, most receiving the same response size (failed login) with occasional different size (successful login). Data exfiltration — unusually large response bodies, connections to unfamiliar IP addresses, DNS queries with long subdomain labels (DNS tunneling). Command and control (C2) — periodic outbound connections to the same IP at regular intervals (beaconing).

wireshark — attack detection filters
# ═══ Wireshark Filters for Detecting Attacks ═══

# Detect port scanning (many SYN-only packets from one source)
tcp.flags.syn == 1 and tcp.flags.ack == 0
# Then: Statistics → Conversations → TCP tab → Sort by packets
# A scanner will have hundreds of entries with 1-2 packets each

# Detect SQL injection attempts
http.request.uri contains "UNION" or http.request.uri contains "SELECT" or http.request.uri contains "'"

# Detect XSS attempts
http.request.uri contains "<script" or http.request.uri contains "onerror"

# Detect brute-force login (many POSTs to login page)
http.request.method == "POST" and http.request.uri contains "login"
# Check if many come from the same IP in a short time

# Detect credential theft (passwords in plain HTTP)
http.request.method == "POST" and http contains "password"

# Detect unusual DNS (possible exfiltration or C2)
dns.qr == 0 and dns.qry.name.len > 50
# Legitimate DNS queries are rarely longer than 50 characters
# DNS exfiltration encodes data in very long subdomain labels

# Detect connections to unusual ports
tcp.dstport > 10000 and tcp.flags.syn == 1
# Outbound connections to high ports may indicate backdoors or C2

# ═══ Forensic Analysis Workflow ═══
# 1. Statistics → Protocol Hierarchy (what protocols are present?)
# 2. Statistics → Conversations (who talked to whom?)
# 3. Statistics → Endpoints (which IPs are most active?)
# 4. Apply attack filters above to find specific evidence
# 5. Follow TCP Stream on suspicious packets to read the full exchange
# 6. File → Export Objects → HTTP (extract files transferred over HTTP)

SEC-NET01.7 — Practice: Capture, Attack, Analyze

Exercise 1 (Easy) — Capture and Analyze a Login

Start a tcpdump capture on port 8081, log into DVWA in your browser with admin/password, stop the capture. Open the pcap in Wireshark. Find the POST request containing the login credentials. Follow the TCP stream to see the complete login exchange. Document: the exact POST data, the session cookie that was set, and the HTTP response code.

solution
#!/bin/bash
set -euo pipefail

mkdir -p ~/captures

echo "Starting capture on port 8081..."
echo "→ Log into DVWA at http://localhost:8081 in your browser"
echo "→ Press Ctrl+C here when done"
echo ""
sudo tcpdump -i lo -nn -w ~/captures/login.pcap port 8081

echo ""
echo "═══ Quick Analysis ═══"

echo "--- POST requests (login form submission) ---"
tcpdump -nn -A -r ~/captures/login.pcap | grep -A 3 "POST /login"

echo ""
echo "--- Credentials in plain text ---"
tcpdump -nn -A -r ~/captures/login.pcap | grep "username="

echo ""
echo "--- Session cookies ---"
tcpdump -nn -A -r ~/captures/login.pcap | grep "PHPSESSID" | head -3

echo ""
echo "Open in Wireshark: wireshark ~/captures/login.pcap"

Exercise 2 (Medium) — Capture and Identify an nmap Scan

Start a capture on the loopback interface with no filter. In another terminal, run an nmap SYN scan against localhost on ports 1-1000. Stop the capture. Open it in Wireshark and answer: How many SYN packets did nmap send? How many ports responded with SYN-ACK (open)? How many with RST (closed)? How many had no response (filtered)? Exercise 3 (Hard) — Full Attack Capture and Forensic Report

Start a capture. Perform a complete DVWA attack session: login, SQL injection (extract all users), XSS (stored script injection), and command injection (read /etc/passwd). Stop the capture. Analyze the pcap in Wireshark and write a forensic report documenting every attack with timestamps, source/destination, exact payloads (extracted from the packets), and the data that was compromised. Include Wireshark screenshots or filter expressions that isolate each attack. This is a real-world forensic analysis exercise.

- Credentials: username=admin, password=password
- Session cookie: PHPSESSID=[value]
- ⚠ Transmitted in plain text (no HTTPS)

### 17:42:15 — SQL Injection Attack
- Filter: http.request.uri contains "UNION"
- Packet #: [N]
- Payload: 1' UNION SELECT user, password FROM users -- -
- Data extracted: 5 user credentials with password hashes
- OWASP: A03 Injection

### 17:42:30 — Stored XSS Injection
- Filter: http contains "<script"
- Packet #: [N]
- Payload: <script>alert(document.cookie)</script>
- Impact: Persistent — all subsequent visitors affected
- OWASP: A07 XSS

### 17:42:45 — Command Injection
- Filter: http.request.method == "POST" and http contains "cat"
- Packet #: [N]
- Payload: 127.0.0.1; cat /etc/passwd
- Data extracted: Full /etc/passwd file (all system users)
- OWASP: A03 Injection

## Indicators of Compromise (IOCs)
- Source IP: 127.0.0.1 (attacker)
- Session cookie used: PHPSESSID=[value]
- Attack signatures: UNION SELECT, <script>, ; cat /etc/passwd

## Recommendations
- Enable HTTPS (prevents credential sniffing)
- Implement parameterized queries (prevents SQLi)
- Add output encoding + CSP (prevents XSS)
- Remove system command calls (prevents cmd injection)
- Deploy a WAF (detects attack signatures in transit)
