ARCH-CMD06.1 — Installing & Updating: pacman -S

The -S flag stands for "Sync" — it synchronizes packages from remote repositories to your local machine. This is what you use to install new software, update your entire system, search for packages, and get information about packages before installing them. The single most important command on Arch Linux is sudo pacman -Syu — it refreshes the package database from the mirrors and then upgrades every installed package to its latest version. On Arch, you should run this at least once a week (ideally before installing anything new) because Arch is a rolling release where packages are always being updated.

The pacman flag convention is elegant: capital letters are the operation (what you want to do) and lowercase letters are options (how you want to do it). So -S means "sync operation," -Sy means "sync and refresh the database," -Syu means "sync, refresh, and upgrade everything," and -Ss means "sync search" (search for a package in the repos).

Pacman's sync operations communicate with configured repositories listed in /etc/pacman.conf (on your system: core, extra, multilib, and optionally blackarch). The database files are downloaded from mirrors listed in /etc/pacman.d/mirrorlist and stored locally at /var/lib/pacman/sync/. The -y flag downloads fresh copies of these database files. The -u flag compares installed package versions against the database and upgrades any where the remote version is newer. Package files are downloaded to /var/cache/pacman/pkg/ as .pkg.tar.zst archives (zstd-compressed tarballs containing the package files, metadata, install scripts, and checksums). Pacman uses libalpm (Arch Linux Package Management library) for all operations, performing dependency resolution, conflict detection, and file ownership tracking before extracting packages.

When you run pacman -Syu, libalpm performs the following sequence: (1) Download sync databases from mirrors using HTTP(S) — each database is a tar archive of package metadata (desc files containing version, dependencies, file size, checksum). (2) Compare each installed package's version string against the sync database using alpm_pkg_vercmp() which implements the Arch versioning scheme (epoch:pkgver-pkgrel, compared component by component). (3) Build a transaction: calculate the set of packages to upgrade, resolve all dependencies transitively (using a topological sort of the dependency graph), detect conflicts (two packages providing the same file), and present the summary to the user. (4) Download package files from mirrors, verifying GnuPG signatures against the Arch Linux keyring (/etc/pacman.d/gnupg/). (5) Remove old package files from the filesystem. (6) Extract new package files to their correct locations. (7) Run post-install scriptlets (.INSTALL files containing shell functions for post_install, post_upgrade, post_remove). (8) Update the local database at /var/lib/pacman/local/. The entire operation is transactional — if interrupted, pacman can detect the inconsistency and recover on the next run.

bash
# ═══ THE Command: Full System Update ═══
sudo pacman -Syu
# S = sync operation
# y = refresh package database from mirrors
# u = upgrade all packages to latest versions
# Run this AT LEAST once a week. Always before installing new packages.

# ═══ Install Packages ═══
sudo pacman -S neovim                # install a single package
sudo pacman -S nmap wireshark-qt sqlmap  # install multiple at once
sudo pacman -S --needed base-devel git   # skip already-installed packages

# ═══ Search for Packages ═══
pacman -Ss docker                    # search repos by name/description
pacman -Ss "^python-"                # regex search: all python-* packages

# ═══ Get Package Info (before installing) ═══
pacman -Si docker                    # detailed info: version, size, deps, URL
pacman -Si docker | grep "Depends"   # see what it pulls in

# ═══ Install a Specific Version (from cache) ═══
sudo pacman -U /var/cache/pacman/pkg/package-1.2.3-1-x86_64.pkg.tar.zst

# ═══ Download Only (don't install yet) ═══
sudo pacman -Sw firefox               # download to cache without installing
Expected Output (pacman -Ss docker)
extra/docker 1:26.1.4-1
    Pack, ship and run any application as a lightweight container
extra/docker-buildx 0.14.1-1
    Docker CLI plugin for extended build capabilities
extra/docker-compose 2.27.1-1
    Fast, isolated development environments using Docker
⚠️ CRITICAL: Never run -Sy without -u

Running pacman -Sy (refresh database) without -u (upgrade) leaves your system in a dangerous state: the package database knows about newer versions, but your installed packages are still old. If you then install a new package with pacman -S package, it may pull in new library versions that are incompatible with your old installed packages, causing "dependency hell" and potentially breaking your system. The safe rule: always -Syu (refresh + upgrade together), never -Sy alone.

🧪 Why is pacman -Sy package (without -u) dangerous?
It refreshes the database to know about newer versions but does not upgrade existing packages — the new package may depend on newer library versions that conflict with your older installed libraries, causing partial upgrade breakage
Because -y deletes the local database
Because pacman cannot install and refresh at the same time

ARCH-CMD06.2 — Removing Packages: pacman -R

The -R operation removes installed packages. The plain -R removes just the named package, but the real power is in the options. The option -Rs removes the package AND any of its dependencies that are no longer needed by any other package — this is what you should use 95% of the time, because it keeps your system clean. Without -s, removing a package leaves behind all its dependency packages, which gradually bloat your system with orphaned packages that nothing uses.

The option -Rns goes further — it also removes any configuration files that the package left behind (the n stands for "nosave"). This gives you a completely clean removal, as if the package was never installed. Use -Rns when you are sure you will not want the package back.

Pacman's remove operation traces the dependency graph in reverse. The -s (recursive) flag causes pacman to check each dependency of the removed package: if the dependency is not required by any other installed package and was not explicitly installed (it was pulled in automatically as a dependency), it is also scheduled for removal. The -n (nosave) flag prevents pacman from saving configuration files with a .pacsave extension — by default, modified config files are renamed to file.pacsave during removal so you can recover them. The -c (cascade) flag is the nuclear option: it removes the package and ALL packages that depend on it. Use this with extreme caution — removing a core library with -Rc can uninstall hundreds of packages. Orphan packages (installed as dependencies but no longer required) can be listed with pacman -Qdt and bulk-removed.

bash
# ═══ Remove Packages ═══

# Remove package + unused dependencies (standard — use this)
sudo pacman -Rs firefox

# Clean remove: package + unused deps + config files
sudo pacman -Rns firefox

# Remove ONLY the package (leave dependencies — rarely what you want)
sudo pacman -R firefox

# ═══ Orphan Management (cleanup unused dependency packages) ═══

# List orphans: packages installed as deps but no longer needed
pacman -Qdt

# Remove ALL orphans in one command
sudo pacman -Rns $(pacman -Qdtq)
# -Qdt lists orphans; q = quiet (names only); -Rns removes them cleanly
# If there are no orphans, this command safely does nothing (empty argument)

# ═══ Check Before Removing (what depends on this package?) ═══

# See what would break if you remove a package
pacman -Qi python | grep "Required By"
# If "Required By: None" → safe to remove
# If it lists other packages → those will break if you remove this
Expected Output (pacman -Qdt)
lib32-glibc
lib32-gcc-libs
libmfx
perl-xml-parser
✅ The removal hierarchy:

Use -Rs for normal removal (package + unused deps). Use -Rns for clean removal (+ config files). Run pacman -Rns $(pacman -Qdtq) periodically to clean up orphans. Always check pacman -Qi pkg | grep "Required By" before removing system packages.

❌ Never use -Rc carelessly:

The -Rc (cascade) flag removes the package AND everything that depends on it. Running pacman -Rc glibc would try to remove nearly every package on your system because almost everything depends on glibc. Always check what -Rc would remove before confirming.

ARCH-CMD06.3 — Querying Installed Packages: pacman -Q

The -Q operation queries the LOCAL database — it tells you about packages that are already installed on your machine. This is how you answer questions like "Is Docker installed?", "What version of Python do I have?", "What files did this package install?", "How many packages do I have?", and "Which packages did I install explicitly versus which came in as dependencies?" This information is invaluable for system maintenance, debugging, and migration (recreating your setup on a new machine).

The local database lives at /var/lib/pacman/local/ with one directory per installed package containing desc (metadata), files (installed file list), and optionally mtree (file checksums for integrity verification). Key query flags: -Qi pkg (detailed info: version, install date, size, dependencies, "Required By"), -Ql pkg (list all files installed by the package), -Qo /path/to/file (reverse: which package owns this file?), -Qe (explicitly installed packages — what you chose to install), -Qd (dependency packages — pulled in automatically), -Qdt (orphans — dependencies no longer needed), -Qm (foreign packages — not in any configured repo, usually AUR packages), -Qn (native packages — from official repos). The -q suffix makes any query output names-only (no versions), useful for piping into other commands.

bash
# ═══ Basic Queries ═══

# Is a package installed? (exit code 0 = yes, 1 = no)
pacman -Q docker                      # docker 1:26.1.4-1
pacman -Q docker &>/dev/null && echo "installed" || echo "not installed"

# Detailed info about an installed package
pacman -Qi docker

# What files did a package install?
pacman -Ql docker                      # list every file
pacman -Ql docker | grep "bin/"       # just the executables

# Which package owns a file? (reverse lookup)
pacman -Qo /usr/bin/git               # /usr/bin/git is owned by git 2.45.2-1
pacman -Qo $(which python)             # which package provides the python binary?

# ═══ Listing & Counting ═══

# How many packages are installed?
pacman -Q | wc -l                     # total count

# Packages YOU explicitly installed (your choices, not auto-deps)
pacman -Qe                              # explicitly installed
pacman -Qeq                             # names only (great for backup!)

# AUR packages (not from official repos)
pacman -Qm                              # foreign/AUR packages

# Recently installed packages (sorted by install date)
pacman -Qi | awk '/^Name/{n=$3} /^Install Date/{print $0, n}' | sort -t: -k2 | tail -10

# ═══ Package Backup & Restore ═══

# Export your package list (for recreating on a new machine)
pacman -Qeq > ~/.config/pkglist-explicit.txt     # explicit packages
pacman -Qmq > ~/.config/pkglist-aur.txt           # AUR packages

# Restore on a new machine:
# sudo pacman -S --needed - < pkglist-explicit.txt
# paru -S --needed - < pkglist-aur.txt

ARCH-CMD06.4 — paru & the AUR: Community Packages

The Arch User Repository (AUR) is a massive community-driven collection of package build scripts for software that is not in the official Arch repositories. This includes things like Spotify, Visual Studio Code, Google Chrome, proprietary drivers, niche developer tools, and thousands of other packages. The AUR does not contain prebuilt packages — it contains PKGBUILD scripts that describe how to download, compile, and package software from source. An AUR helper like paru (which you installed in Part 1) automates this process: it searches the AUR, downloads the PKGBUILD, shows you the build script for review, compiles the package, and installs it with pacman. Paru is a drop-in replacement for pacman — every pacman command works with paru, plus it adds AUR support.

The AUR (https://aur.archlinux.org) is a Git-based repository where each package is a Git repo containing a PKGBUILD file (a bash script defining pkgname, pkgver, source URLs, sha256sums, depends, and a build() function) and optionally a .SRCINFO metadata file. When you run paru -S packagename and the package is in the AUR (not official repos), paru clones the Git repo, displays the PKGBUILD for your review (you SHOULD read it — it is arbitrary code that runs on your system), downloads the source files listed in the PKGBUILD, verifies checksums, runs makepkg to compile and package the software in a sandboxed build environment, and then calls pacman -U to install the resulting .pkg.tar.zst file. Paru also handles AUR dependency chains, where AUR packages depend on other AUR packages. Important: AUR packages are user-contributed and not officially vetted — always review the PKGBUILD before building.

bash
# ═══ paru — Drop-in pacman Replacement + AUR ═══

# System update (official repos + AUR packages)
paru -Syu                              # updates EVERYTHING including AUR packages

# Search (searches both official repos AND AUR)
paru -Ss visual-studio-code

# Install an AUR package
paru -S visual-studio-code-bin        # paru detects it's AUR, shows PKGBUILD, builds it

# Install from official repos (works exactly like pacman)
paru -S neovim                         # same as sudo pacman -S neovim

# Interactive search (paru's killer feature)
paru spotify                            # search + numbered list → type number to install

# ═══ Review & Security ═══

# Always review PKGBUILD before building (paru shows it automatically)
# Look for suspicious commands: curl | bash, rm -rf, wget to unknown URLs
# Legitimate PKGBUILDs download from official project sources

# Show the PKGBUILD without installing
paru -Gp spotify                       # print PKGBUILD to stdout

# List installed AUR packages
pacman -Qm                              # foreign packages = AUR packages

# ═══ AUR Package Tips ═══

# Prefer "-bin" packages (precompiled — faster install, no build deps)
# visual-studio-code-bin ← downloads prebuilt binary (fast)
# visual-studio-code     ← compiles from source (slow, needs build tools)

# "-git" packages track the latest development version
# neovim-git ← latest Git commit (bleeding edge, may break)
# neovim     ← official release (stable)
Expected Output (paru spotify)
1 aur/spotify 1:1.2.31-2 (+3845 14.23) (Installed)
    An proprietary music streaming service
2 aur/spotify-adblock 1.0.3-1 (+156 2.10)
    Spotify ad blocker for Linux
3 extra/playerctl 2.4.1-3 (3.2 MiB 10.1 MiB)
    mpris media player controller

:: Packages to install (eg: 1 2 3, 1-3):
✅ The paru workflow:

Use paru -Syu for full system updates (replaces sudo pacman -Syu and also updates AUR packages). Use paru packagename for interactive search+install. Always read the PKGBUILD when prompted — it is code that runs on your machine. Prefer -bin packages for faster installs when available.

❌ AUR security awareness:

AUR packages are NOT officially reviewed or vetted. Anyone can upload a PKGBUILD. While malicious packages are rare and usually caught quickly by the community, you should always review the PKGBUILD before confirming the build. Look for suspicious commands, unknown download URLs, and post-install scripts that modify unexpected system files. The Arch Wiki recommends treating AUR packages with the same caution as any code you download from the internet.

ARCH-CMD06.5 — Cache & Mirror Management

Every package you install or update is cached at /var/cache/pacman/pkg/. Over months, this cache grows to many gigabytes because it keeps every version of every package you have ever installed. Cleaning the cache periodically reclaims disk space. The tool paccache (from the pacman-contrib package) handles this intelligently — by default it keeps the 3 most recent versions of each package (so you can downgrade if needed) and removes everything older.

Mirrors are the servers that host Arch's package repository. Your /etc/pacman.d/mirrorlist controls which mirrors pacman uses and in what order. Choosing geographically close, fast mirrors significantly speeds up downloads. The reflector tool automatically tests mirror speeds and updates your mirrorlist with the fastest ones.

Cache management: paccache -r removes all but the 3 most recent versions of each cached package. paccache -rk 1 keeps only 1 version. pacman -Sc removes cached packages that are no longer installed (keeps only packages still on the system). pacman -Scc removes ALL cached packages (nuclear option — no downgrading possible). The recommended approach is a weekly systemd timer running paccache -r. Mirror management: reflector --country Germany --protocol https --sort rate --save /etc/pacman.d/mirrorlist finds the fastest HTTPS mirrors in Germany, tests their download speed, and writes the sorted list. You can also enable reflector.timer to automatically update mirrors weekly.

bash
# ═══ Cache Management ═══

# Check cache size
du -sh /var/cache/pacman/pkg/          # might be 5-20 GB!

# Clean: keep 3 most recent versions of each package (recommended)
sudo paccache -r
# ==> finished: 147 packages removed (1.2 GiB freed)

# Aggressive: keep only 1 version
sudo paccache -rk 1

# See what would be removed (dry run)
paccache -d

# Set up automatic weekly cache cleaning
sudo systemctl enable --now paccache.timer

# ═══ Mirror Management ═══

# Find fastest mirrors for Germany (where your FRITZ!Box connects)
sudo reflector \
  --country Germany \
  --protocol https \
  --sort rate \
  --latest 10 \
  --save /etc/pacman.d/mirrorlist

# Automatic weekly mirror updates
sudo systemctl enable --now reflector.timer

# View current mirrorlist
head -5 /etc/pacman.d/mirrorlist

ARCH-CMD06.6 — Troubleshooting & Recovery

Things can go wrong with package management: a system update might break something (a library incompatibility, a config file conflict, a service that will not start after upgrading). Knowing how to handle .pacnew files, downgrade a problematic package, and fix a broken pacman database transforms these situations from "my system is broken, time to reinstall" into "I will fix this in 5 minutes." The key principle is: Arch Linux ALWAYS gives you the tools to fix problems from the command line. You never need to reinstall — you just need to know where to look and what commands to run.

Common issues and fixes: (1) .pacnew files — when a package update includes a new version of a config file that you have modified, pacman saves the new version as file.pacnew and keeps your modified version. You must manually merge changes. Find them with find /etc -name "*.pacnew" or pacdiff (from pacman-contrib). (2) Downgrading — install an older version from the cache with pacman -U /var/cache/pacman/pkg/package-oldversion.pkg.tar.zst. Add the package to IgnorePkg in /etc/pacman.conf to prevent it from being upgraded again until you are ready. (3) Broken keyring — if package signatures fail, refresh with sudo pacman-key --init && sudo pacman-key --populate archlinux. (4) Corrupted database — sudo rm /var/lib/pacman/db.lck removes a stale lock file (only if pacman is not actually running). (5) File conflicts — if two packages claim the same file, use --overwrite as a last resort or check if one package should replace the other.

bash
# ═══ .pacnew Files (config merge after update) ═══

# Find all .pacnew files (new configs from updates)
sudo find /etc -name "*.pacnew" 2>/dev/null

# Interactive merge tool (from pacman-contrib)
sudo pacdiff
# Shows diff between your config and the new version, lets you merge

# ═══ Downgrading a Package ═══

# List cached versions of a package
ls /var/cache/pacman/pkg/python-*

# Install an older version from cache
sudo pacman -U /var/cache/pacman/pkg/python-3.12.0-1-x86_64.pkg.tar.zst

# Prevent it from upgrading (add to pacman.conf)
# Edit /etc/pacman.conf and add under [options]:
# IgnorePkg = python

# ═══ Common Fixes ═══

# Fix: "unable to lock database" (stale lock file)
# ONLY run this if pacman is NOT currently running:
sudo rm /var/lib/pacman/db.lck

# Fix: "invalid or corrupted package (PGP signature)"
sudo pacman-key --init
sudo pacman-key --populate archlinux
sudo pacman -Syu                       # retry the update

# Fix: "file exists in filesystem" (file conflict)
# First check who owns it:
pacman -Qo /path/to/conflicting/file
# If it's from a package that should be replaced:
sudo pacman -Syu --overwrite '/path/to/file'

# ═══ Verify Package Integrity ═══

# Check if any installed files have been modified or are missing
pacman -Qk                              # quick check (file existence)
pacman -Qkk                             # thorough check (checksums)
pacman -Qk docker                      # check specific package

# ═══ View the pacman log ═══
cat /var/log/pacman.log | tail -50   # recent activity
grep "upgraded" /var/log/pacman.log | tail -20   # recent upgrades
🧪 After a system update, your web server's config at /etc/nginx/nginx.conf still works but you notice a file called /etc/nginx/nginx.conf.pacnew. What should you do?
Delete the .pacnew file — your current config works fine
Compare the two files (using diff or pacdiff) and merge any important changes from the .pacnew into your config — the .pacnew may contain new directives, security fixes, or changes required by the updated version
Replace your config with the .pacnew file

Exercise 2 (Medium) — System Maintenance Script

Write a weekly maintenance script that performs a full system update (paru -Syu), cleans the package cache (keep 2 versions), removes orphan packages, checks for .pacnew files, reports how many packages were upgraded, and shows any failed services. Use the production script template from BASH-F08.

solution
#!/bin/bash
set -euo pipefail
G='\033[32m' Y='\033[33m' R='\033[31m' N='\033[0m'
log() { echo -e "[$(date +%H:%M:%S)] $*"; }

log "═══ 🌸 Weekly System Maintenance ═══"

# 1. Full system update
log "${G}📦 Updating system...${N}"
before=$(pacman -Q | md5sum)
paru -Syu --noconfirm || log "${Y}⚠️ Some packages may have failed${N}"
after=$(pacman -Q | md5sum)
[[ "$before" != "$after" ]] && log "   Packages were updated" || log "   Already up to date"

# 2. Clean package cache
log "${G}🧹 Cleaning cache (keeping 2 versions)...${N}"
sudo paccache -rk 2

# 3. Remove orphans
orphans=$(pacman -Qdtq 2>/dev/null) || true
if [[ -n "$orphans" ]]; then
    log "${Y}🗑️ Removing orphans:${N}"
    echo "$orphans"
    sudo pacman -Rns --noconfirm $orphans
else
    log "   No orphans found"
fi

# 4. Check for .pacnew files
pacnews=$(sudo find /etc -name "*.pacnew" 2>/dev/null)
if [[ -n "$pacnews" ]]; then
    log "${Y}⚠️ .pacnew files found (merge these!):${N}"
    echo "$pacnews"
else
    log "   No .pacnew files"
fi

# 5. Check for failed services
failed=$(systemctl --failed --no-legend 2>/dev/null)
if [[ -n "$failed" ]]; then
    log "${R}🚨 Failed services:${N}"
    echo "$failed"
else
    log "${G}✅ All services healthy${N}"
fi

# Summary
log "${G}═══ Maintenance complete ═══${N}"
printf "  Packages: %s  Cache: %s  Kernel: %s\n" \
  "$(pacman -Q | wc -l)" \
  "$(du -sh /var/cache/pacman/pkg/ | awk '{print $1}')" \
  "$(uname -r)"

Exercise 3 (Hard) — Package Diff Report

Write a script that compares your current package list against a saved baseline (from the backup exercise in Section 3). It should report packages that were added since the baseline, packages that were removed, packages that changed versions, and the total size change. Save the current state as the new baseline after reporting.

solution
#!/bin/bash
set -euo pipefail

BASELINE="$HOME/.config/pkglist-baseline.txt"
CURRENT=$(mktemp)
trap 'rm -f "$CURRENT"' EXIT

pacman -Q > "$CURRENT"

if [[ ! -f "$BASELINE" ]]; then
    echo "No baseline found. Saving current state as baseline."
    cp "$CURRENT" "$BASELINE"
    exit 0
fi

echo "═══ Package Diff Report ═══"
echo "Baseline: $(wc -l < "$BASELINE") packages"
echo "Current:  $(wc -l < "$CURRENT") packages"
echo ""

echo "--- Added ---"
comm -13 <(awk '{print $1}' "$BASELINE" | sort) <(awk '{print $1}' "$CURRENT" | sort) | sed 's/^/  + /'

echo "--- Removed ---"
comm -23 <(awk '{print $1}' "$BASELINE" | sort) <(awk '{print $1}' "$CURRENT" | sort) | sed 's/^/  - /'

echo "--- Version Changes ---"
comm -12 <(awk '{print $1}' "$BASELINE" | sort) <(awk '{print $1}' "$CURRENT" | sort) | \
  while read -r pkg; do
    old=$(grep "^$pkg " "$BASELINE" | awk '{print $2}')
    new=$(grep "^$pkg " "$CURRENT" | awk '{print $2}')
    [[ "$old" != "$new" ]] && printf "  ~ %-30s %s → %s\n" "$pkg" "$old" "$new"
  done | head -20
