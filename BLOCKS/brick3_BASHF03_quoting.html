BASH-F03.1 — The Quoting Rules (Big Picture)

There are three types of quoting in bash, and each one controls how much bash "processes" the text inside it. Think of it like three levels of mail handling. No quotes means the post office opens your letter, reads it, rearranges the words, and delivers what it thinks you meant (dangerous!). Double quotes means the post office peeks inside to fill in any address placeholders ($variables) but otherwise delivers it intact. Single quotes means the post office delivers the sealed envelope without opening it at all — completely literal, no changes.

The golden rule for beginners: when in doubt, use double quotes. Specifically, always double-quote your variables: "$var" instead of $var. This single habit prevents the most common category of bash bugs (word splitting and glob expansion on variables containing spaces or special characters).

Bash processes text through several expansion stages in this order: brace expansion → tilde expansion → parameter/variable expansion → command substitution → arithmetic expansion → word splitting → pathname (glob) expansion. Quoting controls which of these stages are applied. Single quotes suppress ALL expansions. Double quotes suppress word splitting and pathname expansion but allow parameter expansion, command substitution, and arithmetic expansion. No quotes allow everything. Understanding this pipeline is essential for predicting bash behavior.

✅ The habit that saves you:

Always write "$variable" with double quotes. Always. Even when you think it is not necessary. The one time it IS necessary will be the one time you forgot, and your script will delete the wrong files or break on a filename with spaces.

❌ The classic disaster:

BASH-F03.2 — Single Quotes (Absolute Literal)

Single quotes are a force field. Absolutely nothing inside them gets interpreted or changed by bash. The dollar sign is just a dollar sign. The asterisk is just a star. The exclamation mark is just punctuation. What you type is exactly what you get. This makes single quotes perfect for regex patterns, awk programs, and any text where you want bash to keep its hands off completely.

The one catch: you cannot put a single quote inside single quotes. If you need a literal single quote inside single-quoted text, you have to end the single quotes, add an escaped single quote, then restart single quotes: 'can'\''t' produces the text can't.

Within single quotes, every character is treated literally. No expansion of any kind occurs: no variable expansion ($var stays as the literal string "$var"), no command substitution, no escape sequences (\n is a literal backslash followed by 'n', not a newline), no glob expansion, no brace expansion. The only character that cannot appear inside single quotes is the single quote itself — it always terminates the quoted string. The workaround 'text'\''more' works because bash concatenates adjacent strings: 'text' + \' (escaped single quote) + 'more' = text'more.

In bash's lexer (file parse.y), when the scanner encounters a single quote character, it enters the SQSTRING state. In this state, every character is accumulated into the current token's buffer verbatim until another single quote is encountered, which terminates the state. The token is then flagged with W_QUOTED and W_NOGLOB attributes, which tell the expansion engine to skip all expansion stages for this token. No further processing is applied — the raw bytes are passed directly to the command as an argument.

bash
#!/bin/bash
name="Mari"

# Single quotes: NOTHING is expanded
echo 'Hello $name'              # Hello $name  (literal dollar sign)
echo 'Today is $(date)'          # Today is $(date)  (literal text)
echo 'Files: *.txt'              # Files: *.txt  (literal asterisk)
echo 'Path: ~/'                  # Path: ~/  (literal tilde)
echo 'Tab:\there'                # Tab:\there  (literal backslash-t)

# Perfect for: awk, grep patterns, regex
awk '{print $1}' file.txt      # $1 is awk's column 1, NOT a bash variable
grep '^[0-9]+$' data.txt      # regex stays intact

# Embedding a single quote (the '\'' trick)
echo 'I can'\''t stop learning'  # I can't stop learning
Expected Output
Hello $name
Today is $(date)
Files: *.txt
Path: ~/
Tab:\there
I can't stop learning
🧪 Why are single quotes used for awk scripts like awk '{print $1}'?
Because awk uses $1 to mean "first column" — without single quotes, bash would try to expand $1 as a bash variable before awk ever sees it
Because awk requires single quotes in its syntax
For readability only — double quotes would work the same way

BASH-F03.3 — Double Quotes (Selective Expansion)

Double quotes are the sweet spot — they let bash fill in your variables and run command substitutions, but they protect you from the two most dangerous automatic behaviors: word splitting (spaces breaking your arguments apart) and glob expansion (asterisks matching random filenames). This is why double quotes are the default choice for almost all situations where you reference variables.

Inside double quotes, three special characters still work: $ (variable/command expansion), ` (old-style command substitution — prefer $() instead), and \ (escape the next character). Everything else is literal.

Double quotes create a "weak quoting" context. The expansion pipeline still runs parameter expansion ($var, ${var}), command substitution ($(cmd)), and arithmetic expansion ($((expr))). However, the post-expansion stages of word splitting (IFS-based tokenization) and pathname expansion (glob matching) are suppressed. This means "$var" always produces exactly one argument to the command, regardless of what $var contains — spaces, tabs, newlines, asterisks, question marks — all are treated as literal characters within that single argument.

When the bash lexer encounters a double quote, it enters the DQSTRING state. In this state, it still recognizes $, `, \, and " as special but treats all other characters literally. After lexing, the token is flagged with W_QUOTED but NOT W_NOGLOB — however, the expansion engine checks the quoted flag and skips the word_split() and glob_expand() stages for quoted regions. The key function is string_extract_double_quoted() in subst.c, which recursively processes nested $() and ${} constructs while maintaining the quoting context.

bash
#!/bin/bash
name="Mari"
path="/home/mari/my project"     # has a space!
pattern="*.py"

# Double quotes: variables expand, spaces are preserved
echo "Hello $name"                # Hello Mari  (expanded!)
echo "Date: $(date +%Y-%m-%d)"   # Date: 2026-03-15  (command runs!)
echo "Path: $path"                # Path: /home/mari/my project  (space preserved!)
echo "Pattern: $pattern"           # Pattern: *.py  (NOT expanded to filenames!)

# THE CRITICAL DIFFERENCE — with and without quotes:
files="file one.txt   file two.txt"

# Without quotes: word splitting happens
for f in $files; do
    echo "  NO QUOTES: [$f]"
done
# Prints 4 iterations: [file] [one.txt] [file] [two.txt]  ← BROKEN

# With quotes: stays as one unit
echo "  QUOTED: [$files]"
# Prints: [file one.txt   file two.txt]  ← CORRECT
Expected Output
Hello Mari
Date: 2026-03-15
Path: /home/mari/my project
Pattern: *.py
  NO QUOTES: [file]
  NO QUOTES: [one.txt]
  NO QUOTES: [file]
  NO QUOTES: [two.txt]
  QUOTED: [file one.txt   file two.txt]

BASH-F03.4 — Command Substitution: $() and Backticks

Command substitution lets you capture the output of a command and use it as text. Wrap any command in $(command) and bash runs it in a subshell, captures everything it prints to stdout, strips trailing newlines, and substitutes the result in place. It is like asking a coworker a question and using their answer in your sentence: "The server has $(df -h / | awk 'NR==2{print $5}') disk usage" becomes "The server has 42% disk usage."

You may see backtick syntax `command` in older scripts — it does the same thing but is harder to read and cannot be nested. Always prefer $(command).

The $() syntax forks a subshell, redirects its stdout to a pipe, executes the enclosed command(s) in that subshell, reads the pipe output into a buffer, strips all trailing newlines from the buffer, and substitutes the resulting string into the command line. Unlike backticks, $() supports clean nesting: $(echo $(date)) works without escaping. The subshell inherits the parent's environment but runs in a separate process — changes to variables inside $() do not affect the parent. If the command produces no output, the substitution results in an empty string.

bash
#!/bin/bash
# ═══ Command Substitution ═══

# Store command output in a variable
kernel=$(uname -r)
ip_addr=$(hostname -I | awk '{print $1}')
git_branch=$(git branch --show-current 2>/dev/null || echo "not a repo")
line_count=$(wc -l < ~/.zshrc)

echo "Kernel: $kernel"
echo "IP: $ip_addr"
echo "Branch: $git_branch"
echo "Zshrc has $line_count lines"

# Nested substitution (try this with backticks — impossible to read!)
echo "Config dir has $(ls $(echo ~/.config) | wc -l) items"

# Use in conditionals
if [[ $(whoami) == "root" ]]; then
    echo "⚠️ Running as root!"
else
    echo "✓ Running as normal user"
fi

# Timestamp in filenames
backup_name="backup_$(date +%Y%m%d_%H%M%S).tar.gz"
echo "Will create: $backup_name"
Expected Output
Kernel: 6.12.8-arch1-1
IP: 192.168.178.42
Branch: main
Zshrc has 85 lines
Config dir has 23 items
✓ Running as normal user
Will create: backup_20260315_143022.tar.gz

BASH-F03.5 — Brace & Tilde Expansion

Brace expansion is bash's copy-paste shortcut. Instead of typing mkdir src tests docs config, you type mkdir {src,tests,docs,config}. Bash expands the braces into all the listed words. You can also do ranges: {1..10} generates numbers 1 through 10, and {a..z} generates the entire alphabet. This is incredibly useful for creating directory structures, generating file lists, and writing compact commands.

Tilde expansion is simpler: ~ expands to your home directory. ~mari expands to user mari's home directory. It only works at the beginning of a word and only when unquoted.

Brace expansion happens FIRST in bash's expansion pipeline — before variable expansion. This means {$a,$b} does NOT work as expected (the braces are expanded before $a and $b are resolved). Brace expansion produces a purely textual substitution: pre{A,B}suf → preAsuf preBsuf. Ranges support zero-padding: {01..10} → 01 02 03 ... 10. Tilde expansion calls getpwnam() to look up the user's home directory from /etc/passwd.

bash
#!/bin/bash
# ═══ Brace Expansion ═══

# Comma-separated list
echo {apple,banana,cherry}
# Output: apple banana cherry

# Create project structure in ONE command
mkdir -p project/{src,tests,docs,config,scripts}

# Number range
echo {1..5}                       # 1 2 3 4 5
echo {01..12}                     # 01 02 03 ... 12 (zero-padded!)
echo {a..f}                       # a b c d e f

# Step value
echo {0..20..5}                   # 0 5 10 15 20

# Combine with prefix/suffix
echo file{1..3}.txt               # file1.txt file2.txt file3.txt
echo /var/log/{syslog,auth,kern}.log

# Backup trick: rename file by adding .bak
cp config.yml{,.bak}              # expands to: cp config.yml config.yml.bak

# ═══ Tilde Expansion ═══
echo ~                             # /home/mari
echo ~/projects                   # /home/mari/projects
echo '~'                           # ~ (literal! single quotes block expansion)

Exercise 2 (Medium) — Safe File Handler

Write a script that takes a filename as an argument (which may contain spaces) and prints its size, line count, and word count. Handle the case where the file does not exist. Use proper quoting throughout.

solution
#!/bin/bash
file="${1:?Usage: $0 <filename>}"

if [[ ! -f "$file" ]]; then
    echo "Error: '$file' not found" >&2
    exit 1
fi

size=$(stat -c %s "$file")
lines=$(wc -l < "$file")
words=$(wc -w < "$file")

echo "File:  $file"
echo "Size:  $size bytes"
echo "Lines: $lines"
echo "Words: $words"
