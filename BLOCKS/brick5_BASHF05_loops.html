BASH-F05.1 — for Loops

A for loop is like going through a checklist. You have a list of items, and for each item, you do something. "For each file in this directory, check if it is a Python file. For each server in my list, ping it. For each package in my install list, install it." The structure is: for item in list; do something; done. The variable item takes on each value from the list, one at a time, and the body runs once for each value.

Bash has two styles of for loop. The "for-in" loop iterates over a list of words (the most common style). The "C-style" loop uses for ((i=0; i<10; i++)) syntax, which feels familiar if you know C, Python's range(), or JavaScript. Both are useful in different situations.

The for name in word-list; do commands; done form expands the word-list (applying all normal expansion rules: brace, tilde, parameter, command substitution, word splitting, and pathname expansion), then iterates over the resulting words, binding each to name and executing the command list. If in word-list is omitted, the loop iterates over $@ (the script's positional parameters). The C-style for ((expr1; expr2; expr3)); do commands; done evaluates expr1 once, then loops: evaluate expr2 (continue if non-zero), execute body, evaluate expr3. All three expressions are arithmetic contexts — no $ needed for variables.

Internally, the for-in loop is represented as a FOR_COM structure containing the variable name, the word list (a WORD_LIST linked list), and the body command. Execution (execute_for_command() in execute_cmd.c) first expands the word list, producing an array of strings. Then it iterates: for each string, it calls bind_variable(name, string) to update the loop variable in the shell's variable hash table, then calls execute_command(body). The C-style for loop is parsed into an ARITH_FOR_COM structure with three arithmetic expression strings. Each expression is evaluated by the arithmetic evaluator (converting strings to intmax_t, performing the operation, converting back). The loop variable lives in the normal variable namespace — there is no special loop scope.

bash
#!/bin/bash
# ═══ for-in Loop: Iterate Over a List ═══

# Simple word list
for color in pink green purple blue; do
    echo "🎨 $color"
done

# Loop over files (glob expansion)
echo "--- Python files ---"
for f in *.py 2>/dev/null; do
    echo "  📄 $f ($(wc -l < "$f") lines)"
done

# Loop over command output
echo "--- Running services (first 5) ---"
for svc in $(systemctl list-units --type=service --state=running --no-legend | head -5 | awk '{print $1}'); do
    echo "  ✓ $svc"
done

# Loop over array elements
tools=("nmap" "gobuster" "sqlmap" "nikto")
for tool in "${tools[@]}"; do
    if command -v "$tool" &>/dev/null; then
        echo "  ✅ $tool installed"
    else
        echo "  ❌ $tool NOT installed"
    fi
done

# ═══ Brace expansion + for (create numbered files) ═══
for i in {01..05}; do
    echo "Creating chapter_${i}.md"
done

# ═══ C-style for loop (counter-based) ═══
echo "--- Countdown ---"
for ((i=5; i>=1; i--)); do
    echo "  $i..."
done
echo "  🚀 Launch!"

✅ The golden rule for for-in loops:

Always double-quote "${array[@]}" when looping over arrays, so elements with spaces stay as single items. Without quotes, an element like "my file.txt" gets split into "my" and "file.txt" — two separate iterations instead of one.

❌ The classic trap — looping over command output:

for f in $(ls) breaks on filenames with spaces. Instead use for f in * (glob expansion handles spaces correctly) or find ... -print0 | while read -r -d '' f for recursive searches.

🧪 Why is for f in $(ls *.txt) dangerous?
Because filenames with spaces get word-split into separate tokens — "my report.txt" becomes two iterations: "my" and "report.txt"
Because ls is slow
Because it only works in bash, not zsh

BASH-F05.2 — while Loops

A while loop keeps running as long as a condition is true. Think of it as a guard standing at a door: "While the door is locked, keep trying to open it. Once it opens, stop." This is perfect for situations where you do not know in advance how many iterations you need — reading lines from a file (you do not know how many lines there are), waiting for a process to finish, retrying a failed network request, or monitoring something until a condition changes.

The most important use of while in bash is reading files line by line with while read. This is the correct and safe way to process text files, because it handles spaces, tabs, and special characters properly (unlike for line in $(cat file) which breaks on all of those).

The while command-list; do body-list; done construct repeatedly executes the body as long as the last command in command-list exits with status 0. The command-list is re-evaluated before each iteration. The while read -r line idiom reads from stdin one line at a time: read returns 0 on successful read and non-zero at EOF. The -r flag prevents backslash interpretation (so \n in the file is read as literal backslash-n, not a newline). Using IFS= before read preserves leading/trailing whitespace. Piping into a while loop creates a subshell, so variable changes inside the loop do not propagate to the parent — use process substitution or done < file redirection instead.

The while loop is compiled into a WHILE_COM structure containing the test command list and the body command list. Execution calls execute_command(test), checks last_command_exit_value, and if 0, calls execute_command(body), then repeats. The read builtin calls the kernel's read() syscall on stdin's file descriptor, reading one byte at a time until it encounters a newline (0x0A) or EOF. The bytes are accumulated in a buffer, trailing newline is stripped, and the result is assigned to the specified variable(s). If multiple variable names are given, the line is split on IFS characters and assigned left-to-right, with the last variable getting all remaining text. The subshell problem with pipes exists because the pipe operator causes both sides to run in forked subprocesses — the while loop's fork cannot modify the parent shell's variable space because they have separate copies of memory (copy-on-write page tables).

bash
#!/bin/bash
# ═══ while loop: Read a File Line by Line ═══

# THE CORRECT way to read files (handles spaces, special chars)
count=0
while IFS= read -r line; do
    ((count++))
    echo "  Line $count: $line"
done < /etc/hostname
echo "Total: $count lines"

# ═══ Process command output (without subshell problem) ═══
total=0
while IFS= read -r line; do
    size=$(echo "$line" | awk '{print $1}')
    dir=$(echo "$line" | awk '{print $2}')
    echo "  📁 $dir → $size"
done < <(du -sh ~/.config/*/ 2>/dev/null | head -5)

# ═══ Counter-based while loop ═══
i=1
while [[ $i -le 5 ]]; do
    echo "  Attempt $i of 5"
    ((i++))
done

# ═══ Retry pattern (wait for something) ═══
max_retries=5
attempt=0
while [[ $attempt -lt $max_retries ]]; do
    if ping -c1 -W1 google.com &>/dev/null; then
        echo "✓ Online (attempt $((attempt+1)))"
        break
    fi
    ((attempt++))
    echo "  ⏳ Retry $attempt/$max_retries..."
    sleep 2
done
[[ $attempt -eq $max_retries ]] && echo "✗ Failed after $max_retries attempts"

✅ The safe pattern:

while IFS= read -r line; do ...; done < file is the canonical way to read files. The IFS= preserves whitespace, -r preserves backslashes, and < file avoids the subshell problem that piping creates.

❌ The subshell trap:

cat file | while read line; do count=$((count+1)); done; echo $count — the count is always 0! Because the pipe runs the while loop in a subshell, and the subshell's variable changes disappear when the pipe ends. Use done < file instead.

🧪 Why does cat file | while read line; do ((n++)); done; echo $n print nothing (or 0)?
Because the pipe runs the while loop in a subshell — changes to $n inside the subshell are lost when the pipe ends
Because read cannot handle the output from cat
Because ((n++)) has a syntax error

BASH-F05.3 — until Loops

An until loop is the exact mirror image of a while loop. While says "keep going while this is true." Until says "keep going until this becomes true." In practice, until is used less often than while, but it reads more naturally in certain situations — especially waiting for something to happen: "Until the server responds, keep retrying." You can always rewrite an until as a while with a negated condition (using !), so choose whichever reads more clearly.

The until command-list; do body-list; done construct executes the body as long as the last command in command-list exits with a non-zero status. It is logically equivalent to while ! command-list; do body-list; done. The internal representation is the same WHILE_COM structure with an inverted test flag. Typical use case: polling loops where you wait for a resource to become available.

bash
#!/bin/bash
# ═══ until: Wait for a Service to Start ═══

echo "Waiting for PostgreSQL..."
until pg_isready -q 2>/dev/null; do
    echo "  ⏳ Not ready yet, waiting 1s..."
    sleep 1
done
echo "✓ PostgreSQL is ready!"

# ═══ until: Wait for a File to Appear ═══
echo "Waiting for /tmp/signal.lock..."
until [[ -f /tmp/signal.lock ]]; do
    sleep 0.5
done
echo "✓ Signal file detected!"
Expected Output
Waiting for PostgreSQL...
  ⏳ Not ready yet, waiting 1s...
  ⏳ Not ready yet, waiting 1s...
✓ PostgreSQL is ready!

BASH-F05.4 — select Menus (Interactive User Choice)

The select loop creates a numbered menu that asks the user to pick an option. It is a quick way to build interactive scripts without manually coding the menu display and input handling. Bash automatically prints the numbered list, shows a prompt (PS3), reads the user's number, and sets the variable to the corresponding choice. The loop repeats until you break out of it, which makes it perfect for "pick an action" menu interfaces.

The select name in word-list; do commands; done construct expands the word-list, displays the items as a numbered menu to stderr, prints the PS3 prompt (default: "#? "), and reads a line from stdin. If the input is a valid number, the corresponding word is assigned to name and REPLY holds the raw input. If the input is empty (just Enter), the menu is re-displayed. If the input is invalid, name is set to empty string. The loop continues until a break statement is encountered or EOF is received on stdin.

bash
#!/bin/bash
# ═══ select: Interactive Tool Launcher ═══

PS3="🌸 Pick a tool (number): "     # custom prompt

select tool in "nmap scan" "wireshark" "burp suite" "system info" "quit"; do
    case "$tool" in
        "nmap scan")
            echo "🔍 Running: nmap -sC -sV localhost"
            ;;
        "wireshark")
            echo "🦈 Launching Wireshark..."
            ;;
        "burp suite")
            echo "🔥 Launching Burp Suite..."
            ;;
        "system info")
            echo "ℹ️ Kernel: $(uname -r), Uptime: $(uptime -p)"
            ;;
        "quit")
            echo "👋 Bye!"
            break
            ;;
        *)
            echo "❌ Invalid choice: $REPLY"
            ;;
    esac
done

BASH-F05.5 — break & continue (Loop Control)

break is the emergency exit — it immediately stops the loop entirely and jumps to the code after done. Use it when you have found what you are looking for or when an error means continuing would be pointless. continue is the skip button — it stops the current iteration and jumps back to the top of the loop for the next one. Use it when the current item does not match your criteria and you want to skip it without stopping the entire loop.

Think of a for loop as walking through a grocery store aisle. continue means "this item is not what I need, move to the next shelf." break means "I found what I need, leave the aisle entirely."

Both break and continue accept an optional numeric argument specifying how many enclosing loops to break/continue from. break 2 exits two levels of nested loops. Without an argument, they affect the innermost loop only. The break builtin sets an internal flag (breaking) and the loop execution code checks this flag after each iteration. continue sets continuing flag, causing the loop to skip the remaining body and re-evaluate the condition. Both return exit status 0 when used within a loop and non-zero when used outside a loop (which is a common error).

bash
#!/bin/bash
# ═══ break: Stop Early ═══

# Find the first .conf file in /etc
echo "Searching for first .conf file..."
for f in /etc/*; do
    if [[ "$f" == *.conf ]]; then
        echo "  Found: $f"
        break      # stop searching, we found one
    fi
done

# ═══ continue: Skip Items ═══

# Process only .py files, skip everything else
echo "--- Python files only ---"
for f in *; do
    if [[ "$f" != *.py ]]; then
        continue   # skip non-Python files
    fi
    echo "  Processing: $f"
done

# ═══ Practical: Process CSV, skip header and empty lines ═══
line_num=0
while IFS=',' read -r name email role; do
    ((line_num++))
    # Skip header
    [[ $line_num -eq 1 ]] && continue
    # Skip empty lines
    [[ -z "$name" ]] && continue
    echo "  User: $name ($email) — $role"
done << 'CSV'
name,email,role
Mari,mari@galaxy.dev,admin
Luna,luna@galaxy.dev,developer

Kai,kai@galaxy.dev,analyst
CSV
Expected Output

  Processing: utils.py

BASH-F05.6 — Practice Exercises

Exercise 1 (Easy) — Bulk File Creator

Write a script that creates 10 numbered Markdown files in a directory called notes/: note_01.md through note_10.md. Each file should contain a header with its number (e.g., # Note 01).

solution
#!/bin/bash
mkdir -p notes
for i in {01..10}; do
    echo "# Note $i" > "notes/note_${i}.md"
    echo "Created notes/note_${i}.md"
done
echo "✓ Created $(ls notes/ | wc -l) files"

Exercise 2 (Medium) — Port Scanner

Write a simple port scanner that checks ports 20-25, 80, 443, and 8080 on localhost. For each port, use /dev/tcp/ (bash's built-in TCP feature) to test if it is open. Print ✓ for open and ✗ for closed.

solution
#!/bin/bash
# ═══ Simple Port Scanner ═══

host="localhost"
ports=(20 21 22 23 24 25 80 443 8080)

echo "Scanning $host..."
for port in "${ports[@]}"; do
    # Bash's /dev/tcp is a virtual device for TCP connections
    # timeout after 1 second to avoid hanging on closed ports
    if (echo >/dev/tcp/$host/$port) 2>/dev/null; then
        echo "  ✅ Port $port — OPEN"
    else
        echo "  ✗  Port $port — closed"
    fi
done
echo "Scan complete."

Exercise 3 (Hard) — Log Analyzer

Write a script that reads /var/log/pacman.log (Arch Linux's package manager log), counts how many packages were installed, upgraded, and removed in the last 7 days, and prints a summary. Use while read, case, and date comparison.

solution
#!/bin/bash
# ═══ Pacman Log Analyzer — Last 7 Days ═══

logfile="/var/log/pacman.log"
cutoff=$(date -d "7 days ago" +%Y-%m-%d)

installed=0
upgraded=0
removed=0

if [[ ! -r "$logfile" ]]; then
    echo "Cannot read $logfile" >&2
    exit 1
fi

while IFS= read -r line; do
    # Extract date from log format: [2026-03-10T14:22:01+0100]
    log_date="${line:1:10}"
    
    # Skip lines older than cutoff
    [[ "$log_date" < "$cutoff" ]] && continue
    
    case "$line" in
        *"[ALPM] installed"*)   ((installed++)) ;;
        *"[ALPM] upgraded"*)    ((upgraded++)) ;;
        *"[ALPM] removed"*)     ((removed++)) ;;
    esac
done < "$logfile"

echo "═══ Pacman Activity (last 7 days) ═══"
echo "  📦 Installed: $installed packages"
echo "  ⬆️  Upgraded:  $upgraded packages"
echo "  🗑️  Removed:   $removed packages"
echo "  📊 Total:     $((installed + upgraded + removed)) operations"
Expected Output
═══ Pacman Activity (last 7 days) ═══
  📦 Installed: 23 packages
  ⬆️  Upgraded:  47 packages
  🗑️  Removed:   3 packages
  📊 Total:     73 operations
