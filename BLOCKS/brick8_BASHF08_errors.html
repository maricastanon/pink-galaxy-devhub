BASH-F08.1 — Exit Codes: The Language of Success and Failure

Every command in Linux, when it finishes running, leaves behind a number called its exit code. This number is like a traffic light for the next command. Exit code 0 means green — "everything went fine, proceed." Any other number (1 through 255) means red — "something went wrong." The specific number gives a hint about what went wrong, though conventions vary by program. You check the exit code of the last command with the special variable $?.

This system is the backbone of how if statements, &&, ||, and set -e all work — they all inspect exit codes to decide what to do next. Understanding exit codes means understanding how bash decides whether your script is succeeding or failing at every single step.

Exit codes are integers 0-255. By POSIX convention, 0 means success and non-zero means failure. Common conventions for non-zero codes include 1 for general errors, 2 for misuse of shell builtins (e.g., wrong arguments), 126 for permission denied (command found but not executable), 127 for command not found, and 128+N for termination by signal N (e.g., 130 = SIGINT/Ctrl+C, 137 = SIGKILL, 143 = SIGTERM). Your own scripts should exit 0 on success and exit 1 (or another meaningful code) on failure. Functions use return instead of exit — exit terminates the entire script, return only exits the function.

When a process calls exit(n) (the C library call), it triggers the _exit() syscall which sets the process's exit status in its task_struct and transitions the process to the zombie state (Z). The parent process (bash) calls waitpid() which reads the 16-bit status word. The low 8 bits contain the signal number if the child was killed by a signal, and bits 8-15 contain the exit code if the child exited normally. Bash extracts this with the WEXITSTATUS() macro and stores it in last_command_exit_value, accessible as $?. Because only 8 bits are used for the exit code, the range is 0-255. If your script does exit 256, bash actually stores 0 (256 mod 256). If you do exit -1, bash stores 255 (unsigned interpretation of -1 in 8 bits).

bash
#!/bin/bash
# ═══ Exit Codes in Action ═══

# Every command sets $?
ls /etc/pacman.conf
echo "ls exit code: $?"           # 0 (file exists)

ls /nonexistent 2>/dev/null
echo "ls exit code: $?"           # 2 (file not found)

grep -q "root" /etc/passwd
echo "grep found: $?"             # 0 (pattern found)

grep -q "zzzzz" /etc/passwd
echo "grep not found: $?"         # 1 (pattern not found)

# Using exit codes in your own scripts
check_root() {
    if [[ "$(id -u)" -ne 0 ]]; then
        echo "Error: must be run as root" >&2
        return 1    # function: use return, not exit
    fi
    return 0
}

# Common exit code conventions:
# 0   = success
# 1   = general error
# 2   = misuse (bad arguments)
# 126 = command not executable
# 127 = command not found
# 130 = killed by Ctrl+C (128 + SIGINT=2)
# 137 = killed by SIGKILL (128 + 9)

BASH-F08.2 — The Holy Trinity: set -euo pipefail

By default, bash scripts are dangerously permissive. If a command fails, bash shrugs and continues to the next line. If you use a variable you never set, bash silently uses an empty string. If a command in the middle of a pipeline fails, bash only checks the last command's exit code. This means your script can be failing catastrophically and you would never know until the damage is done.

The line set -euo pipefail enables three safety nets that fix all of these problems. Think of it as putting on your seatbelt, turning on the headlights, and locking the doors before driving. It should be the SECOND line of every script you ever write (after the shebang). Let me break down what each letter does.

-e (errexit): "Stop immediately if any command fails." Like a circuit breaker that trips when something shorts out, preventing the damage from spreading.

-u (nounset): "Treat undefined variables as errors." Like a spell-checker that catches typos in variable names before they cause silent data loss.

-o pipefail: "A pipeline fails if ANY stage fails, not just the last one." Like a quality inspector at every station on the assembly line, not just at the end.

set -e (errexit) causes bash to exit immediately when any simple command returns a non-zero exit status. Exceptions: commands in if conditions, commands before && or ||, commands in while/until conditions, and negated commands (! cmd). set -u (nounset) causes bash to treat unset variables as errors, printing a message and exiting. This catches typos like $HOEM instead of $HOME. Exceptions: ${var:-default} and ${var:+alternate} still work because they explicitly handle the unset case. set -o pipefail changes pipeline exit status from "status of the last command" to "status of the rightmost command that failed." Without it, bad_cmd | good_cmd exits 0 (good_cmd's status). With it, it exits non-zero (bad_cmd's status). Combined, these three options make bash behave more like a strict programming language where errors are caught immediately rather than silently propagated.

The -e flag sets the errexit_flag in bash's internal state. After each simple command execution in execute_command_internal(), bash checks last_command_exit_value; if it is non-zero and errexit_flag is set, bash calls exit_shell() with that status. The exceptions are implemented as context flags: ERR_TRAP_HANDLED suppresses errexit in conditional contexts. The -u flag sets unbound_vars_is_error; when the expansion engine (parameter_brace_expand() in subst.c) encounters an unset variable, it checks this flag and calls err_unboundvar() instead of returning an empty string. The pipefail option changes execute_pipeline(): instead of returning only the last process's exit status, it iterates through all PIPESTATUS entries and returns the highest non-zero value (rightmost failure). The common set -euo pipefail incantation is processed as three separate set operations: -e, -u, and -o pipefail.

bash
#!/bin/bash
# ═══ THE STANDARD HEADER FOR EVERY PRODUCTION SCRIPT ═══
set -euo pipefail

# ─── What -e does (errexit) ───
# WITHOUT -e: script continues after failure (DANGEROUS)
# cd /nonexistent_directory    ← fails silently
# rm -rf *                     ← now deletes files in WRONG directory!
# WITH -e: script stops at the cd failure. rm never runs. Disaster averted.

# ─── What -u does (nounset) ───
# WITHOUT -u:
# echo "Home is: $HOEM"        ← typo! silently prints "Home is: "
# WITH -u:
# echo "Home is: $HOEM"        ← ERROR: HOEM: unbound variable (script stops)

# ─── What pipefail does ───
# WITHOUT pipefail:
# bad_command | sort | head     ← exit code is 0 (from head)
# WITH pipefail:
# bad_command | sort | head     ← exit code is non-zero (from bad_command)

# ═══ Safe Patterns That Work WITH set -e ═══

# Pattern: Allow a command to fail (check it yourself)
result=$(grep "needle" haystack.txt 2>/dev/null) || true
# The || true prevents set -e from killing the script when grep finds nothing

# Pattern: Check if a command exists before using it
if command -v docker &>/dev/null; then
    docker ps     # safe — we know it exists
else
    echo "Docker not installed, skipping"
fi

# Pattern: Safe default for potentially unset variables
target="${1:-production}"      # default value prevents -u error
verbose="${VERBOSE:-false}"    # safe env var access

echo "Deploying to: $target (verbose: $verbose)"
Expected Output
Deploying to: production (verbose: false)
✅ The line that saves your career:

set -euo pipefail on line 2 of every script. It catches failed commands, typos in variable names, and hidden pipeline failures. The 5 seconds it takes to type this line can prevent hours of debugging or worse — silent data corruption in production.

❌ Without strict mode, this happens:

cd "$DEPLOY_DIR" fails because the variable is misspelled. Script continues. rm -rf ./* runs in the HOME directory instead of the deploy directory. Everything is deleted. This is not hypothetical — this exact scenario has caused real production outages at major companies. set -euo pipefail prevents it entirely.

Bash prints "ENVVAR: unbound variable" and the script exits immediately — protecting you from using empty values that could cause damage
It prompts the user to enter a value

BASH-F08.3 — trap: Guaranteed Cleanup

The trap command registers a function (or command) that runs automatically when your script exits — no matter HOW it exits. Whether the script completes normally, crashes due to an error, or gets killed by Ctrl+C, the trap fires. This is your "finally" block, your safety net, your guarantee that temporary files get cleaned up, locks get released, and resources get freed.

Think of it like hiring a cleaning crew that activates when you leave a room, regardless of whether you walk out calmly, run out in a panic, or are carried out by emergency services. The room always gets cleaned.

The trap 'commands' SIGNAL_LIST builtin registers a handler for the specified signals. The most important pseudo-signal is EXIT — it fires when the shell exits for any reason (normal completion, exit call, errexit trigger, or fatal signal). Other useful signals include ERR (fires after any command returns non-zero, if errexit is not set), INT (Ctrl+C), and TERM (termination request). Traps execute in the current shell, not a subshell, so they can access all script variables. Multiple traps can be set on different signals. Setting a trap on EXIT combined with set -e is the canonical pattern for resource cleanup in bash scripts.

Bash maintains an internal array of trap handlers (trap_list[] in trap.c), indexed by signal number. When a signal is received (via the kernel's signal delivery mechanism), bash's signal handler (sighandler()) sets a flag in pending_traps[]. Between command executions, bash checks pending_traps[] and executes any registered handlers by calling parse_and_execute(trap_command). The EXIT pseudo-signal (index 0) is special: its handler is called from exit_shell() just before the process terminates. The ERR trap fires when last_command_exit_value != 0 and the command is not in a conditional context. Traps are inherited by subshells (but reset to SIG_DFL for non-interactive subshells spawned by command substitution, unless the shell is in POSIX mode).

bash
#!/bin/bash
set -euo pipefail

# ═══ Pattern 1: Basic Temp File Cleanup ═══

TMPDIR=$(mktemp -d)

cleanup() {
    local exit_code=$?    # capture BEFORE any cleanup commands change it
    echo "🧹 Cleaning up temp dir: $TMPDIR" >&2
    rm -rf "$TMPDIR"
    echo "🧹 Exiting with code: $exit_code" >&2
    exit "$exit_code"    # preserve the original exit code
}
trap cleanup EXIT

# Now use $TMPDIR freely — it WILL be cleaned up no matter what
echo "important data" > "$TMPDIR/data.txt"
echo "Working in: $TMPDIR"

# ═══ Pattern 2: Lock File (Prevent Concurrent Runs) ═══

LOCKFILE="/tmp/myscript.lock"

acquire_lock() {
    if [[ -f "$LOCKFILE" ]]; then
        echo "ERROR: Script already running (lockfile exists)" >&2
        exit 1
    fi
    echo $$ > "$LOCKFILE"    # write our PID
}

release_lock() {
    rm -f "$LOCKFILE"
}
trap release_lock EXIT
acquire_lock

# ═══ Pattern 3: Ctrl+C Handler ═══

on_interrupt() {
    echo ""    # newline after ^C
    echo "⚠️ Interrupted! Cleaning up gracefully..." >&2
    # do cleanup specific to interruption
    exit 130   # convention: 128 + signal number (SIGINT=2)
}
trap on_interrupt INT
Expected Output (normal run)
Working in: /tmp/tmp.xK3j2mN
🧹 Cleaning up temp dir: /tmp/tmp.xK3j2mN
🧹 Exiting with code: 0
✅ The cleanup pattern to memorize:

1. Create temp resources. 2. Register trap cleanup EXIT. 3. Work. The trap fires automatically whether the script succeeds, fails, or is interrupted. Capture $? as the first line inside the cleanup function to preserve the original exit code.

❌ Without trap:

If your script creates a temp file and then crashes halfway through, the temp file stays forever. If it creates a lock file and crashes, no future run of the script can ever start (permanent deadlock). If it opens a network connection and crashes, the connection leaks. Traps prevent all of these.

🧪 Why must you capture $? on the first line of your trap function?
Because every command you run inside the trap function changes $? — if you don't capture it immediately, you lose the original exit code that caused the trap to fire
Because $? is automatically cleared when the trap starts
Because trap functions cannot read $?

BASH-F08.4 — Defensive Scripting Patterns

Defensive scripting means assuming everything that CAN go wrong WILL go wrong, and coding accordingly. Files might not exist. Directories might not be writable. Network connections might time out. Variables might be empty. Users might pass garbage arguments. Disks might be full. Commands might not be installed. A defensive script checks every assumption before acting on it, and fails loudly with a helpful error message rather than silently proceeding into disaster.

Key defensive patterns: (1) Validate all inputs before use. (2) Check prerequisites (required commands, files, permissions) early. (3) Use cd dir || exit 1 instead of bare cd dir. (4) Use ${var:?error message} to enforce required variables. (5) Quote all variable expansions. (6) Use mktemp for temporary files (never hardcoded names — race condition vulnerability). (7) Use lock files for scripts that should not run concurrently. (8) Always specify full paths for critical operations (rm, mv) to avoid PATH hijacking. (9) Redirect errors to stderr, data to stdout. (10) Test with empty inputs, missing files, and permission denied scenarios.

bash
#!/bin/bash
set -euo pipefail

# ═══ Pattern: Required Variable with Helpful Error ═══
DEPLOY_DIR="${DEPLOY_DIR:?ERROR: DEPLOY_DIR environment variable is not set}"
# If DEPLOY_DIR is unset, this prints the error message and exits immediately

# ═══ Pattern: Safe cd (never rm in wrong directory) ═══
cd "$DEPLOY_DIR" || {
    echo "FATAL: Cannot cd to $DEPLOY_DIR" >&2
    exit 1
}

# ═══ Pattern: Require commands to exist before use ═══
require_cmd() {
    local cmd="$1"
    if ! command -v "$cmd" &>/dev/null; then
        echo "FATAL: Required command '$cmd' not found. Install it first." >&2
        exit 1
    fi
}
require_cmd "git"
require_cmd "docker"
require_cmd "jq"

# ═══ Pattern: Validate Arguments ═══
if [[ $# -lt 1 ]]; then
    cat >&2 << USAGE
Usage: $(basename "$0") 

Environments:
  staging     Deploy to staging
  production  Deploy to production (requires confirmation)

Example:
  $(basename "$0") staging
USAGE
    exit 2
fi

# ═══ Pattern: Confirm Destructive Operations ═══
env="$1"
if [[ "$env" == "production" ]]; then
    read -rp "⚠️ Deploy to PRODUCTION? Type 'yes' to confirm: " confirm
    if [[ "$confirm" != "yes" ]]; then
        echo "Cancelled."
        exit 0
    fi
fi

# ═══ Pattern: Safe Temporary Files ═══
tmpfile=$(mktemp)          # random name (e.g., /tmp/tmp.xK3j2mN)
tmpdir=$(mktemp -d)       # random directory
trap 'rm -rf "$tmpfile" "$tmpdir"' EXIT
# NEVER use /tmp/myfile.txt — multiple runs create race conditions
🧪 Why should you never use /tmp/myscript_temp.txt as a temp file name?
Because /tmp is too slow
Because if two copies of the script run simultaneously, they overwrite each other's temp file (race condition). Also, an attacker could pre-create that file as a symlink to a sensitive file. mktemp generates random, unique names that prevent both problems.
Because Linux does not allow writing to /tmp

BASH-F08.5 — Error Functions & Structured Logging

Rather than scattering echo "error" >&2; exit 1 throughout your script, create a reusable library of error and logging functions. This gives you consistent formatting, timestamps, color coding, and the ability to write to both the terminal AND a log file simultaneously. When something goes wrong at 3 AM and you are reading the logs the next morning, you will be grateful for structured, timestamped, color-coded messages that tell you exactly what happened, when, and where in the script.

A production logging library typically provides functions at multiple severity levels (DEBUG, INFO, WARN, ERROR, FATAL), each writing to stderr (not stdout, so they do not interfere with data pipelines). Each message includes a timestamp (ISO 8601), the log level, and optionally the function name and line number (via ${FUNCNAME[1]} and ${BASH_LINENO[0]}). The FATAL function should call exit 1 after logging. A die function (common convention) combines error message + exit into one call. Using tee or file descriptor duplication, you can mirror logs to a file while still showing them on the terminal.

bash
#!/bin/bash
set -euo pipefail

# ═══ Production Logging Library ═══

LOG_FILE="/tmp/$(basename "$0" .sh).log"

_log() {
    local level="$1"; shift
    local color="$1"; shift
    local ts=$(date +%Y-%m-%dT%H:%M:%S)
    local caller="${FUNCNAME[2]:-main}:${BASH_LINENO[1]:-0}"
    local msg="[$ts] [$level] [$caller] $*"
    echo -e "${color}${msg}\033[0m" >&2
    echo "$msg" >> "$LOG_FILE"
}

log_debug() { _log "DEBUG" "\033[37m"  "$@"; }
log_info()  { _log "INFO"  "\033[32m"  "$@"; }
log_warn()  { _log "WARN"  "\033[33m"  "$@"; }
log_error() { _log "ERROR" "\033[31m"  "$@"; }
die()       { _log "FATAL" "\033[1;31m" "$@"; exit 1; }

# ═══ Usage ═══

log_info "Script started"
log_info "Checking prerequisites..."

if ! command -v git &>/dev/null; then
    die "git is required but not installed"
fi

log_info "All checks passed"
log_warn "Disk usage is above 75%"
log_info "Deploying..."
log_info "Script completed successfully"

echo ""
echo "Log file: $LOG_FILE"
Expected Output (terminal — colored)

[2026-03-15T15:42:01] [INFO] [main:0] Checking prerequisites...
[2026-03-15T15:42:01] [INFO] [main:0] All checks passed
[2026-03-15T15:42:01] [WARN] [main:0] Disk usage is above 75%
[2026-03-15T15:42:01] [INFO] [main:0] Deploying...
[2026-03-15T15:42:01] [INFO] [main:0] Script completed successfully

BASH-F08.6 — Practice: The Complete Production Script Template

This final exercise brings together EVERYTHING from the entire Bash Fundamentals track — all 8 modules — into a single, complete, production-ready script template. This is the template you copy every time you start a new script. Study it carefully — every line exists for a reason.

bash — the complete template
#!/bin/bash
# ═══════════════════════════════════════════════════════════════
# 🌸 Pink Galaxy Script Template — Production Ready
# Description: [What this script does]
# Usage:       ./script.sh  [arg2]
# Author:      Mari
# Created:     2026-03-15
# ═══════════════════════════════════════════════════════════════

# ─── STRICT MODE (Brick 8: non-negotiable) ───
set -euo pipefail

# ─── CONSTANTS (Brick 2: variables) ───
readonly SCRIPT_NAME="$(basename "$0")"
readonly SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
readonly LOG_FILE="/tmp/${SCRIPT_NAME%.sh}.log"
readonly VERSION="1.0.0"

# ─── COLORS (Brick 2: variables; Brick 7: output) ───
readonly RED='\033[31m' GREEN='\033[32m' YELLOW='\033[33m' NC='\033[0m'

# ─── LOGGING (Brick 8: error handling; Brick 6: functions) ───
log_info()  { echo -e "${GREEN}[INFO]${NC} $*" >&2; echo "[$(date +%T)] [INFO] $*" >> "$LOG_FILE"; }
log_warn()  { echo -e "${YELLOW}[WARN]${NC} $*" >&2; echo "[$(date +%T)] [WARN] $*" >> "$LOG_FILE"; }
log_error() { echo -e "${RED}[ERROR]${NC} $*" >&2; echo "[$(date +%T)] [ERROR] $*" >> "$LOG_FILE"; }
die()       { log_error "$@"; exit 1; }

# ─── CLEANUP (Brick 8: trap) ───
TMPDIR=""
cleanup() {
    local exit_code=$?
    if [[ -n "$TMPDIR" && -d "$TMPDIR" ]]; then
        rm -rf "$TMPDIR"
        log_info "Cleaned up temp dir"
    fi
    if [[ $exit_code -ne 0 ]]; then
        log_error "Script failed with exit code $exit_code"
    else
        log_info "Script completed successfully"
    fi
    exit "$exit_code"
}
trap cleanup EXIT
trap 'echo ""; log_warn "Interrupted by user"; exit 130' INT

# ─── PREREQUISITES (Brick 8: defensive patterns; Brick 4: conditionals) ───
require_cmd() {
    command -v "$1" &>/dev/null || die "Required command not found: $1"
}

# ─── USAGE (Brick 7: heredoc + Brick 4: conditionals) ───
usage() {
    cat << HELP
${SCRIPT_NAME} v${VERSION} — [Description]

Usage: ${SCRIPT_NAME}  [options]

Arguments:
  environment    Target: staging | production

Options:
  -v, --verbose  Enable verbose output
  -d, --dry-run  Show what would be done without doing it
  -h, --help     Show this help message

Examples:
  ${SCRIPT_NAME} staging
  ${SCRIPT_NAME} production --dry-run
HELP
}

# ─── ARGUMENT PARSING (Brick 4: case; Brick 5: loops) ───
VERBOSE=false
DRY_RUN=false
ENVIRONMENT=""

parse_args() {
    while [[ $# -gt 0 ]]; do
        case "$1" in
            -v|--verbose)  VERBOSE=true ;;
            -d|--dry-run)  DRY_RUN=true ;;
            -h|--help)     usage; exit 0 ;;
            -*)            die "Unknown option: $1 (use --help)" ;;
            *)
                if [[ -z "$ENVIRONMENT" ]]; then
                    ENVIRONMENT="$1"
                else
                    die "Unexpected argument: $1"
                fi
                ;;
        esac
        shift
    done
    [[ -z "$ENVIRONMENT" ]] && { usage; die "Missing required argument: environment"; }
    [[ "$ENVIRONMENT" =~ ^(staging|production)$ ]] || die "Invalid environment: $ENVIRONMENT"
}

# ─── BUSINESS LOGIC (Brick 6: functions) ───
deploy() {
    local env="$1"
    
    log_info "Starting deployment to $env"
    
    if [[ "$env" == "production" ]]; then
        read -rp "⚠️ Deploy to PRODUCTION? Type 'yes': " confirm
        [[ "$confirm" == "yes" ]] || die "Deployment cancelled by user"
    fi
    
    if "$DRY_RUN"; then
        log_info "[DRY RUN] Would deploy to $env"
        return 0
    fi
    
    # ... actual deployment logic here ...
    log_info "Deployment to $env completed"
}

# ─── MAIN (Brick 6: main pattern) ───
main() {
    parse_args "$@"
    
    log_info "$SCRIPT_NAME v$VERSION starting"
    log_info "Environment: $ENVIRONMENT, Verbose: $VERBOSE, Dry-run: $DRY_RUN"
    
    # Check prerequisites
    require_cmd "git"
    require_cmd "docker"
    
    # Create temp workspace
    TMPDIR=$(mktemp -d)
    log_info "Temp dir: $TMPDIR"
    
    # Do the work
    deploy "$ENVIRONMENT"
}

# ─── ENTRY POINT (Brick 6: source-safe) ───
if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
    main "$@"
fi
