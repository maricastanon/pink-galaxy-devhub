BASH-F02.1 — Variable Assignment

A variable is a labeled box where you store a value. You give the box a name (like greeting) and put something inside it (like "Hello"). Later, when you need that value, you open the box by putting a dollar sign in front of the name: $greeting. The dollar sign is like saying "give me what's inside the box called greeting."

The critical rule that trips up every beginner: no spaces around the equals sign. In bash, name="Mari" creates a variable, but name = "Mari" tries to run a program called name with arguments = and "Mari". Bash is extremely literal about spaces.

In bash, variables are untyped — they are all stored as strings internally. When you write x=42, bash stores the string "42", not an integer. Arithmetic contexts ($(( )) and let) parse these strings as numbers when needed. Variable names are case-sensitive (Name and name are different variables), must start with a letter or underscore, and can contain letters, digits, and underscores. Assignment uses NAME=value with no spaces. Access uses $NAME or the safer ${NAME} form (braces disambiguate where the variable name ends).

Bash maintains an internal hash table (implemented as a chained hash map in the bash source code, file variables.c) mapping variable names to SHELL_VAR structs. Each struct contains the variable's value (a char*), attributes (integer, readonly, exported, array, etc.), and context level (for function-local scope). When you write x=42, the lexer tokenizes this as a WORD token, the parser identifies the = as an assignment operator (only when it appears before a command), and bind_variable("x", "42") is called. This either creates a new hash entry or updates the existing one's value pointer (freeing the old string via free()). When you reference $x, the expansion phase calls find_variable("x") which hashes the name and walks the chain to find the matching entry, returning the char* value.

bash
#!/bin/bash
# ═══ Variable Assignment Basics ═══

# Simple assignment — NO SPACES around =
name="Mari"
age=25
project="Pink Galaxy"

# Access with $ prefix
echo "Hello, $name!"
echo "Age: $age"
echo "Working on: $project"

# Braces for clarity (ALWAYS use when concatenating)
fruit="apple"
echo "I have 3 ${fruit}s"         # ✓ "I have 3 apples"
echo "I have 3 $fruits"          # ✗ empty — looks for variable "fruits"

# Command substitution — store command output in a variable
today=$(date +%Y-%m-%d)
kernel=$(uname -r)
file_count=$(ls | wc -l)

echo "Today is $today, kernel $kernel, $file_count files here"
Expected Output
Hello, Mari!
Age: 25
Working on: Pink Galaxy
I have 3 apples
I have 3
Today is 2026-03-15, kernel 6.12.8-arch1-1, 12 files here
✅ If you master variables:

Every script you write uses them. Config values, user input, command output, file paths, counters — all stored in variables. Understanding the no-spaces rule and brace syntax prevents 90% of beginner debugging time.

❌ Common mistakes:

Adding spaces around = (breaks silently). Forgetting braces when concatenating (wrong variable name). Using unquoted variables with spaces (word splitting breaks paths with spaces).

BASH-F02.2 — Strings

Strings are text — any sequence of characters. In bash, almost everything is a string: names, file paths, messages, even numbers (until you do math on them). You can join strings together (concatenation), measure their length, cut out pieces (substrings), and search/replace within them. Think of a string as a beaded necklace — each character is a bead, and you can count beads, cut the necklace at a certain point, or replace specific beads.

Bash strings are null-terminated C strings internally. There is no separate string type — all variables are strings by default. String operations are performed via parameter expansion (${#var} for length, ${var:offset:length} for substring, ${var/pattern/replacement} for substitution). These are evaluated during the expansion phase of command processing, before the command is executed. Double-quoted strings allow variable expansion and command substitution; single-quoted strings are completely literal (no expansion).

When bash processes ${#var}, it calls find_variable("var"), gets the char* value pointer, and runs strlen() on it. For multi-byte encodings (UTF-8 under LC_ALL), bash uses mbstowcs() to count characters rather than bytes. The substring operation ${var:2:5} computes a pointer offset into the string's char array. Pattern matching in ${var/pattern/replace} uses bash's internal pattern matcher (a glob-style matcher, not regex), which compiles the pattern into an internal representation and matches character by character against the value string.

bash
#!/bin/bash
# ═══ String Operations ═══

greeting="Hello, Pink Galaxy!"

# Length
echo "Length: ${#greeting}"              # 20 characters

# Substring (offset:length)
echo "Sub: ${greeting:7:4}"              # "Pink" (start at index 7, take 4)

# Search & Replace (first occurrence)
echo "${greeting/Pink/Dark}"             # "Hello, Dark Galaxy!"

# Replace ALL occurrences (double slash)
path="/home/mari/my project/docs"
echo "${path// /_}"                      # "/home/mari/my_project/docs"

# Remove prefix (shortest match)
file="/home/mari/projects/app.py"
echo "Filename: ${file##*/}"             # "app.py" (remove everything up to last /)

# Remove suffix
echo "Without ext: ${file%.py}"          # "/home/mari/projects/app"

# Uppercase / Lowercase (bash 4+)
name="mari"
echo "Upper: ${name^^}"                 # "MARI"
echo "Capital: ${name^}"                # "Mari" (first letter only)

# Concatenation (just put them next to each other)
first="Pink"
second="Galaxy"
combined="${first} ${second} Bible"
echo "$combined"                        # "Pink Galaxy Bible"
Expected Output
Length: 20
Sub: Pink
Hello, Dark Galaxy!
/home/mari/my_project/docs
Filename: app.py
Without ext: /home/mari/projects/app
Upper: MARI
Capital: Mari
Pink Galaxy Bible

BASH-F02.3 — Integers & Arithmetic

Bash does not natively understand decimals (floating-point numbers). It only does whole-number (integer) math. For anything with decimal points, you hand the calculation to an external tool like bc or Python. Integer math in bash uses the $(( )) syntax — double parentheses create an arithmetic context where bash temporarily treats strings as numbers.

Arithmetic expansion $(( expression )) evaluates integer expressions using C-like syntax. It supports + - * / % ** ++ -- and comparison operators. The result replaces the $(( )) in the command line. Inside (( )), variable names do not require the $ prefix. The declare -i attribute marks a variable as integer, making assignments automatically evaluate arithmetic. Division is integer division (truncates); use bc -l or awk for floating-point.

Inside $(( )), bash invokes its arithmetic evaluator (file expr.c), a recursive-descent parser that handles operator precedence identically to C. It calls atol() (or strtoimax()) to convert string values to intmax_t (typically 64-bit signed integer). Overflow is silent — it wraps around like C integer overflow. The ** exponentiation operator uses a simple loop multiplication (not pow()), so $((2**63)) is exact but $((2**64)) overflows to 0. The result is converted back to a string via itos() and substituted into the command line as a WORD token.

bash
#!/bin/bash
# ═══ Integer Arithmetic ═══

a=10
b=3

# Arithmetic expansion $(( ))
echo "Add: $((a + b))"          # 13
echo "Sub: $((a - b))"          # 7
echo "Mul: $((a * b))"          # 30
echo "Div: $((a / b))"          # 3 (integer division! truncated)
echo "Mod: $((a % b))"          # 1 (remainder)
echo "Pow: $((2 ** 10))"        # 1024

# Increment / decrement
count=0
((count++))                      # count is now 1
((count += 5))                   # count is now 6
echo "Count: $count"            # 6

# Floating point? Use bc or awk
result=$(echo "scale=2; 10/3" | bc)
echo "Float div: $result"       # 3.33

# Or use awk for quick math
awk 'BEGIN{printf "%.2f\n", 10/3}'   # 3.33
Expected Output
Add: 13
Sub: 7
Mul: 30
Div: 3
Mod: 1
Pow: 1024
Count: 6
Float div: 3.33
3.33

BASH-F02.4 — Arrays

An array is a list of values stored under one name. Instead of creating fruit1, fruit2, fruit3 as separate variables, you create one array called fruits that holds all of them. You access individual items by their position number (index), starting from 0. Think of an array as a numbered row of mailboxes — mailbox 0, mailbox 1, mailbox 2, etc. — all under the same street name.

Bash supports two array types: indexed arrays (integer keys, zero-based) and associative arrays (string keys, declared with declare -A). Arrays are sparse — you can set index 0 and index 100 without filling 1-99. ${arr[@]} expands to all elements (each as a separate word when quoted: "${arr[@]}"). ${#arr[@]} gives the count of set elements. ${!arr[@]} gives the list of indices/keys. Arrays cannot be exported to child processes.

Bash arrays are implemented as linked lists of ARRAY_ELEMENT structs (file array.c), each containing an arrayind_t index and a char* value. This means access by index is O(n) in the worst case (linear scan), not O(1) like C arrays. Associative arrays use the same hash table implementation as regular variables. When you write ${arr[5]}, bash walks the linked list comparing each element's index until it finds 5 or reaches the end. For large arrays (1000+ elements), this becomes noticeably slow compared to other languages.

bash
#!/bin/bash
# ═══ Arrays ═══

# Create an indexed array
tools=("nmap" "wireshark" "burp" "metasploit" "sqlmap")

# Access by index (0-based!)
echo "First:  ${tools[0]}"              # nmap
echo "Third:  ${tools[2]}"              # burp

# All elements
echo "All: ${tools[@]}"                 # nmap wireshark burp metasploit sqlmap

# Count
echo "Count: ${#tools[@]}"              # 5

# Loop through array
for tool in "${tools[@]}"; do
    echo "  → $tool"
done

# Add element
tools+=("hashcat")
echo "Now: ${#tools[@]} tools"           # 6

# Remove element (leaves a gap)
unset 'tools[2]'                        # removes "burp"
echo "After unset: ${tools[@]}"          # nmap wireshark metasploit sqlmap hashcat

# ═══ Associative Array (like a dictionary/map) ═══
declare -A ports
ports[ssh]=22
ports[http]=80
ports[https]=443
ports[mysql]=3306

echo "SSH runs on port ${ports[ssh]}"     # 22
echo "All services: ${!ports[@]}"         # ssh http https mysql (keys)
Expected Output
First:  nmap
Third:  burp
All: nmap wireshark burp metasploit sqlmap
Count: 5
  → nmap
  → wireshark
  → burp
  → metasploit
  → sqlmap
Now: 6 tools
After unset: nmap wireshark metasploit sqlmap hashcat
SSH runs on port 22
All services: ssh http https mysql

BASH-F02.5 — Environment Variables

Regular variables live only inside your current shell session — they are private notes on your desk. Environment variables are like notices posted on the office bulletin board: they are visible to every program you launch. When you run a program, it inherits copies of all environment variables. The most important one is PATH — it tells the shell where to find programs. If you install something and it says "command not found," the first thing to check is whether its location is in your PATH.

Environment variables are shell variables that have been export-ed. When bash calls fork()/exec() to run a child process, the kernel copies the parent's environment (an array of KEY=VALUE strings) into the child's memory space. The child can read these via getenv() in C or $VAR in bash, but modifying them in the child does NOT affect the parent (they are copies). Key environment variables: PATH (executable search path), HOME (user home directory), USER (username), SHELL (default shell), LANG (locale), EDITOR (default text editor).

The environment is passed to child processes through the third argument of execve(): execve(path, argv, envp) where envp is a NULL-terminated array of char* pointers, each pointing to a string of the form "KEY=VALUE". The kernel copies this array into the new process's stack during exec. In the child, the C library provides environ (a global char**) and getenv()/setenv()/unsetenv() functions. Bash's export command sets the att_exported attribute on a SHELL_VAR; only variables with this attribute are included in the envp array when fork/exec is called.

bash
#!/bin/bash
# ═══ Environment Variables ═══

# View important environment variables
echo "User: $USER"
echo "Home: $HOME"
echo "Shell: $SHELL"
echo "Path: $PATH"

# Create a regular variable (NOT inherited by children)
secret="local only"

# Export it (NOW inherited by children)
export MY_APP_ENV="production"

# One-shot: set env var for ONE command only
LANG=C sort file.txt           # sort using C locale, just for this command

# See all environment variables
env                                # or: printenv

# Add to PATH (common in .zshrc/.bashrc)
export PATH="$HOME/.local/bin:$PATH"    # prepend your local bin
Expected Output
User: mari
Home: /home/mari
Shell: /bin/zsh
Path: /home/mari/.local/bin:/usr/local/bin:/usr/bin:/bin

BASH-F02.6 — Scope (local vs export)

Variable scope is about who can see a variable. There are three levels. A local variable (inside a function with local) is like a whispered secret — only the function that created it can hear it. A shell variable (created at the top level without export) is like talking in a room — everyone in the same shell session hears it, but nobody in other rooms. An environment variable (export-ed) is like a posted announcement — every new program you launch gets a copy of it. But here is the key rule: children can read the announcement, but they cannot change the parent's copy. Changes only flow downward, never upward.

Functions in bash do NOT have local scope by default — variables set inside a function are global to the shell. You must explicitly declare local varname inside a function to create function-local scope. This is a common source of bugs: a function accidentally overwrites a global variable. Best practice: always use local for every variable inside functions. Subshells (commands in parentheses or pipes) get copies of all variables but changes do not propagate back to the parent. This is because subshells are forked processes with their own memory space.

bash
#!/bin/bash
# ═══ Scope Demonstration ═══

global_var="I'm global"

function demo() {
    local local_var="I'm local to demo()"
    leaked_var="I leaked! (no 'local' keyword)"
    echo "Inside: global=$global_var"
    echo "Inside: local=$local_var"
}

demo
echo "Outside: global=$global_var"         # ✓ works
echo "Outside: local=$local_var"           # ✗ empty (local to function)
echo "Outside: leaked=$leaked_var"         # ⚠ works! (this is a bug)
Expected Output
Inside: global=I'm global
Inside: local=I'm local to demo()
Outside: global=I'm global
Outside: local=
Outside: leaked=I leaked! (no 'local' keyword)
✅ Best practice:

Always use local inside functions. Always export variables that child processes need. Use readonly for constants that should never change.

❌ The leaked variable bug:

Forgetting local inside a function silently creates/overwrites a global variable. In a 200-line script with 20 functions, this creates impossible-to-debug race conditions where function A's variable gets clobbered by function B.

BASH-F02.7 — Practice Exercises

Exercise 1 (Easy)

Write a script that stores your name, the current date (using $(date)), and the number of files in the current directory (using $(ls | wc -l)) in variables, then prints a formatted greeting.

solution
#!/bin/bash
name="Mari"
today=$(date +"%A, %B %d")
files=$(ls | wc -l)
echo "Hello $name! Today is $today. You have $files files here."
Output
Hello Mari! Today is Saturday, March 15. You have 12 files here.

Exercise 2 (Medium)

Create an associative array mapping 5 BlackArch tool names to their categories (e.g., nmap→scanner, burpsuite→webapp). Loop through the array and print each tool with its category in a formatted table.

solution
#!/bin/bash
declare -A tools
tools[nmap]="scanner"
tools[burpsuite]="webapp"
tools[metasploit]="exploit"
tools[hashcat]="crypto"
tools[wireshark]="network"

printf "%-15s %s\n" "TOOL" "CATEGORY"
printf "%-15s %s\n" "────" "────────"
for tool in "${!tools[@]}"; do
    printf "%-15s %s\n" "$tool" "${tools[$tool]}"
done
Output
TOOL            CATEGORY
────            ────────
nmap            scanner
burpsuite       webapp
metasploit      exploit
hashcat         crypto
wireshark       network

Exercise 3 (Hard)

Write a script that takes a file path as an argument, extracts the directory, filename, and extension into separate variables using parameter expansion (no external commands like basename or dirname), and prints all three.

solution
#!/bin/bash
path="${1:?Usage: $0 <filepath>}"    # exit with message if no argument

dir="${path%/*}"                       # remove everything after last /
filename="${path##*/}"                 # remove everything before last /
name="${filename%.*}"                  # remove extension
ext="${filename##*.}"                  # keep only extension

echo "Full path:  $path"
echo "Directory:  $dir"
echo "Filename:   $filename"
echo "Name only:  $name"
echo "Extension:  $ext"

Directory:  /home/mari/projects
