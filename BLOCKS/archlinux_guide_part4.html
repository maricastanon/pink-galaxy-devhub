15.1 Firewall Hardening (ufw)

We set up ufw in Part 1 with basic deny-incoming/allow-outgoing defaults and an SSH allow rule. Now let's add rate limiting and application-specific rules for your development services.

bash
# Rate-limit SSH (blocks IPs that try more than 6 connections in 30 seconds)
sudo ufw limit ssh

# Allow local development ports (only from localhost)
sudo ufw allow from 127.0.0.1 to any port 3000   # Node.js dev server
sudo ufw allow from 127.0.0.1 to any port 5432   # PostgreSQL
sudo ufw allow from 127.0.0.1 to any port 6379   # Redis
sudo ufw allow from 127.0.0.1 to any port 8080   # local web apps

# Block all Docker-exposed ports from external access
# (Docker creates its own iptables rules that bypass ufw — this fixes it)
sudo tee /etc/docker/daemon.json << 'EOF'
{
    "iptables": false
}
EOF
sudo systemctl restart docker

# Verify firewall status
sudo ufw status verbose

15.2 fail2ban — Intrusion Prevention

bash
# Install fail2ban
sudo pacman -S fail2ban

# Create local config (never edit the default jail.conf directly)
sudo cat > /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
bantime  = 1h        # ban for 1 hour
findtime = 10m       # within a 10-minute window
maxretry = 5         # 5 failed attempts = ban
banaction = ufw      # use ufw to implement bans

[sshd]
enabled = true
port    = ssh
filter  = sshd
logpath = /var/log/auth.log
maxretry = 3         # stricter for SSH: 3 attempts
EOF

# Enable and start
sudo systemctl enable --now fail2ban

# Check status
sudo fail2ban-client status sshd
Expected Output
Status for the jail: sshd
|- Filter
|  |- Currently failed: 0
|  |- Total failed:     0
|  `- File list:        /var/log/auth.log
`- Actions
   |- Currently banned: 0
   |- Total banned:     0
   `- Banned IP list:

Technical detail: fail2ban monitors log files (in this case, /var/log/auth.log where sshd records failed login attempts) using configurable regex patterns. When an IP address triggers more than maxretry matches within findtime, fail2ban executes the ban action — in our case, it runs ufw insert 1 deny from <ip> to add a firewall rule blocking that IP. After bantime expires, the rule is automatically removed. This protects against brute-force SSH attacks, which are extremely common — any internet-facing SSH server receives thousands of automated login attempts per day.

15.3 Kernel Hardening (sysctl)

bash
# Create security-focused sysctl configuration
sudo cat > /etc/sysctl.d/99-security.conf << 'EOF'
# ═══ Pink Galaxy Security Hardening ═══

# Prevent IP spoofing
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1

# Disable ICMP redirect acceptance (prevents MITM via routing table poisoning)
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.default.accept_redirects = 0
net.ipv6.conf.all.accept_redirects = 0

# Disable source routing (prevents attackers from specifying packet path)
net.ipv4.conf.all.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0

# Enable SYN flood protection (SYN cookies)
net.ipv4.tcp_syncookies = 1

# Log suspicious packets (martians = packets with impossible source addresses)
net.ipv4.conf.all.log_martians = 1

# Disable IPv4 forwarding (you're a workstation, not a router)
net.ipv4.ip_forward = 0

# Restrict dmesg access to root
kernel.dmesg_restrict = 1

# Restrict kernel pointer exposure (prevents info leaks for exploits)
kernel.kptr_restrict = 2

# Restrict ptrace (prevents process memory snooping between users)
kernel.yama.ptrace_scope = 1
EOF

# Apply immediately
sudo sysctl --system

These sysctl settings harden the kernel's network stack and process security without affecting normal usage. The SYN cookie protection mitigates SYN flood DoS attacks by using a cryptographic challenge instead of allocating connection state for every incoming SYN packet. Restricting ptrace prevents one user's process from attaching a debugger to another user's process — a common privilege escalation technique. These settings persist across reboots because they are in /etc/sysctl.d/.

15.4 SSH Key Hardening

bash
# Harden SSH server configuration
sudo tee /etc/ssh/sshd_config.d/99-hardening.conf << 'EOF'
# Disable password authentication (key-based only)
PasswordAuthentication no
ChallengeResponseAuthentication no

# Disable root login entirely
PermitRootLogin no

# Use only strong ciphers
Ciphers aes256-gcm@openssh.com,chacha20-poly1305@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com
KexAlgorithms curve25519-sha256,curve25519-sha256@libssh.org

# Limit login attempts and sessions
MaxAuthTries 3
MaxSessions 3

# Disconnect idle sessions after 10 minutes
ClientAliveInterval 300
ClientAliveCountMax 2
EOF

# Restart SSH to apply
sudo systemctl restart sshd
⚠️ IMPORTANT — KEY-BASED AUTH

After enabling PasswordAuthentication no, you can ONLY log in via SSH key. Make sure you have already set up your SSH key pair (ssh-keygen -t ed25519) and copied the public key to the server (ssh-copy-id user@host) before restarting sshd. If you lock yourself out, you will need physical access to the machine to fix it.

16.1 Timeshift — System Snapshots

Layman version: Timeshift is like a "System Restore" for Linux. It takes snapshots of your system files (not your personal data) so that if a pacman update breaks something, you can roll back to a working state in minutes. Think of it as a save game before a boss fight — if you die, reload and try again.

bash
# Install Timeshift
paru -S timeshift

# Create your first snapshot (before any major changes)
sudo timeshift --create --comments "Fresh Arch install — Parts 1-4 complete"

# List snapshots
sudo timeshift --list

# Restore a snapshot (if something breaks)
# sudo timeshift --restore

# Configure automatic snapshots (daily + weekly)
sudo timeshift   # opens GUI for schedule configuration
# Recommended: Keep 5 daily + 3 weekly snapshots

# PRO TIP: Always run Timeshift before major updates:
# sudo timeshift --create --comments "pre-update" && sudo pacman -Syu
Expected Output

Snapshot saved to: /timeshift/snapshots/2026-03-15_14-30-00

16.2 Borg — Encrypted Data Backup

Timeshift protects the system, but your personal data (projects, dotfiles, documents, SSH keys) needs separate backup. Borg is a deduplicating, encrypting backup tool — it only stores the bytes that changed since the last backup, and everything is encrypted at rest.

bash
# Install Borg
sudo pacman -S borg

# Initialize a backup repository (on an external drive or NAS)
# Replace /mnt/backup with your backup location
borg init --encryption=repokey /mnt/backup/borg-repo

# Create a backup script
cat > ~/.local/bin/backup.sh << 'EOF'
#!/bin/bash
# ═══ Pink Galaxy Backup Script ═══
REPO="/mnt/backup/borg-repo"
TIMESTAMP=$(date +%Y-%m-%d_%H-%M)

echo "🌸 Starting backup: $TIMESTAMP"

borg create --progress --stats \
    "$REPO::home-$TIMESTAMP" \
    ~/projects \
    ~/Documents \
    ~/Pictures \
    ~/.config \
    ~/.local/bin \
    ~/.ssh \
    ~/.gitconfig \
    ~/.zshrc \
    ~/.tmux.conf \
    --exclude '*.pyc' \
    --exclude '__pycache__' \
    --exclude 'node_modules' \
    --exclude '.venv' \
    --exclude '.cache' \
    --exclude '*.o' \
    --exclude '.git/objects'

# Prune old backups: keep 7 daily, 4 weekly, 6 monthly
borg prune --stats "$REPO" \
    --keep-daily=7 \
    --keep-weekly=4 \
    --keep-monthly=6

echo "✅ Backup complete!"
EOF
chmod +x ~/.local/bin/backup.sh

# Run your first backup
~/.local/bin/backup.sh

# List backup archives
borg list /mnt/backup/borg-repo

17.5 Kernel Management

bash
# Install a fallback kernel (in case a kernel update causes issues)
sudo pacman -S linux-lts linux-lts-headers

# Create a boot entry for the LTS kernel
sudo cat > /boot/loader/entries/arch-lts.conf << 'EOF'
title   Arch Linux — LTS Kernel (Fallback)
linux   /vmlinuz-linux-lts
initrd  /intel-ucode.img
initrd  /initramfs-linux-lts.img
options root=UUID=YOUR-ROOT-UUID-HERE rw
EOF

# Now your boot menu has two options:
# 1. Arch Linux — Pink Galaxy 🌸 (latest kernel)
# 2. Arch Linux — LTS Kernel (stable fallback)
# If the latest kernel causes issues, select LTS at boot
