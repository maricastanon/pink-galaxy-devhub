11.1 Python Ecosystem (pyenv + Poetry)

Layman version: Python comes pre-installed on Arch, but for development you need more control — different projects may need different Python versions, and you want isolated environments so one project's dependencies don't break another. Pyenv lets you install and switch between any Python version (3.10, 3.11, 3.12, etc.), and Poetry manages your project's dependencies in a clean, reproducible way — like npm for Python, but better.

Technical detail: Pyenv works by inserting shim executables into your $PATH (at ~/.pyenv/shims/) that intercept calls to python, pip, etc. When you run python, the shim checks the .python-version file in the current directory (or the global setting) to determine which actual Python binary to delegate to. Each Python version is compiled from source and installed in ~/.pyenv/versions/. Poetry creates a pyproject.toml manifest and a poetry.lock file that pins exact dependency versions with SHA-256 hashes, ensuring deterministic builds — the same lockfile produces the same environment on any machine.

bash
# Install Python build dependencies (pyenv compiles Python from source)
sudo pacman -S python python-pip tk

# Install pyenv via paru
paru -S pyenv

# Add pyenv to your .zshrc (put these BEFORE the starship init line)
cat >> ~/.zshrc << 'EOF'
# Pyenv
export PYENV_ROOT="$HOME/.pyenv"
[[ -d $PYENV_ROOT/bin ]] && export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
EOF

# Reload shell
exec zsh

# Install Python 3.12 (takes 2-5 minutes — compiling from source)
pyenv install 3.12
pyenv global 3.12

# Verify
python --version

# Install Poetry
sudo pacman -S python-poetry

# Configure Poetry to create virtualenvs inside project directories
# (so .venv lives next to your code — VS Code detects it automatically)
poetry config virtualenvs.in-project true

# Example: start a new Python project
mkdir ~/projects/my-etl-pipeline && cd ~/projects/my-etl-pipeline
poetry init --no-interaction      # creates pyproject.toml
poetry add polars pandas boto3     # add your core libraries
poetry shell                        # activate the virtualenv
Expected Output
Python 3.12.x
Creating virtualenv my-etl-pipeline in /home/mari/projects/my-etl-pipeline/.venv
Using python3.12 (3.12.x)
Spawning shell within .venv

With this setup, every project gets its own isolated .venv directory containing its own Python binary and installed packages. VS Code automatically detects .venv in the project root and activates it when you open a terminal or run Python files. The pyproject.toml file is the modern replacement for setup.py and requirements.txt — it declares your project metadata, dependencies, build system, and tool configuration (Black, Ruff, pytest) all in one place.

11.3 Docker & Docker Compose

# Enable and start Docker daemon

# Add your user to the docker group (avoids needing sudo for every docker command)

docker run hello-world

# Verify Docker Compose
docker compose version
Expected Output
Hello from Docker!
This message shows that your installation appears to be working correctly.

Docker Compose version v2.x.x

Technical detail: Docker uses Linux kernel namespaces (pid, net, mnt, uts, ipc, user) to isolate containers from the host and from each other. Each container gets its own PID namespace (PID 1 inside the container is actually a child process of the Docker daemon on the host), its own network namespace (with its own IP address via a virtual Ethernet bridge), and its own mount namespace (sees only its own filesystem layers, built from the image's overlayfs). Cgroups (control groups) enforce resource limits — CPU, memory, I/O bandwidth. Your Dell 7420's Intel VT + VT-d (which we confirmed are enabled in BIOS) provide hardware virtualization acceleration that Docker can leverage for better performance in certain workloads.

The databricks-connect package is particularly powerful for your workflow: you write PySpark code locally in VS Code with full IDE support (autocomplete, type checking, debugging), but when you call spark.sql() or spark.read, the computation is transparently shipped to your Databricks cluster for execution. 13.1 Adding the BlackArch Repository

Layman version: BlackArch is not a separate Linux distribution — it is an additional software repository that layers on top of your existing Arch Linux installation. Think of it like adding a new aisle to your grocery store: the store (pacman) stays the same, but now it has 2800+ new products (security tools) available on the shelves. You install BlackArch tools exactly the same way you install any other Arch package — with pacman -S. The difference is that these tools are specifically curated for penetration testing, forensics, reverse engineering, and security research.

Technical detail: The BlackArch bootstrap script (strap.sh) does three things: downloads the BlackArch GPG signing key and imports it into pacman's keyring (so pacman can verify package signatures), adds the [blackarch] repository section to /etc/pacman.conf with the mirror URL, and synchronizes the package database. After this, pacman's dependency resolver treats BlackArch packages identically to official Arch packages — they can depend on official packages and vice versa, and updates flow through the normal pacman -Syu process.

⚠️ LEGAL DISCLAIMER

These tools are intended for authorized security testing only. Using them against systems you do not own or have explicit written permission to test is illegal in most jurisdictions, including under German law (§202a-c StGB — Computer Crime). Always get written authorization before performing any penetration testing. Use these tools for CTF competitions, your own lab environments, and authorized professional engagements.

bash
# Step 1: Download the BlackArch bootstrap script
curl -O https://blackarch.org/strap.sh

# Step 2: Verify the script's SHA1 hash (ALWAYS verify downloaded scripts!)
# Check the expected hash at https://blackarch.org/downloads.html
sha1sum strap.sh

# Step 3: Make it executable and run it
chmod +x strap.sh
sudo ./strap.sh

# Step 4: Sync the new repository
sudo pacman -Sy
Expected Output

[+] installing blackarch mirrorlist...
[+] BlackArch Linux is ready!
:: Synchronizing package databases...

After the sync completes, you have access to 2800+ security tools. You do not need to install all of them — the full collection is approximately 70 GB. Instead, BlackArch organizes tools into themed groups that you can install selectively based on your needs.

13.3 Setting Up a Security Lab

For practicing with these tools legally and safely, set up local vulnerable targets using Docker. These are intentionally broken applications designed for learning penetration testing.

bash
# ═══ Vulnerable Web Applications (Docker-based) ═══

# DVWA — Damn Vulnerable Web Application (classic beginner target)
docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa
# Open http://localhost in Firefox → login: admin/password

# Juice Shop — OWASP's modern vulnerable app (covers OWASP Top 10)
docker run -d -p 3000:3000 --name juice bkimminich/juice-shop
# Open http://localhost:3000

# WebGoat — OWASP interactive security lessons
docker run -d -p 8080:8080 --name webgoat webgoat/webgoat
# Open http://localhost:8080/WebGoat

# Metasploitable 3 (full vulnerable OS — requires more setup)
# See: https://github.com/rapid7/metasploitable3

# ═══ Quick Security Audit of Your Own System ═══
sudo lynis audit system
# Generates a detailed report with hardening recommendations
Expected Output
(Docker containers start in background)
(Access vulnerable apps via browser to practice attacks safely)

These Docker containers run isolated vulnerable applications on your local machine. You can practice SQL injection against DVWA, explore XSS and CSRF on Juice Shop, and work through structured security lessons on WebGoat — all without touching any real system. When you are done, docker stop dvwa juice webgoat && docker rm dvwa juice webgoat cleans everything up. For CTF competitions, platforms like HackTheBox, TryHackMe, and OverTheWire provide additional remote targets with varying difficulty levels.

13.4 Wireshark Group Permission

Wireshark needs raw network access to capture packets. Instead of running it as root (dangerous), add your user to the wireshark group which grants capture privileges through a setuid helper.

bash
# Add your user to the wireshark group
sudo usermod -aG wireshark mari

# Log out and back in, then launch
wireshark

# Or capture from terminal:
tshark -i wlan0 -c 100   # capture 100 packets on Wi-Fi interface
