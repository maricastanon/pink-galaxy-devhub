ARCH-CMD01.1 — pwd, cd, ls (and eza, zoxide)

These three commands are your eyes and legs in the terminal. pwd (print working directory) tells you where you are. cd (change directory) moves you somewhere else. ls (list) shows what is in the current location. You will type these commands hundreds of times per day — they become muscle memory within a week. On your Arch system, we also installed two modern replacements: eza (a Rust-based ls with icons and colors) and zoxide (a smart cd that remembers your most-visited directories and lets you jump to them with fuzzy matching).

Every process has a current working directory (CWD) stored as an inode reference in the kernel's task_struct. The pwd command reads /proc/self/cwd (a symlink the kernel maintains). The cd builtin calls the chdir() syscall, which resolves the path through the VFS layer, validates permission to traverse each directory component (execute permission on directories), and updates the process's CWD inode reference. The ls command calls opendir() then iterates with readdir(), reading directory entries (dentries) from the filesystem. With -l, it calls stat() on each entry to get the inode metadata (permissions, owner, size, timestamps). Note: cd must be a shell builtin (not an external program) because a child process cannot change the parent shell's CWD — the chdir() call only affects the calling process.

Path resolution for cd /home/mari/projects follows this algorithm in the VFS layer (fs/namei.c): (1) Start at the root inode (inode 2 on ext4). (2) Look up "home" in root's directory data blocks — each directory is a list of (inode_number, name_length, name) tuples. (3) Get home's inode, check execute (x) permission for the calling process's UID/GID against the inode's mode bits. (4) Look up "mari" in home's directory blocks. (5) Check permissions. (6) Look up "projects" in mari's directory blocks. (7) Check permissions. (8) Update current->fs->pwd to point to the projects dentry/inode. Each step involves a dentry cache lookup (hash table keyed by parent-inode + name) before hitting the disk. On modern systems with warm caches, all lookups hit the dentry cache (O(1) hash lookup) rather than reading from the SSD. The ls -la command triggers one stat() syscall per entry, each returning a 128-byte struct stat containing the inode number, mode (permissions + file type), link count, uid, gid, size, and three timestamps (access, modify, change).

bash
# ═══ pwd — Where Am I? ═══
pwd                             # /home/mari

# ═══ cd — Move Around ═══
cd /etc                        # absolute path
cd pacman.d                    # relative (into subdirectory)
cd ..                          # go UP one level → /etc
cd ../..                       # go UP two levels → /
cd ~                           # go HOME → /home/mari
cd                             # no argument = go HOME (same as cd ~)
cd -                           # toggle to PREVIOUS directory
cd ~/projects/pink-galaxy     # tilde expands to your home

# ═══ ls — What's Here? ═══
ls                             # basic listing
ls -l                         # long format: permissions, owner, size, date
ls -la                        # long + hidden files (starting with .)
ls -lah                       # + human-readable sizes (4.2K instead of 4096)
ls -lt                        # sort by modification time (newest first)
ls -lS                        # sort by size (largest first)

The ls -l output has a specific structure worth understanding: the first column shows the file type and permissions (we cover these in detail in ARCH-CMD03), the second is the hard link count, then owner, group, size, modification date, and finally the filename. Files starting with a dot (.) are hidden — they only appear with the -a flag. This convention is used for configuration files so they do not clutter normal directory listings.

🧪 Why must cd be a shell builtin rather than an external program like /usr/bin/cd?
Because an external program runs in a child process, and chdir() only changes the CWD of the calling process — the parent shell's directory would remain unchanged after the child exits
Because cd is too simple to need a separate binary
Because external programs cannot access the filesystem

ARCH-CMD01.2 — mkdir / rmdir (Creating & Removing Directories)

mkdir creates directories. The -p flag is your best friend — it creates the entire path at once (including any missing parent directories) and does not complain if the directory already exists. Without -p, you would have to create each level separately and get errors on duplicates. The rmdir command removes EMPTY directories only. For removing directories with contents, you use rm -r (covered in the next section).

The mkdir command calls the mkdir() syscall, which creates a new directory entry in the parent directory's data blocks and allocates a new inode with the directory file type bit set. The new directory automatically gets two entries: . (pointing to itself) and .. (pointing to the parent). With -p, mkdir iterates through each path component, calling mkdir() for each missing segment and ignoring EEXIST errors. The -m MODE flag sets permissions on creation (e.g., mkdir -m 700 secret). The rmdir syscall refuses to remove non-empty directories (returns ENOTEMPTY).

bash
# ═══ mkdir — Create Directories ═══

# Create a single directory
mkdir testdir

# Create nested directories in one command (-p = parents)
mkdir -p projects/webapp/{src,tests,docs,config,scripts}
# Creates: projects/ → webapp/ → src/, tests/, docs/, config/, scripts/

# Create with specific permissions
mkdir -m 700 ~/.secrets       # only owner can read/write/enter

# ═══ rmdir — Remove EMPTY Directories ═══
rmdir testdir                  # only works if empty
rmdir -p a/b/c               # removes c, then b, then a (if all empty)

# ═══ Practical: Project Scaffolding (one-liner from BASH-F03) ═══
mkdir -p myapp/{src/{models,routes,utils},tests,docs,config}
touch myapp/{README.md,.gitignore,src/{__init__,main}.py,tests/__init__.py}
eza --tree myapp

ARCH-CMD01.3 — cp and mv (Copy & Move/Rename)

cp copies files and directories. mv moves them (or renames them — in Linux, renaming IS moving). The most important flags to know are -r (recursive — required for copying directories), -i (interactive — asks before overwriting), and -v (verbose — shows what is happening). A common pattern you will use constantly is making a backup copy before editing a config file: cp /etc/pacman.conf /etc/pacman.conf.bak — or even more elegantly with brace expansion: cp /etc/pacman.conf{,.bak}.

The cp command creates a new file (new inode) and copies the source's data blocks to the new file's data blocks. With -a (archive), it also preserves ownership, permissions, timestamps, and symbolic links. The mv command behaves differently depending on whether the source and destination are on the same filesystem: if they are on the same filesystem, mv calls rename() which only updates directory entries (extremely fast — no data is copied); if they are on different filesystems, mv falls back to copy-then-delete (slow for large files). This is why mv within the same partition is instant regardless of file size, but moving files between partitions takes time proportional to the data size.

For same-filesystem moves, the rename() syscall in the VFS layer performs: (1) look up the source dentry, (2) look up or create the target dentry, (3) atomically update the parent directory's data blocks to remove the source entry and add the target entry (or update the target if it exists), (4) update the link count and .. entry if moving a directory. The inode itself is unchanged — its data blocks, permissions, timestamps, and owner remain identical. This is an O(1) metadata operation, not an O(n) data operation. For cross-filesystem moves, the kernel returns EXDEV (cross-device link), and the mv command falls back to cp -a source dest && rm source. The cp command uses read()/write() in a loop (typically 128KB chunks) to copy data blocks, or on modern kernels, copy_file_range() which can use reflinks on filesystems that support them (btrfs, XFS with reflink), making the copy instant and zero-cost until the data diverges (copy-on-write).

bash
# ═══ cp — Copy ═══

# Copy a file
cp file.txt file_backup.txt

# Copy a directory (must use -r for recursive)
cp -r projects/webapp projects/webapp-backup

# Copy preserving everything (permissions, timestamps, symlinks)
cp -a source/ destination/

# Interactive — ask before overwriting
cp -i newfile.txt existing.txt

# Backup config before editing (brace expansion trick from BASH-F03)
sudo cp /etc/pacman.conf{,.bak}
# Expands to: sudo cp /etc/pacman.conf /etc/pacman.conf.bak

# ═══ mv — Move / Rename ═══

# Rename a file (same directory = rename)
mv old_name.py new_name.py

# Move a file to another directory
mv report.pdf ~/Documents/

# Move multiple files at once
mv *.py src/                  # move all Python files into src/

# Safe move — ask before overwriting
mv -i draft.md final.md

# Move and rename simultaneously
mv ~/Downloads/archlinux-x86_64.iso ~/ISOs/arch-2026-03.iso
Key Insight
mv within the same filesystem is INSTANT regardless of file size
(it only changes the directory entry, not the actual data)
mv between different filesystems copies data then deletes the source (slow for big files)
🧪 You run mv bigfile.iso /mnt/usb/ and it takes 5 minutes. On the same partition, mv bigfile.iso backup.iso is instant. Why?
Because renaming is always faster than moving
Because the USB is a different filesystem — mv must copy all data blocks then delete the original. On the same filesystem, mv only updates the directory entry (a metadata change, no data is copied)
Because USB drives are slower than SSDs

ARCH-CMD01.4 — rm: The Danger Zone ⚠️

⚠️ CRITICAL WARNING: Linux has no recycle bin. When you rm a file, it is gone. There is no "undo", no "restore from trash", no "are you sure?" (unless you add -i). The command rm -rf / will happily destroy your entire system in seconds without asking a single question. Treat rm with the respect you give to a loaded weapon. Always double-check your arguments before pressing Enter, especially when using variables (rm -rf "$dir" — if $dir is empty, this becomes rm -rf which tries to delete everything in the current directory).

rm removes files. rm -r removes directories and everything inside them (recursive). rm -f forces removal without asking (even for read-only files). rm -rf combines both — it is the nuclear option that removes everything you point it at without any confirmation. This is the most dangerous command in Linux. The reason it is so dangerous is that Linux trusts you to know what you are doing — it does not second-guess your instructions. This trust is what makes Linux powerful, but it also means mistakes are permanent.

Three safety habits that will save you: (1) always use ls first to preview what you are about to delete, (2) for important deletions use rm -ri (interactive — asks for each file), (3) never put an unquoted variable after rm -rf — always quote it AND check that it is non-empty first.

The rm command calls unlink() for files and rmdir() for directories. The unlink() syscall removes the directory entry (the name→inode mapping) and decrements the inode's link count. If the link count reaches zero AND no process has the file open, the kernel frees the inode and its data blocks. If a process still has the file open (common with log files), the data persists until the last file descriptor is closed — the file becomes "deleted but still in use." The -f flag suppresses errors for non-existent files and overrides the write-protection prompt (which normally asks before deleting files you do not have write permission on). The --preserve-root flag (enabled by default in GNU coreutils) refuses to operate on /, but this only protects against the literal rm -rf /, not against rm -rf /* which expands to all top-level entries.

The unlink() syscall in the VFS layer performs: (1) resolve the path to find the target dentry and its parent, (2) check write+execute permission on the parent directory (not the file itself — because you are modifying the parent's directory entry), (3) call the filesystem's unlink inode operation, which removes the dentry from the parent's data blocks, decrements the inode's i_nlink counter, and updates the parent's mtime/ctime. (4) If i_nlink == 0 and i_count == 0 (no open file descriptors), the inode and its data blocks are freed — the bitmap entries for those blocks are cleared, making them available for reuse. On ext4, the inode's data block pointers are zeroed, the inode bitmap bit is cleared, and the block bitmap bits are cleared. The actual disk sectors are NOT overwritten with zeros — the data remains physically on the SSD until those blocks are reused or TRIMmed. This is why forensic tools can sometimes recover "deleted" files, and why shred or blkdiscard exist for secure deletion.

bash
# ═══ rm — Remove Files ═══

# Delete a single file
rm unwanted_file.txt

# Delete multiple files
rm file1.txt file2.txt file3.txt

# Delete all .tmp files in current directory
rm *.tmp

# Delete a directory and all its contents
rm -r old_project/

# Force delete (no prompts, no errors on missing files)
rm -rf build/ dist/ __pycache__/

# ═══ SAFE DELETION PATTERNS ═══

# 1. Preview before deleting (ALWAYS do this)
ls *.log                       # see what would be deleted
rm *.log                       # then delete

# 2. Interactive mode (asks for each file)
rm -ri old_project/           # confirms each file — tedious but safe

# 3. SAFE variable deletion (from BASH-F08)
dir_to_clean="/tmp/build-output"
if [[ -n "$dir_to_clean" && -d "$dir_to_clean" ]]; then
    rm -rf "$dir_to_clean"    # safe — verified non-empty and exists
fi
# NEVER: rm -rf $dir (if $dir is empty, this deletes current directory!)

# 4. Use trash-cli instead (moves to trash instead of deleting)
# paru -S trash-cli
# trash-put file.txt    ← moves to trash (recoverable!)
# trash-list            ← see what's in trash
# trash-restore         ← recover a file
✅ Safe deletion habits:

Always ls before rm. Always quote variables in rm commands. Always check that path variables are non-empty before using them with rm -rf. Consider installing trash-cli which moves files to a recoverable trash instead of permanently deleting them. Set up Timeshift snapshots (from Part 4) as a safety net.

❌ The disaster scenarios:

rm -rf / destroys everything (mitigated by --preserve-root default). rm -rf $UNSET_VAR/* expands to rm -rf /* when the variable is empty. rm -rf ~/. documents (accidental space) deletes your entire home directory (the . matches ~/. which is your home directory). All of these are preventable with the habits above.

🧪 You have dir="" (empty variable) and run rm -rf $dir/*. What happens?
Nothing — rm sees an empty argument and does nothing
rm reports an error about the empty variable
The command expands to rm -rf /* which deletes everything in the root filesystem! The empty variable vanishes, leaving the /* glob which matches all top-level directories

ARCH-CMD01.5 — ln: Hard Links and Symbolic Links

A link is an alternative name for a file. Linux has two types. A hard link is like having two names in the phone book that ring the same phone — both names are equally "real", and the phone (the actual data) only gets disconnected when ALL names pointing to it are removed. A symbolic link (symlink) is like a redirect sign that says "go look over there" — it points to another filename, not to the actual data. If you remove the original file, the symlink breaks (it points to nothing — a "dangling symlink").

Symlinks are by far the more common type in daily use. You use them for: pointing dotfile configs from your home directory into a dotfiles Git repo, creating shorter paths to deeply nested directories, pointing /usr/local/bin/python to your actual Python binary, and managing multiple versions of anything (the "current" symlink always points to whichever version is active).

A hard link (ln source linkname) creates a new directory entry pointing to the same inode as the source. Both names share the same inode number, data blocks, permissions, and timestamps. The inode's link count (i_nlink) increments with each hard link. Hard links cannot cross filesystem boundaries (because inodes are per-filesystem) and cannot link to directories (to prevent circular loops in the filesystem tree). A symbolic link (ln -s target linkname) creates a new inode whose data content is the path string of the target. When the kernel resolves a symlink, it reads the stored path and restarts path resolution from that point. Symlinks can cross filesystems, link to directories, and link to non-existent targets (dangling links).

Hard links are implemented at the filesystem level: ln source link calls link() syscall which adds a new directory entry (name + inode number) in the target directory pointing to the source's inode, and increments the inode's i_nlink field. No new inode or data blocks are allocated. Symbolic links are a distinct file type: ln -s target link calls symlink() syscall which creates a new inode with file type S_IFLNK (0xA000). For short targets (≤60 bytes on ext4), the path string is stored directly in the inode's i_block[] field ("fast symlinks" — no additional data block needed). For longer targets, a data block is allocated. When the VFS encounters a symlink during path resolution (lookup_fast() or lookup_slow() in namei.c), it calls the inode's get_link() operation to read the target path, then restarts resolution. A maximum of 40 symlink indirections are followed before returning ELOOP (to prevent infinite loops).

bash
# ═══ Symbolic Links (most common — use these 99% of the time) ═══

# Create a symlink: ln -s TARGET LINKNAME
ln -s ~/projects/pink-galaxy ~/pg
# Now: cd ~/pg  → takes you to ~/projects/pink-galaxy

# Dotfiles management pattern (from Part 4)
ln -sf ~/dotfiles/zshrc ~/.zshrc         # -f = force (overwrite existing)
ln -sf ~/dotfiles/hypr ~/.config/hypr    # symlink entire directory

# Check where a symlink points
readlink -f ~/.zshrc           # /home/mari/dotfiles/zshrc
ls -la ~/.zshrc                # shows: .zshrc → /home/mari/dotfiles/zshrc

# ═══ Hard Links (less common, but useful to understand) ═══

# Create a hard link
ln original.txt hardlink.txt
# Both names point to the same inode (same data on disk)

# Verify they share the same inode number
ls -li original.txt hardlink.txt
# The first column (inode number) is the same for both

# Delete the original — hard link STILL works (data persists)
rm original.txt
cat hardlink.txt               # still accessible! Data lives until ALL links gone
Expected Output (ls -li)
1048577 -rw-r--r-- 2 mari mari 42 Mar 15 16:00 hardlink.txt
1048577 -rw-r--r-- 2 mari mari 42 Mar 15 16:00 original.txt
                   ↑ link count = 2 (two names point to same inode 1048577)
🧪 You create a symlink ln -s /tmp/data.txt ~/link.txt then delete /tmp/data.txt. What happens when you cat ~/link.txt?
Error: "No such file or directory" — the symlink becomes "dangling" because it points to a path that no longer exists (symlinks point to names, not data)
It still works because the symlink preserves the data
The symlink is automatically deleted

ARCH-CMD01.6 — find and fd: Searching the Filesystem

find is the classic file search tool — extremely powerful but with a syntax that feels backwards to most people. It walks through directory trees checking each file against your criteria (name, size, type, age, permissions, owner) and either prints matches or executes a command on them. It can answer questions like "find all Python files modified in the last 24 hours" or "find all files larger than 100MB" or "find all empty directories and delete them."

fd is the modern replacement (installed on your Arch system) — it has a much simpler syntax, uses regex by default, respects .gitignore, has colorized output, and is significantly faster because it is written in Rust with parallel directory traversal. For everyday searches, use fd. For complex searches with advanced conditions and actions, use find.

The find command performs a depth-first traversal of the filesystem tree using opendir()/readdir() calls, applying test expressions to each entry. The expression language supports tests (-name, -type, -size, -mtime, -perm, -user), actions (-print, -exec, -delete), and logical operators (-and, -or, -not, \( \) for grouping). The -exec {} \; syntax runs a command for each match (one fork/exec per file). The -exec {} + syntax batches matches into a single command invocation (much faster — equivalent to xargs). The fd tool from the fd package uses the ignore crate to respect .gitignore patterns and the rayon crate for parallel directory traversal, giving 5-10x speedup over find on large trees.

bash
# ═══ find — The Classic Search Tool ═══

# Find by name
find . -name "*.py"                          # Python files in current tree
find /etc -name "*.conf" 2>/dev/null           # config files in /etc
find . -iname "readme*"                        # case-insensitive search

# Find by type
find . -type f -name "*.log"                 # files only (not directories)
find . -type d -empty                         # empty directories
find . -type l                                 # symbolic links only

# Find by size
find . -size +100M                             # files larger than 100MB
find . -size -1k                               # files smaller than 1KB

# Find by time
find . -mtime -1                               # modified in last 24 hours
find . -mmin -30                               # modified in last 30 minutes

# Find and DO something (-exec)
find . -name "*.pyc" -delete                  # delete all .pyc files
find . -name "*.py" -exec wc -l {} \;        # count lines in each .py file
find . -name "*.py" -exec wc -l {} +          # same but batched (faster!)
find . -type d -empty -delete                  # delete empty directories

# ═══ fd — The Modern Alternative ═══

fd ".py$"                                      # find Python files (regex!)
fd "readme" -i                                 # case-insensitive
fd -e py                                        # by extension (shorthand)
fd -e py -x wc -l                              # execute wc -l on each match
fd -t d -e                                      # empty directories
fd --size +10m                                  # files > 10MB
fd --changed-within 1h                          # modified in last hour
fd "test" src/                                  # search only in src/
Expected Output (fd -e py)
src/main.py
src/utils.py
src/models/user.py
tests/test_main.py
✅ When to use find vs fd:

Use fd for everyday searches — it is faster, has better defaults, and the syntax is simpler. Use find when you need complex boolean conditions (-and, -or, -not), permission-based searches, or when writing POSIX-portable scripts. Both are essential tools in your arsenal.

❌ Performance trap:

find . -exec cmd {} \; forks a new process for EVERY match. find . -exec cmd {} + (note the plus instead of semicolon) batches matches into a single command invocation. For thousands of files, this can be 100x faster. The fd equivalent fd -x cmd runs in parallel by default for even more speed.

Exercise 2 (Medium) — Cleanup Script

Write a script that finds all files in /tmp owned by your user that are older than 7 days and larger than 1MB. Print them with their sizes using find + -exec. Add a --delete flag that actually removes them (only if the flag is passed).

solution
#!/bin/bash
set -euo pipefail

DELETE=false
[[ "${1:-}" == "--delete" ]] && DELETE=true

echo "🧹 Old large files in /tmp (>7 days, >1MB, owned by $USER):"
echo "───────────────────────────────────────────────────"

find /tmp -user "$USER" -type f -mtime +7 -size +1M \
    -exec ls -lh {} \; 2>/dev/null

if "$DELETE"; then
    echo ""
    read -rp "Delete these files? [y/N] " confirm
    if [[ "${confirm,,}" == "y" ]]; then
        find /tmp -user "$USER" -type f -mtime +7 -size +1M -delete
        echo "✓ Deleted"
    fi
else
    echo ""
    echo "Run with --delete to remove these files"
fi

Exercise 3 (Hard) — Project Analyzer

Write a script that takes a project directory as an argument and generates a report: total files, total directories, total size, file count by extension (top 10), largest files (top 5), most recently modified files (top 5), and any symlinks found. Use a combination of find, fd, awk, sort, and printf for formatting.

solution
#!/bin/bash
set -euo pipefail

dir="${1:?Usage: $0 }"
[[ -d "$dir" ]] || { echo "Not a directory: $dir" >&2; exit 1; }

echo "═══ Project Report: $dir ═══"
echo ""
printf "  %-20s %s\n" "Total files:" "$(find "$dir" -type f | wc -l)"
printf "  %-20s %s\n" "Total dirs:" "$(find "$dir" -type d | wc -l)"
printf "  %-20s %s\n" "Total size:" "$(du -sh "$dir" | awk '{print $1}')"
printf "  %-20s %s\n" "Symlinks:" "$(find "$dir" -type l | wc -l)"

echo ""
echo "─── Top Extensions ───"
find "$dir" -type f | sed 's/.*\.//' | sort | uniq -c | sort -rn | head -10 | \
    while read -r count ext; do
        printf "  %-6d .%s\n" "$count" "$ext"
    done

echo ""
echo "─── Largest Files ───"
find "$dir" -type f -exec du -h {} + 2>/dev/null | sort -rh | head -5

echo ""
echo "─── Recently Modified ───"
find "$dir" -type f -printf '%T@ %Tc  %p\n' 2>/dev/null | sort -rn | head -5 | \
    awk '{$1=""; print}'
